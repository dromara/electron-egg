开发模式是日常开发的核心命令。`ee-bin dev` 同时启动前端开发服务器和 Electron 进程，支持文件监听与自动重建，让开发者修改代码后无需手动重启。

## 完整流程

`ServeProcess.dev()` 的执行流程：

<Steps>
### 设置环境变量

将 `NODE_ENV` 设为 `dev`，ee-core 和 esbuild 会根据此值决定调试行为（如自动启用 inline sourcemap）。

```js
process.env.NODE_ENV = 'dev';
```

### 加载配置

从 `./cmd/bin.js` 加载用户配置，与 `bin_default.ts` 默认配置深度合并，得到完整的 `BinConfig`。

### 解析服务列表

解析 `--serve` 选项确定要启动的服务。未指定时，默认启动 `dev` 配置中定义的所有命令（通常是 `frontend` 和 `electron`）。

```js
// 未指定 --serve 时，自动拼接所有 dev 配置键名
let command = serve;
if (!command) {
  command = Object.keys(binCmdConfig).join(',');
}
```

### 打包 Electron

如果服务列表包含 `electron`，先执行 esbuild 打包（`this.bundle()`），将 `electron/` 目录的代码打包到 `public/electron/`。

然后切换 `package.json` 的 `main` 字段，从 `./electron/main.js` 指向 `./public/electron/main.js`。

### 启动监听模式（可选）

如果 `dev.electron.watch` 为 `true`，使用 chokidar 监听 `electron/` 目录变化：

1. 文件变更触发 debounce 延迟（默认 1000ms）
2. 延迟到期后重新执行 `this.bundle()`
3. 使用 `tree-kill` 杀死旧 Electron 进程
4. 重新启动 Electron 子进程

### 启动所有服务

调用 `this.multiExec()` 启动所有配置的服务进程。前端和 Electron 并行运行。
</Steps>

## --serve 选项

通过 `--serve` 精确控制启动哪些服务：

<Tabs>
<Tab title="全部启动">
```bash
ee-bin dev
# 等同于
ee-bin dev --serve frontend,electron
```
同时启动前端开发服务器和 Electron 进程。
</Tab>
<Tab title="仅前端">
```bash
ee-bin dev --serve frontend
```
只启动前端开发服务器，不启动 Electron。适用于纯前端调试。
</Tab>
<Tab title="仅 Electron">
```bash
ee-bin dev --serve electron
```
只启动 Electron 进程。适用于前端已经通过其他方式启动的场景。
</Tab>
</Tabs>

<Warning>
当 `dev.frontend.protocol` 为 `file://` 时，dev 模式会自动跳过前端启动。因为 `file://` 协议表示前端由 Electron 直接加载本地文件，不需要独立的前端开发服务器进程。
</Warning>

## 前端开发服务器配置

`dev.frontend` 配置段控制前端开发服务器行为：

```js
// cmd/bin.js
module.exports = {
  dev: {
    frontend: {
      directory: './frontend',      // 前端项目目录
      cmd: 'npm',                   // 执行命令
      args: ['run', 'dev'],         // 命令参数
      protocol: 'http://',          // 服务协议（http:// 或 file://）
      hostname: 'localhost',        // 开发服务器主机名
      port: 8080,                   // 开发服务器端口
      indexPath: 'index.html',      // 入口页面文件名
      force: false,                 // 是否强制刷新页面
      sync: false,                  // 同步执行模式（阻塞直到前端构建完成）
    },
  },
};
```

关键字段说明：

| 字段 | 说明 |
|------|------|
| `protocol` | `http://` 表示前端通过 HTTP 开发服务器提供；`file://` 表示直接读取本地文件 |
| `hostname` | 前端开发服务器监听地址，ee-core 在创建 BrowserWindow 时使用此值 |
| `port` | 前端开发服务器端口，ee-core 组合 `protocol + hostname + port + indexPath` 加载页面 |
| `sync` | `true` 时前端命令同步执行（阻塞），适用于需要前端先完成的场景 |

## Electron 开发配置

`dev.electron` 配置段控制 Electron 进程行为：

```js
module.exports = {
  dev: {
    electron: {
      directory: './',                // 项目根目录
      cmd: 'electron',               // 执行命令
      args: ['.', '--env=local'],     // 命令参数
      loadingPage: '/public/html/loading.html', // 启动过渡页
      watch: false,                   // 是否监听文件变更自动重建
      sync: false,                    // 同步执行模式
      delay: 1000,                    // 监听模式 debounce 延迟（毫秒）
    },
  },
};
```

## 文件监听模式

启用 `watch: true` 后，chokidar 会持续监听 `electron/` 目录的文件变更：

```js
// 启用监听模式
module.exports = {
  dev: {
    electron: {
      watch: true,
      delay: 1000,  // debounce 延迟
    },
  },
};
```

### 监听工作流

1. chokidar 检测到 `electron/` 目录下文件变更
2. 触发 debounce 定时器，延迟 `delay` 毫秒
3. 延迟期间如有新变更，重置定时器（防止连续变更频繁重建）
4. 定时器到期后执行 `this.bundle()` 重新打包
5. 使用 `tree-kill` 以 `SIGKILL` 杀死旧 Electron 进程
6. 重新通过 `multiExec` 启动 Electron

<Note>
`delay` 默认值为 1000ms。如果你的 IDE 保存时会触发多次写入（如格式化 + 保存），建议适当增大此值。
</Note>

## LoadingPage 启动过渡页

`loadingPage` 字段指定 Electron 启动时显示的过渡页面。ee-core 在创建 BrowserWindow 时先加载此页面，等前端开发服务器就绪后再跳转到主页面。

```js
loadingPage: '/public/html/loading.html'
```

这解决了 Electron 启动时前端尚未就绪导致的白屏问题。用户看到一个加载提示页面，前端服务器就绪后自动跳转。

## 信号处理

开发模式下注册了 `SIGINT`（Ctrl+C）和 `SIGTERM` 信号处理器：

1. 杀死所有追踪的子进程（通过 `tree-kill`）
2. 恢复 `package.json` 的 `main` 字段到原始值
3. 等待 500ms 确保进程退出
4. 调用 `process.exit(0)` 退出

<Warning>
在 Windows 平台上，Electron 进程退出时不一定会终止父进程。ee-bin 会在 Electron 退出后提示用户按 Ctrl+C 手动退出。
</Warning>

## 子进程管理策略

`multiExec` 对每个命令创建独立子进程：

- **异步模式**（`sync: false`）：使用 `cross-spawn` 创建子进程，存入 `execProcess` 表追踪。子进程退出时在 dev 模式下打印提示。
- **同步模式**（`sync: true`）：使用 `cross-spawn.sync` 阻塞执行，不追踪进程（进程已完成）。

所有子进程的 `maxBuffer` 设为 1GB（`1024 * 1024 * 1024`），防止大型构建命令输出被截断。
