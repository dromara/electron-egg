# 控制器加载

控制器是 electron-egg 中处理业务请求的核心单元。ControllerLoader 在启动阶段负责发现、加载和注册所有控制器，其工作方式在开发模式和打包模式下截然不同。

## 两种加载路径

<Tabs>
  <Tab title="开发模式 — 文件系统扫描">
    开发模式下，ControllerLoader 使用 globby 扫描 `electron/controller/` 目录，逐个 `require()` 加载：

    ```js
    // ControllerLoader.load() — 开发模式
    load() {
      const controllerDir = path.join(this.app.appDir, 'electron', 'controller');

      // 使用 globby 扫描所有 JS 文件
      const files = globby.sync(['**/*.js'], {
        cwd: controllerDir,
      });

      // 遍历文件，require 加载并注册
      for (const file of files) {
        const filePath = path.join(controllerDir, file);
        const module = require(filePath);
        const propertyName = this._fileToProperty(file);
        this.registry[propertyName] = module;
      }
    }
    ```

    加载流程：

    ```mermaid
    flowchart LR
      A[globby 扫描目录] --> B[遍历文件列表] --> C[require 加载模块] --> D[计算属性名] --> E[注册到 registry]
    ```
  </Tab>

  <Tab title="打包模式 — 注册表注入">
    打包模式下，控制器在构建阶段已被 `controllerRegistryPlugin` 注册到 `globalThis.__EE_CONTROLLER_REGISTRY__`：

    ```js
    // ControllerLoader.load() — 打包模式
    load() {
      const registry = globalThis.__EE_CONTROLLER_REGISTRY__;
      if (!registry) {
        // 降级到文件系统扫描
        return this._loadFromFileSystem();
      }

      // 直接从注册表读取
      for (const [name, entry] of Object.entries(registry)) {
        this.registry[name] = entry.module;
      }
    }
    ```

    注册表中的每个控制器是**惰性 getter**，只有在首次访问时才执行 `require()`：

    ```js
    // ee-bin 生成的注册表代码
    globalThis.__EE_CONTROLLER_REGISTRY__ = {
      get 'user'() {
        return { module: require('./controller/user.js') };
      },
      get 'user/profile'() {
        return { module: require('./controller/user/profile.js') };
      },
    };
    ```

    <Note>
    惰性 getter 的设计确保控制器不会在注册阶段被实际加载，只在请求到达时才执行 `require()`。这避免了初始化顺序问题——控制器之间可能存在相互依赖，惰性加载使得所有控制器先注册再按需加载。
    </Note>
  </Tab>
</Tabs>

## ControllerLoader 包装机制

ControllerLoader 不仅仅是加载文件，它还负责将原始类导出包装为可安全调用的中间件。

### wrapClass — 原型链遍历

当控制器文件导出一个类时，`wrapClass()` 会遍历该类的原型链，将每个非构造器方法包装为中间件：

```js
function wrapClass(ControllerClass) {
  const proto = ControllerClass.prototype;
  const methods = {};

  // 遍历原型链上的所有方法
  for (const methodName of Object.getOwnPropertyNames(proto)) {
    // 排除构造器和内置方法
    if (methodName === 'constructor' ||
        methodName.startsWith('_')) {
      continue;
    }

    const method = proto[methodName];
    if (typeof method !== 'function') continue;

    // 包装为中间件
    methods[methodName] = methodToMiddleware(
      ControllerClass,
      methodName,
      method
    );
  }

  return methods;
}
```

原型链遍历确保继承的方法也能被发现：

```js
// BaseController 定义了通用方法
class BaseController {
  ok(data) { return { code: 0, data }; }
  error(msg) { return { code: -1, msg }; }
}

// UserController 继承 BaseController
class UserController extends BaseController {
  async login(args) { /* ... */ }
  async logout(args) { /* ... */ }
}

// wrapClass(UserController) 会发现：
// - login, logout（自身方法）
// - ok, error（继承自 BaseController）
```

### methodToMiddleware — 并发安全设计

`methodToMiddleware()` 是控制器并发安全的关键。每次请求到来时，它会创建一个新的控制器实例：

```js
function methodToMiddleware(ControllerClass, methodName, originalMethod) {
  return async function middleware(args) {
    // 每次请求创建新的控制器实例
    const controllerInstance = new ControllerClass();

    // 注入 app 上下文
    controllerInstance.app = this.app;
    controllerInstance.service = this.app.service;
    controllerInstance.config = this.app.config;
    controllerInstance.logger = this.app.logger;

    // 调用原始方法
    const result = await originalMethod.call(controllerInstance, args);

    return result;
  };
}
```

<Warning>
为什么每次请求都要创建新实例？因为 Electron 应用可能同时处理多个并发请求（HTTP、SocketIO、IPC）。如果所有请求共享同一个控制器实例，实例属性（如 `this.requestData`）会产生竞态条件，导致数据混乱。新实例模式确保每个请求拥有独立的上下文。
</Warning>

并发安全的对比：

| 模式 | 单实例共享 | 每次请求新实例 |
|------|-----------|---------------|
| 并发请求 | `this.data` 被覆盖 | 每个请求独立 `this` |
| 内存开销 | 低（1 个实例） | 稍高（每请求 1 个） |
| 状态安全 | 不安全 | 安全 |

## 控制器文件命名规则

### 目录结构约定

控制器文件放在 `electron/controller/` 目录下，支持嵌套子目录：

```
electron/controller/
├── user.js             → 注册为 controller.user
├── admin.js            → 注册为 controller.admin
├── user/
│   ├── profile.js      → 注册为 controller['user/profile']
│   └── settings.js     → 注册为 controller['user/settings']
└── api/
    └── v1/
        └── todo.js     → 注册为 controller['api/v1/todo']
```

### caseStyle 转换规则

FileLoader 使用 `caseStyle: 'lower'` 配合 `defaultCamelize` 将文件路径转换为属性名：

<Steps>
  <Step title="文件名 → 属性名映射">
    `defaultCamelize` 的转换规则：

    - 单个文件名保持原样：`user.js` → `user`
    - 多级目录路径拼接：`user/profile.js` → `user/profile`
    - 下划线/连字符转为驼峰：`user_profile.js` → `userProfile`
    - 以 `lower` 模式处理，首字母保持小写

    ```js
    // defaultCamelize 实现（简化）
    function defaultCamelize(filepath, caseStyle) {
      const parts = filepath.split('/');
      const names = parts.map((part, index) => {
        // 将下划线/连字符转为驼峰
        let name = part.replace(/[-_](\w)/g, (_, c) => c.toUpperCase());
        // caseStyle: 'lower' — 保持小写
        if (caseStyle === 'lower') {
          name = name.charAt(0).toLowerCase() + name.slice(1);
        }
        return name;
      });
      return names.join('/');
    }
    ```
  </Step>

  <Step title="打包模式的注册表名称">
    `controllerRegistryPlugin` 在构建时复用相同的 `defaultCamelize` 逻辑，确保开发模式和打包模式生成一致的属性名：

    ```js
    // ee-bin/plugins/controller_registry_plugin.ts
    const propertyName = defaultCamelize(relativePath, 'lower');
    // 生成的注册表代码：
    // get 'userProfile'() { return { module: require('./controller/user_profile.js') }; }
    ```
  </Step>
</Steps>

## FileLoader API

FileLoader 是 ee-core 中负责文件发现与注册的核心工具类：

```js
class FileLoader {
  /**
   * 从注册表解析并加载（打包模式）
   * @param {Object} registry - globalThis.__EE_CONTROLLER_REGISTRY__
   * @returns {Object} 注册后的控制器映射
   */
  parseFromRegistry(registry) {
    const result = {};
    for (const [name, entry] of Object.entries(registry)) {
      result[name] = wrapClass(entry.module);
    }
    return result;
  }

  /**
   * 从文件系统扫描并加载（开发模式）
   * @param {Object} options - { directory, target, pattern }
   * @returns {Object} 注册后的控制器映射
   */
  parseFromFileSystem(options) {
    const files = globby.sync(options.pattern, { cwd: options.directory });
    const result = {};
    for (const file of files) {
      const module = require(path.join(options.directory, file));
      const name = defaultCamelize(file.replace(/\.js$/, ''), options.caseStyle);
      result[name] = wrapClass(module);
    }
    return result;
  }
}
```

## 控制器编写示例

```js
// electron/controller/user.js
const { Controller } = require('ee-core');

class UserController extends Controller {
  /**
   * 用户登录
   */
  async login(args) {
    const { username, password } = args;
    const user = await this.service.user.verify(username, password);
    if (!user) {
      return this.error('用户名或密码错误');
    }
    return this.ok(user);
  }

  /**
   * 获取用户信息
   */
  async getInfo() {
    const user = await this.service.user.getCurrent();
    return this.ok(user);
  }
}

module.exports = UserController;
```

<Note>
控制器中的 `this` 指向当前请求的独立实例，包含 `app`、`service`、`config`、`logger` 等上下文。不要在控制器中存储请求级别的状态到类属性上——每次请求都会创建新实例，状态不会保留。
</Note>
