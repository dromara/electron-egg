# 框架总览

## 项目简介

electron-egg (ee-dev-v5) 是一个企业级 Electron 桌面应用框架，采用 pnpm workspace monorepo 结构。框架的核心能力由两个独立 npm 包提供：

- **ee-core**：运行时框架核心，负责应用生命周期、配置加载、控制器注册、IPC通信、日志、存储、后台任务等
- **ee-bin**：CLI 构建工具，负责开发服务、esbuild 打包、加密、图标生成、增量更新等

## 项目结构

```
ee-dev/
├── packages/
│   ├── ee-core/        # 运行时框架核心（TypeScript, 双 CJS+ESM 输出）

│   │       ├── app/        # 入口：boot, application, events, dir
│   │       ├── config/     # 配置加载/合并
│   │       ├── controller/ # 控制器加载（registry + filesystem）
│   │       ├── core/       # FileLoader, utils

│   │       ├── electron/   # Electron app/window 管理
│   │       ├── jobs/       # ChildJob, ChildPoolJob, LoadBalancer
│   │       ├── loader/     # loadFile, requireFile, resolveModule
│   │       ├── log/        # Pino 日志（pino-roll + pino-pretty）
│   │       ├── socket/     # SocketServer, HttpServer, IpcServer
│   │       ├── storage/    # SqliteStorage (better-sqlite3)
│   │       ├── types/      # TypeScript 类型定义
│   │       └── utils/      # extend, is, json, wrap, helper 等
│   └── ee-bin/          # CLI 构建工具（TypeScript, 双 CJS+ESM 输出）
│       └── src/
│           ├── config/     # bin_default.ts 默认配置
│           ├── plugins/    # bundle_registry_plugin.ts (esbuild)
│           ├── tools/      # serve, encrypt, move, iconGen, incrUpdater
│           ├── lib/        # extend, helpers, utils 内部库

├── cmd/                 # bin.js（项目配置）, builder*.json
```

## 架构分层

框架采用严格的分层启动流程：

```
Boot层 (ElectronEgg)
  ├─ 收集环境信息 → 写入 process.env
  ├─ 基础层初始化 (init)
  │   ├─ loadException  ← 注册全局异常处理
  │   ├─ loadConfig     ← 加载配置（依赖异常处理）
  │   ├─ loadDir        ← 创建必要目录（依赖配置）
  │   └─ loadLog        ← 初始化日志（依赖配置+目录）
  │
业务层 (Application.run)
  ├─ loadController  ← 加载控制器（注册 IPC handler）
  ├─ loadSocket      ← 启动通信服务（SocketIO/HTTP/IPC）
  │
生命周期层
  ├─ emitLifecycle(Ready) ← 触发 ready 事件钩子
  │
Electron层
  ├─ loadElectron   ← 创建窗口、加载页面
```

**启动顺序不可随意变更**：每个后续模块都依赖前序模块的输出（配置、日志、异常处理等）。

## 双包关系

ee-core 和 ee-bin 是**构建时-运行时**的 producer-consumer 关系：

| 维度 | ee-bin (构建时) | ee-core (运行时) |
|------|-----------------|-----------------|
| 职责 | CLI 工具，打包开发构建 | 框架核心，应用运行 |
| 运行时机 | 开发/构建命令执行时 | Electron 进程启动后 |
| 数据交换 | 生成 `global.__EE_*_REGISTRY__` | 消费 registry，或 fallback 到 filesystem |
| 包关系 | esbuild external，不在 bundle 中 | esbuild external，不在 bundle 中 |
| 安装方式 | npm install ee-bin（开发依赖） | npm install ee-core（运行依赖） |

**核心契约**：ee-bin 的 `bundleRegistryPlugin` 在构建时扫描 controller/ 和 config/ 目录，生成虚拟模块设置 `global.__EE_CONTROLLER_REGISTRY__` 和 `global.__EE_CONFIG_REGISTRY__`。ee-core 在运行时检测这些全局变量，如果存在则使用 registry 模式（bundle mode），否则使用 filesystem 扫描模式（dev mode）。两种模式产生的数据结构完全一致。

## 通信架构

框架提供三通道并行通信：

1. **Electron IPC** (`IpcServer`)：主进程↔渲染进程，使用 `ipcMain.handle/on` + `ipcRenderer.invoke/send`
2. **SocketIO** (`SocketServer`)：主进程↔外部进程（Go/Python 等），使用 socket.io 协议
3. **HTTP** (`HttpServer`)：Koa RESTful API，支持 CORS 和 multipart

三个通道共享同一套控制器路由机制（`resolveControllerFn`），通道格式：`controller/{name}/{method}`。

## 代码约定

- ee-core 源码中 ESM import 必须带 `.js` 扩展名（NodeNext 要求）
- ee-core 的 `exports` map 由 `scripts/gen-exports.js` 自动生成，勿手动编辑
- ee-core 和 ee-bin 是 standalone npm 包，修改源码后需 `npm run build` 才能生效
- 运行时路径通过 `process.env` 全局变量传递（EE_BASE_DIR, EE_ELECTRON_DIR 等）
- 日志使用 `debug` 库命名空间（`ee-core:*`），开发调试用 `DEBUG=ee-*` 环境变量
