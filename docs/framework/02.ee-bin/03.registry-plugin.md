# bundleRegistryPlugin 详细实现

## 概述

bundleRegistryPlugin 是 ee-bin 打包流程的核心插件，基于 esbuild Plugin API 实现。它在构建时扫描 `electron/controller/` 和 `electron/config/` 目录，生成三个虚拟模块，将控制器和配置模块注册到全局变量上，确保打包后的代码能正确定位这些模块。懒加载 getter 设计避免了初始化顺序问题。

**源文件**: `packages/ee-bin/src/plugins/bundle_registry_plugin.ts`

## 核心流程

### 插件总体架构

```
bundleRegistryPlugin(controllerDir, mainJsPath, configDir)
  └── 返回 esbuild Plugin { name: 'bundle-registry', setup(build) }
      ├── onStart → 扫描 controller/ 和 config/ 目录
      ├── onResolve + onLoad → app:controller-registry (虚拟模块)
      ├── onResolve + onLoad → app:config-registry (虚拟模块)
      └── onResolve + onLoad → app:bundle-entry (虚拟模块)
```

### onStart: 每次构建前扫描目录

**源码**: 第 92-107 行

```typescript
build.onStart(() => {
  const controllerFiles = globbySync(['**/*.js', '**/*.jsc', '**/*.ts'], { cwd: controllerDir });
  registryEntries = controllerFiles.map((filepath) => ({
    fullpath: 'controller/' + filepath.replace(/\\/g, '/'),
    properties: computeProperties(filepath),
    relPath: filepath,
  }));

  const configFiles = globbySync(['**/*.js', '**/*.jsc', '**/*.ts'], { cwd: configDir });
  configEntries = configFiles.map((filepath) => ({
    filename: filepath.replace(/\\/g, '/').substring(0, filepath.lastIndexOf('.')),
    relPath: filepath,
  }));
});
```

关键设计：
- `onStart` 在每次构建/rebuild 开始时重新扫描目录（支持文件增删的 watch 模式）
- 使用 `globbySync` 扫描匹配 `**/*.js`、`**/*.jsc`、`**/*.ts` 的文件
- Windows 路径反斜杠通过 `replace(/\\/g, '/')` 统一为正斜杠
- 扫描结果存入闭包变量 `registryEntries` 和 `configEntries`，供 `onLoad` 回调使用

### 虚拟模块 1: app:controller-registry

**源码**: 第 111-135 行

**onResolve**: 拦截路径 `app:controller-registry`，设置命名空间 `controller-registry`

```typescript
build.onResolve({ filter: /^app:controller-registry$/ }, (args) => {
  return { path: args.path, namespace: 'controller-registry' };
});
```

**onLoad**: 生成控制器注册表代码，设置 `global.__EE_CONTROLLER_REGISTRY__`

```typescript
build.onLoad({ filter: /.*/, namespace: 'controller-registry' }, () => {
  const lines: string[] = ['// Auto-generated controller registry - do not edit'];
  lines.push('global.__EE_CONTROLLER_REGISTRY__ = [');
  for (const [i, entry] of registryEntries.entries()) {
    const requirePath = JSON.stringify('./' + entry.relPath);
    lines.push(`  { fullpath: ..., properties: ..., get module() { return require(${requirePath}); } }`);
  }
  lines.push('];');
  return { contents: lines.join('\n'), loader: 'js', resolveDir: controllerDir };
});
```

生成代码示例（假设 controller/ 目录包含 `foo_bar/my-controller.js`）：

```javascript
// Auto-generated controller registry - do not edit
global.__EE_CONTROLLER_REGISTRY__ = [
  { fullpath: "controller/foo_bar/my-controller", properties: ["fooBar", "myController"], get module() { return require("./foo_bar/my-controller.js"); } },
];
```

**关键设计: 懒加载 getter**

注册表中的每个条目使用 `get module() { return require(...); }` 懒加载 getter，而非直接 `require()`。原因：
- 如果在注册时直接 `require()` 所有控制器模块，可能导致初始化顺序问题
- 某些控制器可能有依赖其他模块的初始化代码，在注册阶段执行会导致循环依赖或时序错误
- ee-core 的 `FileLoader.parseFromRegistry()` 只在实际需要该模块时才调用 getter，确保模块按正确顺序加载

**resolveDir 的作用**: 设置为 `controllerDir`（`./electron/controller` 的绝对路径），使得 `require()` 的相对路径能正确解析到控制器目录下的文件。在打包输出中，这些 require 路径会被 esbuild 的 `bundle: true` 机制内联到 main.js 中。

**RegistryEntry 数据结构**:

```typescript
interface RegistryEntry {
  fullpath: string;    // 'controller/subdir/filename' — ee-core 用于匹配
  properties: string[]; // ['fooBar', 'myController'] — 属性路径数组
  relPath: string;      // 'foo_bar/my-controller.js' — require() 相对路径
}
```

### 虚拟模块 2: app:config-registry

**源码**: 第 139-159 行

**onResolve**: 拦截路径 `app:config-registry`，设置命名空间 `config-registry`

**onLoad**: 生成配置注册表代码，设置 `global.__EE_CONFIG_REGISTRY__`

```typescript
build.onLoad({ filter: /.*/, namespace: 'config-registry' }, () => {
  const lines: string[] = ['// Auto-generated config registry - do not edit'];
  lines.push('global.__EE_CONFIG_REGISTRY__ = [');
  for (const [i, entry] of configEntries.entries()) {
    lines.push(`  { filename: ${JSON.stringify(entry.filename)}, get module() { return require(${requirePath}); } }`);
  }
  lines.push('];');
  return { contents: lines.join('\n'), loader: 'js', resolveDir: configDir };
});
```

生成代码示例（假设 config/ 目录包含 `config.default.js` 和 `config.prod.js`）：

```javascript
// Auto-generated config registry - do not edit
global.__EE_CONFIG_REGISTRY__ = [
  { filename: "config.default", get module() { return require("./config.default.js"); } },
  { filename: "config.prod", get module() { return require("./config.prod.js"); } },
];
```

**ConfigEntry 数据结构**:

```typescript
interface ConfigEntry {
  filename: string;  // 'config.default' — ee-core ConfigLoader 用于按名查找
  relPath: string;   // 'config.default.js' — require() 相对路径
}
```

**filename 不含扩展名**: ee-core 的 `ConfigLoader._loadConfig()` 按 filename（不含扩展名）查找配置。例如查找 `'default'` 对应 `config.default.js`，而不是 `'default.js'`。这与 ee-core 的文件名解析逻辑一致。

**同样使用懒加载 getter**: 配置模块也可能有初始化依赖（如 `config.prod.js` 可能依赖 `config.default.js` 的输出），因此也使用 `get module()` 懒加载。

### 虚拟模块 3: app:bundle-entry

**源码**: 第 163-186 行

**onResolve**: 拦截路径 `app:bundle-entry`，设置命名空间 `bundle-entry`

**onLoad**: 生成打包入口代码，按固定顺序加载三个模块

```typescript
build.onLoad({ filter: /.*/, namespace: 'bundle-entry' }, () => {
  const mainRelative = JSON.stringify('./' + path.basename(mainJsPath));
  const contents = [
    '// Auto-generated bundle entry - do not edit',
    `require('app:config-registry');`,
    `require('app:controller-registry');`,
    `require(${mainRelative});`,
  ].join('\n');
  return { contents, loader: 'js', resolveDir: path.dirname(mainJsPath) };
});
```

生成代码示例：

```javascript
// Auto-generated bundle entry - do not edit
require('app:config-registry');
require('app:controller-registry');
require('./main.js');
```

**加载顺序是关键**: config 注册表 → controller 注册表 → main.js

顺序原因：
1. `global.__EE_CONFIG_REGISTRY__` 必须在 main.js 执行前设置好，因为 ee-core 的 `ConfigLoader._loadConfig()` 在生命周期早期就检查此变量
2. `global.__EE_CONTROLLER_REGISTRY__` 必须在 main.js 执行前设置好，因为 ee-core 的 `ControllerLoader.load()` 检查此变量
3. ee-core 的 ElectronEgg 生命周期：loadConfig → loadController → loadElectron，注册表的注入顺序必须与生命周期一致

**resolveDir 的作用**: 设置为 `mainJsPath` 的父目录（`./electron`），使得 `require('./main.js')` 能正确解析到入口文件。

### computeProperties 属性路径计算

**源码**: 第 58-71 行

此函数复现了 ee-core 的 `getProperties()` + `defaultCamelize(caseStyle: 'lower')` 逻辑，确保构建时生成的属性路径与运行时的文件系统扫描结果完全一致。

```
computeProperties(filepath)
  ├── 去除扩展名: 'foo_bar/my-controller.js' → 'foo_bar/my-controller'
  ├── 按 '/' 分割: → ['foo_bar', 'my-controller']
  ├── 对每个片段进行 camelCase 转换:
  │     ├── 规则: _/- 后的字母大写 ('foo-bar' → 'fooBar')
  │     └── 正则: /[_-][a-z]/gi → s.substring(1).toUpperCase()
  └── 首字符小写 (caseStyle: 'lower'): 'FooBar' → 'fooBar'
```

转换示例:

| 文件路径 | 去扩展名 + 分割 | camelCase 转换 | 最终 properties |
|---------|-----------------|---------------|----------------|
| `foo_bar/my-controller.js` | `['foo_bar', 'my-controller']` | `['fooBar', 'myController']` | `['fooBar', 'myController']` |
| `user/UserController.js` | `['user', 'UserController']` | `['user', 'userController']` | `['user', 'userController']` |
| `api/v1/log-handler.js` | `['api', 'v1', 'log-handler']` | `['api', 'v1', 'logHandler']` | `['api', 'v1', 'logHandler']` |

**合法性检查**: 如果片段不匹配 `[a-z][a-z0-9_-]*` 正则，抛出错误。这防止了不合法的文件名（如包含特殊字符）导致运行时加载失败。

### 三个虚拟模块的协作关系

```
app:bundle-entry (虚拟入口)
  ├── require('app:config-registry')
  │     └── 设置 global.__EE_CONFIG_REGISTRY__
  │          ├── [{ filename: 'config.default', get module() { ... } }]
  │          └── [{ filename: 'config.local', get module() { ... } }]
  ├── require('app:controller-registry')
  │     └── 设置 global.__EE_CONTROLLER_REGISTRY__
  │          ├── [{ fullpath: 'controller/xxx', properties: [...], get module() { ... } }]
  │          └── [{ fullpath: 'controller/yyy', properties: [...], get module() { ... } }]
  └── require('./main.js')
        └── Electron 主进程入口代码
              ├── ee-core 启动 → 检测 __EE_CONFIG_REGISTRY__ → loadConfig
              ├── ee-core 启动 → 检测 __EE_CONTROLLER_REGISTRY__ → loadController
              └── ee-core 启动 → 启动 Electron 应用
```

### 与 ee-core 的运行时连接

bundleRegistryPlugin 在构建时生成的全局变量，被 ee-core 在运行时检测和使用：

**ee-core ConfigLoader**:
- 检测 `global.__EE_CONFIG_REGISTRY__` 是否存在
- 存在（打包模式）→ `ConfigLoader._loadConfig()` 从注册表按 filename 查找配置模块，调用 getter 获取模块内容
- 不存在（开发模式）→ 回退到文件系统扫描 + `loadFile()` 加载

**ee-core ControllerLoader**:
- 检测 `global.__EE_CONTROLLER_REGISTRY__` 是否存在
- 存在（打包模式）→ `FileLoader.parseFromRegistry()` 从注册表读取控制器信息
- 不存在（开发模式）→ 回退到 globby 文件系统扫描 + `require()`

**EE_BUNDLED 标记**:
- esbuild 的 `banner` 注入 `process.env.EE_BUNDLED = "true"`
- ee-core 也检测此值作为打包模式的辅助信号

```
构建时 (ee-bin):
  bundleRegistryPlugin → global.__EE_CONTROLLER_REGISTRY__
  bundleRegistryPlugin → global.__EE_CONFIG_REGISTRY__
  esbuild banner → process.env.EE_BUNDLED = "true"

运行时 (ee-core):
  ├── ConfigLoader._loadConfig() → 检测 __EE_CONFIG_REGISTRY__
  │     ├── 存在 → parseFromRegistry (注册表模式)
  │     └── 不存在 → globby + loadFile (文件系统模式)
  ├── ControllerLoader.load() → 检测 __EE_CONTROLLER_REGISTRY__
  │     ├── 存在 → parseFromRegistry (注册表模式)
  │     └── 不存在 → globby + require (文件系统模式)
  └── 整体检测 → process.env.EE_BUNDLED
```

## 关键设计决策

### 1. onStart 每次构建前重新扫描

`onStart` 钩子在每次构建/rebuild 开始时重新扫描目录。这是为了支持 watch 模式：
- 用户可能新增/删除控制器或配置文件
- 重新扫描确保注册表反映最新的文件状态
- 如果只在插件初始化时扫描一次，增删文件后 rebuild 的注册表会过时

### 2. 懒加载 getter 而非直接 require

所有注册表条目使用 `get module() { return require(...); }` 懒加载 getter。直接 `require()` 的问题：
- 所有控制器/配置模块在注册时就被执行，可能导致初始化顺序问题
- 模块间可能存在依赖关系（如控制器依赖服务，服务依赖配置），过早加载会导致循环依赖
- 懒加载确保模块只在实际被使用时才加载，遵循 ee-core 的生命周期顺序

### 3. 配置先于控制器加载

`app:bundle-entry` 的加载顺序：config → controller → main.js。原因：
- ee-core 的生命周期是 loadConfig → loadController → loadElectron
- 配置模块可能包含控制器依赖的设置（如 IPC 通道名称、服务端口等）
- 如果先加载控制器，某些依赖配置的初始化代码可能读到未定义的配置值

### 4. 命名空间隔离

三个虚拟模块使用不同的命名空间（`controller-registry`、`config-registry`、`bundle-entry`），每个 `onLoad` 的 `filter` 匹配整个命名空间下的所有路径。这确保：
- esbuild 不会混淆不同虚拟模块的内容
- 每个虚拟模块的 `resolveDir` 可以独立设置
- onResolve 和 onLoad 的匹配规则不会交叉干扰

### 5. resolveDir 精确设置

- `controller-registry` 的 `resolveDir` = `controllerDir`（`./electron/controller`）
- `config-registry` 的 `resolveDir` = `configDir`（`./electron/config`）
- `bundle-entry` 的 `resolveDir` = `path.dirname(mainJsPath)`（`./electron`）

这确保 require() 调用中的相对路径能正确解析到源文件位置。在 bundle:true 模式下，esbuild 会将这些 require 路径解析为实际文件并打包进输出。

### 6. .jsc 文件支持

globby 的匹配模式包含 `**/*.jsc`，支持已编译的 bytecode 文件。这意味着：
- 如果 controller/ 目录包含 `.jsc` 文件（加密后的控制器），注册表也能正确识别
- `computeProperties` 函数会正确去除 `.jsc` 扩展名计算属性路径
- 这与 ee-core 的加密模式兼容

## API 导出

| 导出项 | 类型 | 说明 |
|--------|------|------|
| `bundleRegistryPlugin` | `(controllerDir: string, mainJsPath: string, configDir: string) => Plugin` | 创建 esbuild 注册表插件 |

**参数说明**:

| 参数 | 类型 | 说明 |
|------|------|------|
| `controllerDir` | `string` | `electron/controller/` 的绝对路径 |
| `mainJsPath` | `string` | 入口文件绝对路径（`main.ts` 或 `main.js`） |
| `configDir` | `string` | `electron/config/` 的绝对路径 |

**内部接口**:

| 接口 | 说明 |
|------|------|
| `RegistryEntry` | 控制器注册表条目：fullpath + properties + relPath |
| `ConfigEntry` | 配置注册表条目：filename + relPath |

**内部函数**:

| 函数 | 说明 |
|------|------|
| `computeProperties(filepath)` | 从文件路径计算属性路径数组，复现 ee-core 的 defaultCamelize(caseStyle:'lower') |

## 模块关系

```
plugins/bundle_registry_plugin.ts
  ├── 外部依赖:
  │     ├── esbuild → Plugin, PluginBuild (插件接口)
  │     ├── globbySync → 目录扫描
  │     └── path → 路径处理
  ├── 调用方:
  │     └── tools/serve.ts → _bundleWithRegistry() 中创建插件实例
  └── 运行时消费者 (ee-core):
        ├── FileLoader.parseFromRegistry() → 读取 __EE_CONTROLLER_REGISTRY__
        ├── ConfigLoader._loadConfig() → 读取 __EE_CONFIG_REGISTRY__
        └── 整体模式检测 → process.env.EE_BUNDLED
```

插件本身不依赖任何 ee-bin 内部模块（helpers, utils 等），是一个独立的 esbuild 插件实现。它的唯一输入是三个目录路径参数，唯一输出是 esbuild Plugin 对象。
