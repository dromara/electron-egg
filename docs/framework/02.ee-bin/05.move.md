# 资源拷贝

## 概述

ee-bin 的资源拷贝模块（move）负责将项目内的文件/目录从源路径复制到目标路径，最常见的用途是将前端构建产物（`./frontend/dist`）拷贝到 Electron 可加载的静态资源目录（`./public/dist`）。

该模块在 `pnpm build-frontend` 后被调用，确保 Electron 应用能够通过 `file://` 或本地 HTTP 协议加载前端页面资源。

## 核心流程

### move 主流程

```
move(options)
  │
  ├─ 加载配置：loadConfig(config) → 获取 binCfg.move
  │
  ├─ 解析 --flag 参数：formatCmds(flag) → flags 数组
  │     例: "frontend_dist" → ["frontend_dist"]
  │     例: "frontend_dist,backend_api" → ["frontend_dist", "backend_api"]
  │
  └─ 逐个处理每个 flag
      │
      ├─ 查找 MoveConfig：moveConfig[f]
      │     配置不存在 → 打印错误，跳过
      │
      ├─ 路径别名解析
      │     source = dist || src   （dist 优先覆盖 src）
      │     destination = target || dest （target 优先覆盖 dest）
      │
      ├─ 校验源路径：fs.existsSync(srcResource)
      │     源不存在 → 警告并跳过（前端可能尚未构建）
      │
      └─ 原子拷贝策略
          │
          ├─ 目标已存在？
          │     ├─ 清理上次失败残留的 .bak → rm(backupDest)
          │     └─ 将当前目标重命名为 .bak 备份 → fs.renameSync(dest, backupDest)
          │
          ├─ 执行拷贝：copyDirSync(src, dest)
          │     ├─ 拷贝失败 → 恢复 .bak 备份（删除残缺 dest，重命名 .bak 回 dest）
          │     └─ 拷贝成功 → 删除 .bak 备份 → rm(backupDest)
          │
          └─ 完成日志输出
```

相关源文件：`packages/ee-bin/src/tools/move.ts` 中 `move()` 函数

### 原子拷贝策略细节

```
destResource 已存在时的拷贝流程：

  原始状态:  destResource (旧内容)
             backupDest (.bak) ← 上次失败残留，先删除

  步骤 1:   destResource → backupDest   (重命名，旧内容安全保存)
  步骤 2:   srcResource → destResource   (拷贝新内容)

  成功时:
  步骤 3:   rm(backupDest)               (删除旧备份)

  失败时:
  步骤 3:   rm(destResource)             (删除残缺的新拷贝)
  步骤 4:   backupDest → destResource    (恢复旧内容)
  步骤 5:   抛出原始错误
```

此策略保证：拷贝失败时目标路径不会处于空/残缺状态，旧内容始终可恢复。

## 关键设计决策

### 1. "Move" 实为 Copy（非移动）

模块命名为 "move"，但实际执行的是**拷贝操作**——源文件在拷贝后保持不变。这是有意为之的设计：

- 前端 `dist` 目录可能被后续开发重新使用，不应被删除
- "move" 在命令语义上表示"资源转移到位"，而非文件系统层面的 `mv` 操作

### 2. 原子性保护策略

采用 `.bak` 备份 → 拷贝 → 删除备份的三步策略，而非直接覆盖：

- 直接覆盖的危险场景：拷贝过程中如果出错（磁盘空间不足、权限问题等），目标路径可能只剩下部分文件
- 备份恢复策略确保：无论拷贝成功还是失败，目标路径始终处于完整状态

### 3. 源不存在时跳过而非报错

当源路径不存在时（如前端尚未构建），模块输出警告并跳过，而非抛出错误终止流程。原因：

- 用户可能在只构建 Electron 时不执行 move
- 某些 flag 对应的源路径可能仅在特定场景下才存在

### 4. 路径别名机制（dist/target）

`MoveConfig` 支持 `dist` 和 `target` 字段分别覆盖 `src` 和 `dest`。这允许用户在不修改默认配置的情况下，通过设置别名字段来指定不同的源或目标路径：

```js
// 默认配置
move: {
  frontend_dist: {
    src: './frontend/dist',
    dest: './public/dist',
  }
}

// 用户自定义覆盖（只改目标路径，不改源路径）
move: {
  frontend_dist: {
    src: './frontend/dist',
    dest: './public/dist',
    target: './public/custom-dist',  // target 覆盖 dest
  }
}
```

## API 导出

| 导出 | 类型 | 说明 |
|------|------|------|
| `move(options)` | function | 主入口函数，执行资源拷贝操作 |

**MoveOptions 接口**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `config` | `string?` | 自定义 bin.js 配置文件路径 |
| `flag` | `string?` | 逗号分隔的 move 配置键名 |

**MoveConfig 接口**（定义于 `packages/ee-bin/src/types/config.ts`）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `src` | `string` | 源路径（相对于项目根目录） |
| `dest` | `string` | 目标路径（相对于项目根目录） |
| `dist` | `string?` | 源路径别名，设置后覆盖 `src` |
| `target` | `string?` | 目标路径别名，设置后覆盖 `dest` |

## 模块关系

```
move.ts
  ├─ 引入 helpers.ts
  │     ├─ chalk        → 日志颜色输出
  │     ├─ copyDirSync  → 递归目录拷贝核心实现
  │     └─ formatCmds   → 逗号分隔的 flag 字符串解析
  │
  ├─ 引入 utils.ts
  │     ├─ loadConfig   → 加载并合并 bin.js 配置
  │     └─ rm           → 递归强制删除（清理备份和失败残缺）
  │
  └─ 引用 types/config.ts
        └─ MoveConfig    → move 配置项类型定义
```

**典型调用链**：

```
CLI 命令 → move({ config, flag: 'frontend_dist' })
  → loadConfig() → extend(deepMerge) → binCfg
  → formatCmds('frontend_dist') → ['frontend_dist']
  → 查找 moveConfig.frontend_dist → { src, dest }
  → 路径别名解析 → source, destination
  → copyDirSync(source, destination)
```
