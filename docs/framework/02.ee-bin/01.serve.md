# ServeProcess 核心（dev/build/start/exec 全流程）

## 概述

ServeProcess 是 ee-bin 的核心调度类，管理开发、构建、启动、自定义执行的全生命周期。它是整个 CLI 中最复杂的模块，负责 Electron 代码打包、子进程管理、文件监控、package.json 动态切换等关键功能。

**源文件**: `packages/ee-bin/src/tools/serve.ts`

导出方式：单例 `serveProcess` + 类 `ServeProcess`

```typescript
export const serveProcess = new ServeProcess();
export { ServeProcess };
```

## 核心流程

### dev 模式完整流程

**源码**: `ServeProcess.dev()` (第 114-183 行)

```
dev(options)
  ├── 1. 设置 NODE_ENV=dev
  ├── 2. loadConfig(config) → 加载 bin.js 配置
  ├── 3. 解析 command（默认启动所有 dev 配置中的服务）
  ├── 4. 如果包含 electron 命令:
  │     ├── bundle(binCfg.build.electron) → 打包 Electron 代码
  │     ├── _switchPkgMain() → 切换 package.json main 字段
  │     └── 如果 watch=true:
  │           ├── chokidarWatch('./electron') → 监控文件变化
  │           ├── 变化事件 → debounce(delay) → 防抖
  │           ├── re-bundle → 重新打包
  │           ├── tree-kill 旧进程 → 终止旧 Electron 进程
  │           └── multiExec 重新启动 → 重启 Electron
  ├── 5. multiExec(opt) → 启动所有命令（前端 + Electron）
```

**watch 模式关键设计**: 使用 `chokidarWatch` 监控 `./electron` 目录，文件变化触发 debounce（默认 1000ms），防抖后执行 re-bundle + kill 旧进程 + 重新 spawn。这避免了频繁文件变化导致多次重复构建。

### build 模式完整流程

**源码**: `ServeProcess.build()` (第 223-259 行)

```
build(options)
  ├── 1. 设置 NODE_ENV=prod（或用户指定的 --env）
  ├── 2. loadConfig(config) → 加载配置
  ├── 3. 解析 cmds（必须有值，否则打印警告并返回）
  ├── 4. 如果 cmds 包含 'electron':
  │     ├── bundle(binCfg.build.electron) → 打包 Electron
  │     ├── 从 commands 列表中删除 'electron'（仅触发打包，不启动进程）
  │     └── _switchPkgMain() → 切换 package.json main
  ├── 5. multiExec(opt) → 执行剩余命令（如 frontend, win64, mac 等）
  └── 6. _restorePkgMain() → 构建完成后恢复 package.json
```

**关键设计**: `electron` 在 build 命令中仅触发打包，不启动 Electron 进程，因此从命令列表中移除。其余命令（如 `frontend`、平台打包命令）由 `multiExec` 执行。

### start 模式流程

**源码**: `ServeProcess.start()` (第 189-204 行)

```
start(options)
  ├── 1. 设置 NODE_ENV=prod
  ├── 2. loadConfig(config) → 加载配置
  ├── 3. multiExec(opt) → 直接执行 electron . --env=prod
```

start 模式不做打包，前提是项目已经通过 build 命令构建完成。

### exec 模式流程

**源码**: `ServeProcess.exec()` (第 262-274 行)

```
exec(options)
  ├── 1. loadConfig(config) → 加载配置
  ├── 2. 从 binCfg.exec 获取用户自定义命令配置
  ├── 3. multiExec(opt) → 执行用户定义的命令
```

### multiExec 子进程执行

**源码**: `ServeProcess.multiExec()` (第 285-359 行)

```
multiExec(opt)
  ├── 遍历 commands 列表
  ├── 对每个命令:
  │     ├── 跳过无效/未配置的命令
  │     ├── dev 模式下跳过 frontend + file:// 协议的组合
  │     ├── sync=true: crossSpawnSync (阻塞执行，不存入 execProcess)
  │     │     ├── 错误处理: spawn 失败 → throw Error
  │     │     └── 错误处理: 非零退出码 → throw Error
  │     └── sync=false: crossSpawn (异步执行，存入 execProcess)
  │           ├── 存入 this.execProcess[cmd] → 用于 SIGINT/SIGTERM 时 kill
  │           ├── on('error') → 删除 execProcess 记录，打印错误
  │           └── on('exit') → dev 模式打印退出提示；非 dev 模式打印完成提示
```

**MAX_BUFFER**: 子进程缓冲区大小设为 1GB（`1024 * 1024 * 1024`），防止大型构建命令（如 electron-builder）输出被截断。

**file:// 协议跳过**: dev 模式中，如果前端配置的 `protocol` 为 `file://`，表示前端已通过 HTTP 开发服务器提供服务，不需要单独启动前端进程。

### bundle 打包流程

**源码**: `ServeProcess.bundle()` (第 370-383 行)

```
bundle(bundleConfig)
  ├── 1. 清空输出目录 (rm outdir)
  ├── 2. bundleType === 'copy':
  │     └── copyDirSync(electronDir, outdir) → 直接复制整个目录
  └── 3. bundleType === 'bundle' (或默认):
        └── _bundleWithRegistry(bundleConfig) → esbuild 打包
```

### _bundleWithRegistry 详细流程

**源码**: `ServeProcess._bundleWithRegistry()` (第 439-514 行)

```
_bundleWithRegistry(bundleConfig)
  ├── 1. 检测入口文件类型
  │     ├── main.ts 存在 → TypeScript 入口 → isTypeScript=true
  │     └── main.js 存在 → JavaScript 入口 → isTypeScript=false
  ├── 2. 构建框架外部依赖列表
  │     └── frameworkExternal = ['ee-core', 'ee-bin', 'electron',
  │           'better-sqlite3', 'proxy-agent', 'pino-roll', 'pino-pretty']
  ├── 3. 创建 bundleRegistryPlugin(controllerDir, entryMain, configDir)
  ├── 4. 构建 esbuild BuildOptions
  │     ├── _resolveBaseBuildOptions(bundleConfig) → 共享编译设置
  │     ├── entryPoints: ['app:bundle-entry'] → 虚拟入口
  │     ├── bundle: true → 打包所有可达代码
  │     ├── packages: 'external' → 自动排除所有 node_modules
  │     ├── external: frameworkExternal + userExternal
  │     ├── banner: process.env.EE_BUNDLED = "true" → 标记打包模式
  │     └── plugins: [plugin] → 注册表插件
  ├── 5. await esbuild.build(options) → 执行构建
  ├── 6. 重命名输出文件
  │     ├── app_bundle-entry.js → main.js
  │     └── app_bundle-entry.js.map → main.js.map (如存在)
  └── 7. _copyUnbundledFiles(cwd, outdir, bundleConfig) → 复制非打包文件
```

### _resolveBaseBuildOptions 共享编译设置

**源码**: `ServeProcess._resolveBaseBuildOptions()` (第 396-422 行)

集中管理所有 esbuild 编译设置，确保主包 `main.js` 和逐文件转译的 `jobs`/`copy` 代码使用完全一致的编译参数：

```
_resolveBaseBuildOptions(bundleConfig)
  ├── 判断 isDev (NODE_ENV === dev || local)
  ├── sourcemap 逻辑:
  │     ├── 'inline' 或 true → 强制 inline sourcemap
  │     ├── 'external' → 生成独立 .map 文件
  │     └── false 或 undefined (默认) → 自动模式: dev→inline, prod→off
  ├── format: bundleConfig.format || 'cjs'
  └── 返回 BuildOptions:
        ├── platform: 'node'
        ├── target: 'node20'
        ├── format, sourcemap
        ├── minify, keepNames
        ├── drop, legalComments, define
```

### _copyUnbundledFiles 两层策略

**源码**: `ServeProcess._copyUnbundledFiles()` (第 576-607 行)

```
_copyUnbundledFiles(cwd, outdir, bundleConfig)
  ├── 1. 获取共享编译设置: _resolveBaseBuildOptions(bundleConfig)
  ├── 2. 处理 preload/bridge.*:
  │     ├── globbySync('preload/bridge.{ts,js,mts,cts,tsx,jsx}')
  │     └── _transpileDir(bridgeSrc, bridgeDest, baseOptions) → 转译为 bridge.js
  ├── 3. 处理 copy targets:
  │     ├── 去重: [...new Set(['jobs', ...(bundleConfig.copy || [])])]
  │     └── 对每个 target: _transpileDir(src, dest, baseOptions)
  │           ├── 脚本文件 → esbuild 逐文件转译 (bundle:false)
  │           └── 非脚本文件 → copyFileSync 原样复制
```

### _transpileDir 逐文件转译

**源码**: `ServeProcess._transpileDir()` (第 531-566 行)

```
_transpileDir(src, dest, baseOptions)
  ├── 判断 src 类型:
  │     ├── 单文件 → transpileFile(srcFile, destFile) → 返回
  │     └── 目录 → globbySync('**/*') 遍历所有文件
  ├── 对每个文件:
  │     ├── 脚本扩展名 (.ts/.js/.mts/.cts/.tsx/.jsx):
  │     │     └── esbuild.build({
  │     │           ...baseOptions,  ← 共享编译设置保持一致
  │     │           entryPoints: [srcFile],
  │     │           outfile: destFile + '.js',  ← 输出为 .js
  │     │           bundle: false,  ← 逐文件转译，不打包
  │     │           logLevel: 'silent'
  │     │         })
  │     └── 非脚本扩展名 (.json 等):
  │           └── mkdirSync (创建目标目录) → copyFileSync (原样复制)
```

### package.json main 字段切换

**源码**: `ServeProcess._switchPkgMain()` (第 616-626 行), `_restorePkgMain()` (第 634-642 行)

```
_switchPkgMain()
  ├── 读取 package.json
  ├── 保存原始 pkg.main → this.originalPkgMain
  ├── 设置 pkg.main = './public/electron/main.js'
  └── writeJsonSync 写回 package.json

_restorePkgMain()
  ├── 如果 originalPkgMain 有值:
  │     ├── 读取 package.json
  │     ├── 恢复 pkg.main = this.originalPkgMain
  │     ├── writeJsonSync 写回
  │     └── 清空 this.originalPkgMain = undefined
```

### 进程生命周期管理

**源码**: `ServeProcess._init()` (第 65-75 行), `_closeProcess()` (第 85-99 行)

```
_init() → constructor 中调用
  ├── process.on('SIGINT') → _closeProcess()
  └── process.on('SIGTERM') → _closeProcess()

_closeProcess()
  ├── 遍历 execProcess → kill(p.pid) 终止所有子进程
  ├── _restorePkgMain() → 恢复 package.json
  ├── sleep(500) → 等待子进程退出
  └── process.exit(0)
```

**500ms sleep 说明**: 理想方案是监听每个子进程的 exit 事件后再退出，但多进程竞态和嵌套进程使实现过于复杂。500ms 是一个折衷方案，大多数子进程能在此时间内完成退出。

## 关键设计决策

### 1. 单例模式而非每次创建实例

`serveProcess` 作为单例导出，四个命令共享同一实例。原因：
- `execProcess` 子进程追踪表需要在 SIGINT/SIGTERM 时跨命令访问
- `originalPkgMain` 的保存和恢复需要在 dev 模式的信号处理器中使用
- 信号处理器只需注册一次

### 2. execProcess 只追踪异步进程

`execProcess` 只存储异步 `ChildProcess` 实例，同步进程（`sync: true`）不存入。原因：
- 同步进程通过 `crossSpawnSync` 执行，执行完毕后进程已结束，无需追踪
- 只有异步进程需要在退出时被 kill

### 3. electron 在 build 中仅触发打包

`build` 命令中，如果 `cmds` 包含 `electron`，仅触发 `bundle()` 打包，然后从命令列表中移除 `electron`。原因：
- `electron` 在构建场景下意味着"打包 Electron 代码"，不是"启动 Electron 进程"
- 打包后的代码由 `start` 命令或用户手动启动

### 4. debounce 防抖机制

watch 模式使用 `setTimeout` 实现 debounce，默认延迟 1000ms。文件快速连续变化（如 IDE 保存触发多个文件写入）只触发一次重建，避免资源浪费和进程频繁重启。

### 5. tree-kill 强制终止

watch 模式重启 Electron 时使用 `tree-kill`（SIGKILL）而非简单的 `process.kill()`。原因：
- Electron 进程可能产生子进程（如 GPU 进程、渲染进程），需要终止整个进程树
- SIGKILL 是强制终止信号，确保旧进程完全退出

### 6. banner 注入 EE_BUNDLED 标记

`_bundleWithRegistry` 通过 esbuild 的 `banner` 选项注入 `process.env.EE_BUNDLED = "true"`。ee-core 运行时检测此值：
- 存在 → 使用注册表模式（`FileLoader.parseFromRegistry()`）
- 不存在 → 使用文件系统扫描模式（globby + `require()`）

### 7. packages: 'external' + explicit external 列表

esbuild 配置使用 `packages: 'external'`（自动排除所有 node_modules 包），再加上显式的 `frameworkExternal` 列表。两层设计的原因：
- `packages: 'external'` 是性能优化——避免 esbuild 扫描和解析所有 node_modules
- 显式列表是文档性质——明确标注哪些包必须作为外部依赖，便于维护者理解

## API 导出

| 导出项 | 类型 | 说明 |
|--------|------|------|
| `serveProcess` | `ServeProcess` 单例实例 | dev/build/start/exec 四个命令的执行入口 |
| `ServeProcess` | 类 | 允许外部创建独立实例（如测试场景） |

**ServeProcess 类关键字段**:

| 字段 | 类型 | 说明 |
|------|------|------|
| `execProcess` | `Record<string, ChildProcess>` | 异步子进程追踪表 |
| `originalPkgMain` | `string | undefined` (private) | package.json main 字段原始值 |

**ServeProcess 类关键方法**:

| 方法 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `dev(options)` | `ServeOptions` | `Promise<void>` | 开发模式 |
| `build(options)` | `ServeOptions` | `Promise<void>` | 构建模式 |
| `start(options)` | `ServeOptions` | `Promise<void>` | 生产启动 |
| `exec(options)` | `ServeOptions` | `void` | 自定义命令执行 |
| `multiExec(opt)` | `{binCmd, binCmdConfig, command}` | `void` | 多命令执行器 |
| `bundle(bundleConfig)` | `BundleConfig` | `Promise<void>` | 打包入口 |
| `_bundleWithRegistry(bundleConfig)` | `BundleConfig` (private) | `Promise<void>` | esbuild 打包 |
| `_resolveBaseBuildOptions(bundleConfig)` | `BundleConfig` (private) | `BuildOptions` | 共享编译设置 |
| `_copyUnbundledFiles(cwd, outdir, bundleConfig)` | (private) | `Promise<void>` | 复制非打包文件 |
| `_transpileDir(src, dest, baseOptions)` | (private) | `Promise<void>` | 逐文件转译 |
| `_switchPkgMain()` | (private) | `void` | 切换 package.json main |
| `_restorePkgMain()` | (private) | `void` | 恢复 package.json main |
| `_closeProcess()` | (private) | `Promise<void>` | 关闭所有子进程 |
| `_init()` | (private) | `void` | 注册信号处理器 |

## 模块关系

```
tools/serve.ts (ServeProcess)
  ├── lib/helpers.ts → createDebug, chalk, copyDirSync, formatCmds
  ├── lib/utils.ts → loadConfig, readJsonSync, writeJsonSync, rm, toArray
  ├── plugins/bundle_registry_plugin.ts → bundleRegistryPlugin
  ├── tools/move.ts → 无直接依赖（通过 CLI 路由调用）
  ├── tools/encrypt.ts → 无直接依赖（通过 CLI 路由调用）
  ├── 外部依赖:
  │     ├── esbuild → build, BuildOptions (打包引擎)
  │     ├── chokidar → watch (文件监控)
  │     ├── tree-kill → kill (进程树终止)
  │     ├── cross-spawn → crossSpawn, crossSpawnSync (跨平台子进程)
  │     └── globbySync → 文件扫描
  └── 类型依赖:
        └── types/config.ts → ExecConfig, BundleConfig, ServeOptions
```

ServeProcess 是 ee-bin 中唯一与其他核心模块（bundleRegistryPlugin）有直接依赖关系的工具模块。其余工具模块（move, encrypt, iconGen, incrUpdater）是独立函数，仅通过 CLI 入口路由调用。
