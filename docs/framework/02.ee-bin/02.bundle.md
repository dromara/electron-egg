# esbuild 打包流程

## 概述

ee-bin 的打包流程是 Electron 主进程代码从源码到生产产物的核心转换环节。基于 esbuild 实现，采用"虚拟入口 + 注册表插件 + 逐文件转译"三层架构，将 `electron/` 目录下的控制器、服务、配置等代码打包为单一 `main.js`，同时将不可打包的文件（jobs、preload）通过逐文件转译保留为独立文件。

**核心源文件**: `packages/ee-bin/src/tools/serve.ts`

**插件源文件**: `packages/ee-bin/src/plugins/bundle_registry_plugin.ts`

**配置默认值**: `packages/ee-bin/src/config/bin_default.ts`

**类型定义**: `packages/ee-bin/src/types/config.ts`（BundleConfig 接口）

## 核心流程

### 打包总体架构

```
ServeProcess.bundle(bundleConfig)
  ├── 清空输出目录 (rm ./public/electron)
  ├── bundleType === 'copy':
  │     └── copyDirSync(electron/, public/electron/) → 整目录复制
  └── bundleType === 'bundle' (默认):
        └── _bundleWithRegistry(bundleConfig) → esbuild 打包流程
              ├── 1. 检测入口类型
              ├── 2. 创建注册表插件
              ├── 3. esbuild 构建
              ├── 4. 重命名输出
              └── 5. 复制非打包文件
```

### 1. 入口类型检测

**源码**: `ServeProcess._bundleWithRegistry()` 第 443-447 行

```typescript
const mainJsPath = path.join(cwd, ELECTRON_DIR, 'main.js');
const mainTsPath = path.join(cwd, ELECTRON_DIR, 'main.ts');
const isTypeScript = fs.existsSync(mainTsPath);
const entryMain = isTypeScript ? mainTsPath : mainJsPath;
```

检测逻辑：
- 若 `electron/main.ts` 存在 → TypeScript 入口，esbuild 会自动处理 TS 解析
- 若 `electron/main.ts` 不存在 → 使用 `electron/main.js` 作为入口
- 优先级：main.ts > main.js

### 2. 注册表插件创建

**源码**: `ServeProcess._bundleWithRegistry()` 第 468 行

```typescript
const plugin = bundleRegistryPlugin(controllerDir, entryMain, configDir);
```

三个参数：
- `controllerDir`: `./electron/controller` 的绝对路径
- `entryMain`: 入口文件绝对路径（main.ts 或 main.js）
- `configDir`: `./electron/config` 的绝对路径

插件生成三个虚拟模块，详见 [03.registry-plugin.md](03.registry-plugin.md)。

### 3. esbuild 构建配置

**源码**: `ServeProcess._bundleWithRegistry()` 第 470-492 行

esbuild BuildOptions 的完整构建过程：

```
BuildOptions 构建
  ├── 共享编译设置 (_resolveBaseBuildOptions)
  │     ├── platform: 'node'
  │     ├── target: 'node20'
  │     ├── format: bundleConfig.format || 'cjs'
  │     ├── sourcemap: 自动模式 (dev→inline, prod→off)
  │     ├── minify, keepNames, drop, legalComments, define
  ├── 打包专属设置
  │     ├── entryPoints: ['app:bundle-entry'] → 虚拟入口
  │     ├── bundle: true → 打包所有可达代码
  │     ├── packages: 'external' → 排除所有 node_modules
  │     ├── outdir: './public/electron'
  │     ├── external: frameworkExternal + userExternal
  │     ├── banner: { js: 'process.env.EE_BUNDLED = "true";' }
  │     ├── plugins: [bundleRegistryPlugin]
  │     └── logLevel: 'info'
```

**框架外部依赖列表** (`frameworkExternal`):

| 包名 | 排除原因 |
|------|---------|
| `ee-core` | `child_process.fork()` 需要其入口文件作为磁盘上的真实文件 |
| `ee-bin` | CLI 构建工具，不属于 Electron 运行时 |
| `electron` | 原生模块，无法被 esbuild 打包 |
| `better-sqlite3` | 原生模块（C++ addon），无法被 esbuild 打包 |
| `proxy-agent` | 依赖网络/系统原生 API |
| `pino-roll` | 依赖 fs 操作，打包后路径错误 |
| `pino-pretty` | 依赖 fs 操作，打包后路径错误 |

**packages: 'external' 说明**: 这个选项告诉 esbuild 将所有通过 `import/require` 引用的 npm 包视为外部依赖。与显式 `external` 列表的双重设计：
- `packages: 'external'` 是性能优化——避免扫描所有 node_modules
- 显式列表是文档性质——明确标注关键外部依赖

### 4. 输出文件重命名

**源码**: `ServeProcess._bundleWithRegistry()` 第 498-508 行

```
构建完成后:
  ├── app_bundle-entry.js → main.js  (esbuild 将 ':' 替换为 '_')
  ├── app_bundle-entry.js.map → main.js.map  (如存在 sourcemap)
```

esbuild 在生成输出文件名时，会将虚拟模块名 `app:bundle-entry` 中的 `:` 替换为 `_`，因此输出文件名为 `app_bundle-entry.js`。需要手动重命名为 `main.js` 以匹配 Electron 的入口文件名约定。

### 5. _copyUnbundledFiles 非打包文件复制

**源码**: `ServeProcess._copyUnbundledFiles()` 第 576-607 行

两层策略：

```
_copyUnbundledFiles(cwd, outdir, bundleConfig)
  ├── 层1: preload/bridge.* → bridge.js
  │     ├── globbySync 搜索 bridge.{ts,js,mts,cts,tsx,jsx}
  │     └── _transpileDir → 转译为 bridge.js
  │     └── 原因: BrowserWindow preload 必须是独立文件，不能打包进 main.js
  └── 层2: copy targets (去重处理)
        ├── 默认: ['jobs'] → child_process.fork 需要独立文件
        ├── 用户: bundleConfig.copy → 如 ['assets', 'data/db.json']
        ├── 去重: [...new Set(['jobs', ...(copy || [])])]
        └── 对每个 target: _transpileDir(src, dest, baseOptions)
```

**为什么 preload/bridge.js 不能打包**:
- BrowserWindow 的 `webPreferences.preload` 选项要求 preload 脚本是磁盘上的独立文件
- Electron 通过文件路径直接加载 preload 脚本，打包进 main.js 后路径无效

**为什么 jobs 不能打包**:
- ee-core 通过 `child_process.fork()` 启动 jobs，`fork()` 需要磁盘上的真实文件路径
- jobs 文件由 ee-core 的 `require()` 加载器在 fork 子进程中执行

### 6. _transpileDir 逐文件转译机制

**源码**: `ServeProcess._transpileDir()` 第 531-566 行

```
_transpileDir(src, dest, baseOptions)
  ├── src 是文件 → transpileFile → 返回
  ├── src 是目录 → globbySync('**/*') 遍历
  ├── 对每个文件:
  │     ├── 脚本扩展名 (.ts/.js/.mts/.cts/.tsx/.jsx):
  │     │     └── esbuild.build({
  │     │           ...baseOptions,   ← 与 main.js 完全一致的编译设置
  │     │           entryPoints: [srcFile],
  │     │           outfile: destFile → .js,  ← 输出扩展名改为 .js
  │     │           bundle: false,    ← 逐文件转译，不打包依赖
  │     │           logLevel: 'silent'
  │     │         })
  │     │     └── 原因: bundle:false 确保 import/require 保持为运行时调用
  │     └── 非脚本扩展名 (.json, .html 等):
  │           └── mkdirSync (创建目录) → copyFileSync (原样复制)
```

**bundle:false 的关键意义**:
- 脚本文件中的 `import/require` 不会被打包进文件，而是保持为运行时调用
- 相对导入（`./foo`）在运行时解析到同级转译后的 `.js` 文件
- 包导入（`ee-core/*`）在运行时从 `node_modules` 解析
- 这确保了 `child_process.fork()` 能正确加载 jobs 文件

### _resolveBaseBuildOptions 共享编译设置详解

**源码**: `ServeProcess._resolveBaseBuildOptions()` 第 396-422 行

此方法集中管理所有编译设置，确保 `main.js`（打包输出）和 jobs/preload/copy 文件（逐文件转译输出）使用完全一致的编译参数。不一致会导致运行时问题（如 main.js 使用 CJS 格式而 jobs 使用 ESM 格式）。

```
_resolveBaseBuildOptions(bundleConfig)
  ├── isDev 判断: NODE_ENV === 'dev' || 'local'
  ├── sourcemap 自动逻辑:
  │     ├── bundleConfig.sourcemap === 'inline' 或 true → 强制 inline
  │     ├── bundleConfig.sourcemap === 'external' → 生成独立 .map 文件
  │     └── bundleConfig.sourcemap === false/undefined → 自动:
  │           ├── isDev → 'inline' (可调试)
  │           └── !isDev → false (更小输出)
  ├── format: bundleConfig.format || 'cjs' (推荐 CJS 用于 Electron)
  └── 返回:
        ├── platform: 'node'
        ├── target: 'node20'
        ├── format, sourcemap
        ├── minify: bundleConfig.minify ?? false
        ├── keepNames: bundleConfig.keepNames ?? false
        ├── drop: bundleConfig.drop (可选)
        ├── legalComments: bundleConfig.legalComments (可选)
        └── define: bundleConfig.define (可选)
```

**sourcemap 自动模式详细说明**:

| 环境 | 默认行为 | 用户覆盖方式 |
|------|---------|-------------|
| dev | inline sourcemap（可调试） | 设置 `sourcemap: 'external'` 生成 .map 文件 |
| prod | 关闭 sourcemap（更小输出） | 设置 `sourcemap: 'inline'` 或 `true` 开启调试 |

### 输出目录结构

打包完成后，`public/electron/` 的目录结构：

```
public/electron/
├── main.js              # 打包后的主进程入口（包含控制器、服务、配置）
├── main.js.map          # sourcemap 文件（可选，取决于配置）
├── preload/
│   └── bridge.js        # 转译后的 preload 脚本（源文件可能是 .ts）
└── jobs/                # 逐文件转译后的 jobs 目录
    ├── exampleJob.js    # .ts 源 → .js 输出
    └── childPoolJob.js  # .ts 源 → .js 输出
```

加上用户定义的 `bundleConfig.copy` 目标（如 `assets/`, `data/`），脚本文件转译为 `.js`，非脚本文件原样复制。

## BundleConfig 配置选项

**类型定义**: `packages/ee-bin/src/types/config.ts` 第 69-101 行

**默认值**: `packages/ee-bin/src/config/bin_default.ts` 第 62-69 行

| 选项 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `bundleType` | `'bundle' \| 'copy'` | `'bundle'` | 打包模式：bundle 使用 esbuild 打包为单文件；copy 直接复制目录 |
| `external` | `string[]` | `[]` | 用户定义的 esbuild 外部包（框架外部依赖自动添加） |
| `sourcemap` | `boolean \| 'inline' \| 'external'` | `false` | false=自动模式（dev→inline, prod→off）；强制指定用 'inline'/'external' |
| `minify` | `boolean` | `false` | 是否压缩代码（推荐生产环境启用） |
| `keepNames` | `boolean` | `false` | 压缩时保留函数/类名（便于错误追踪） |
| `drop` | `('console' \| 'debugger')[]` | `[]` | 移除的语句类型（如 `['console', 'debugger']`） |
| `legalComments` | `'inline' \| 'eof' \| 'none'` | `'none'` | 许可证注释处理方式 |
| `define` | `Record<string, string>` | `{}` | 编译时常量定义（如 `{ 'process.env.VERSION': '"1.0"' }`） |
| `copy` | `string[]` | `[]` | 需从 electron/ 复制到输出的额外目录/文件 |
| `format` | `'cjs' \| 'esm'` | `'cjs'` | 输出格式：CJS 推荐 Electron 主进程；ESM 需全部业务代码兼容 |

## 关键设计决策

### 1. 虚拟入口而非直接入口

为什么不直接用 `electron/main.js` 作为 esbuild 入口？因为：
- `main.js` 依赖控制器和配置模块，打包时需要先注册这些模块到全局变量
- 虚拟入口 `app:bundle-entry` 按顺序加载：config 注册表 → controller 注册表 → main.js
- 这个加载顺序对应 ee-core 的生命周期：loadConfig → loadController → loadElectron

### 2. banner 注入而非 define 标记

为什么不使用 esbuild 的 `define` 选项来设置 `EE_BUNDLED`？因为：
- `define` 只替换 `process.env.X` 形式的标识符，而 ee-core 运行时可能通过 `process.env.EE_BUNDLED` 的不同方式检测
- `banner` 在文件顶部注入 `process.env.EE_BUNDLED = "true"`，确保在任何代码执行前就设置好环境变量
- banner 方式更可靠——无论 ee-core 如何检测（直接读取 `process.env` 或代码中的引用），都能生效

### 3. 逐文件转译而非复制

为什么 jobs 和 preload 文件不直接复制，而是通过 esbuild 转译？因为：
- 源文件可能是 TypeScript（`.ts`），Node.js 无法直接执行 `.ts` 文件
- 转译确保所有脚本文件统一为 `.js` 格式，与 `main.js` 使用相同的编译设置
- `bundle: false` 保持运行时依赖解析，不影响文件的功能正确性

### 4. 清空输出目录再构建

`bundle()` 方法在每次构建前 `rm(outdir)` 清空输出目录。原因：
- 避免旧文件残留导致运行时加载错误的模块
- 确保构建产物是干净的、完整的
- watch 模式每次重建都会清空输出，保证一致性

### 5. copy 模式作为备选方案

`bundleType: 'copy'` 直接复制整个 `electron/` 目录，不使用 esbuild。适用场景：
- 用户代码不适合打包（如使用了 esbuild 无法处理的动态 require）
- 开发调试时需要保持文件独立性
- copy 模式不做任何代码转换，仅文件复制

## 模块关系

```
打包流程涉及的模块:
  ├── tools/serve.ts
  │     ├── bundle() → 打包入口
  │     ├── _bundleWithRegistry() → 核心打包逻辑
  │     ├── _resolveBaseBuildOptions() → 共享编译设置
  │     ├── _copyUnbundledFiles() → 非打包文件复制
  │     └── _transpileDir() → 逐文件转译
  ├── plugins/bundle_registry_plugin.ts
  │     └── bundleRegistryPlugin() → 注册表插件
  ├── config/bin_default.ts
  │     └── build.electron → BundleConfig 默认值
  ├── types/config.ts
  │     └── BundleConfig → 类型定义
  └── 外部依赖:
        ├── esbuild → build, BuildOptions (打包引擎)
        └── globbySync → 文件扫描 (_transpileDir, _copyUnbundledFiles)

ee-core 运行时连接:
  ├── process.env.EE_BUNDLED → 检测打包模式
  ├── global.__EE_CONTROLLER_REGISTRY__ → FileLoader.parseFromRegistry()
  ├── global.__EE_CONFIG_REGISTRY__ → ConfigLoader._loadConfig()
  └── child_process.fork() → 加载 jobs/ 目录中的文件
```
