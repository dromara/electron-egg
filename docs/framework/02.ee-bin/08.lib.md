# 内部库（extend/helpers/utils）

## 概述

ee-bin 的内部库模块提供 CLI 工具各功能模块共用的基础工具函数，涵盖对象深合并、类型检测、彩色输出、目录拷贝、调试日志、命令解析、配置加载和文件 I/O 八个核心能力。

各子模块职责分明：
- **extend**：自定义深合并（替代 lodash.merge），数组全量覆盖而非逐索引合并，防原型链污染
- **helpers**：轻量替代库集合——is（替代 is-type-of）、chalk（替代 chalk npm）、copyDirSync（替代 fs-extra）、createDebug（替代 debug npm）、formatCmds（命令解析）
- **utils**：核心工具集——配置加载/合并、文件读取（支持 JSON5）、JSON 文件读写、递归删除、数组归一化

这些内部库的设计动机：ee-bin 是 CLI 工具，应保持最小依赖体积。每个替代实现只保留 ee-bin 实际需要的功能子集，而非引入完整的 npm 包。

## 核心流程

### extend 深合并流程

```
extend(deep, target, ...sources)
  │
  ├─ 创建结果副本：result = { ...target }（不修改原始 target）
  │
  ├─ 逐个遍历 source 对象
  │     source 为 null/undefined → 跳过
  │
  └─ 逐个遍历 source 的 key
      ├─ key === '__proto__' → 跳过（防原型链污染）
      ├─ key === 'constructor' → 跳过（防构造函数覆盖）
      ├─ key === 'prototype' → 跳过（防全局原型链污染）
      ├─ result === copy → 跳过（防自引用无限递归）
      │
      ├─ deep=true && isPlainObject(copy)
      │     ├─ src 是 plain object → 在 src 上递归合并
      │     │   result[key] = extend(true, src, copy)
      │     └─ src 不是 plain object → 创建新 {} 再合并
      │     │   result[key] = extend(true, {}, copy)
      │
      ├─ typeof copy !== 'undefined' → 直接覆盖（数组、字符串、数字等）
      │
      └─ typeof copy === 'undefined' → 跳过（不覆盖已有属性）
```

相关源文件：`packages/ee-bin/src/lib/extend.ts` 中 `extend()` 和 `isPlainObject()` 函数

### isPlainObject 判断逻辑

```
isPlainObject(obj)
  │
  ├─ obj 为空 或 toString !== '[object Object]' → false
  │     （排除 null、undefined、数组、原始类型）
  │
  ├─ proto = Object.getPrototypeOf(obj)
  │     proto === null → true（Object.create(null) 创建的对象）
  │
  └─ Ctor = proto.constructor
      typeof Ctor === 'function' && Ctor === Object → true
      （排除 Date、RegExp、Map、Set 及自定义类实例）
```

关键区别：ee-core 的 `isPlainObject` 使用 `hasOwnProperty('constructor')` 和 `isPrototypeOf` 判断，而 ee-bin 的实现直接比较 `Ctor === Object`，更简洁但覆盖场景略少（不处理 `constructor` 属性被覆盖的情况）。

相关源文件：`packages/ee-bin/src/lib/extend.ts` 中 `isPlainObject()` 函数

### helpers.is 类型检测

```
is.function(val)
  └─ typeof val === 'function' → true
     （包含普通函数、async 函数、generator 函数）

is.class(val)
  └─ typeof val === 'function'
     && val.toString().startsWith('class ') → 类声明检测
     && !!val.prototype → 确认有原型（排除箭头函数和 bind 函数）
```

双条件检测防止误判：`class` 关键字检测排除普通函数，`prototype` 存在性检测排除箭头函数和 `bind()` 返回的函数。

相关源文件：`packages/ee-bin/src/lib/helpers.ts` 中 `is` 对象

### helpers.copyDirSync 递归拷贝

```
copyDirSync(src, dest)
  │
  ├─ src 是文件？
  │     ├─ 创建 dest 父目录 → fs.mkdirSync(dirname(dest), { recursive: true })
  │     └─ 拷贝文件 → fs.copyFileSync(src, dest)
  │     └─ return
  │
  └─ src 是目录 → _copyDirRecursive(src, dest)
      ├─ 创建目标目录 → fs.mkdirSync(dest, { recursive: true })
      ├─ 读取目录条目 → fs.readdirSync(src, { withFileTypes: true })
      └─ 遍历条目
          ├─ 子目录 → _copyDirRecursive(srcPath, destPath)（递归）
          └─ 文件 → fs.copyFileSync(srcPath, destPath)
```

相关源文件：`packages/ee-bin/src/lib/helpers.ts` 中 `copyDirSync()` 和 `_copyDirRecursive()` 函数

### helpers.createDebug 调试日志

```
createDebug(namespace)
  │
  ├─ 缓存变量
  │     cachedDebugEnv: string | undefined
  │     cachedEnabled: boolean = false
  │
  └─ 返回日志函数
      (...args) => {
        if (checkEnabled()) → console.log(namespace, ...args)
      }

checkEnabled() 缓存逻辑：
  │
  ├─ currentEnv = process.env.DEBUG
  │     currentEnv === cachedDebugEnv → 返回 cachedEnabled（无需重新解析）
  │
  └─ 重新计算
      ├─ dbg = '*' → cachedEnabled = true（全局开启）
      └─ dbg.split(',').some(ns => ...)
          ├─ ns.endsWith(':*') → namespace.startsWith(ns.slice(0, -1))（通配匹配）
          └─ 否则 → ns === namespace（精确匹配）
      └─ cachedDebugEnv = currentEnv
      └─ cachedEnabled = 计算结果
```

缓存优化：只在 `DEBUG` 环境变量值变化时重新解析，避免每次日志调用都解析逗号分隔字符串。

相关源文件：`packages/ee-bin/src/lib/helpers.ts` 中 `createDebug()` 函数

### helpers.formatCmds 命令解析

```
formatCmds(command)
  │
  ├─ 空字符串 → []
  ├─ 无逗号 → [command]（单元素数组）
  └─ 有逗号 → command.split(',').map(trim).filter(非空)
     例: "frontend, electron" → ["frontend", "electron"]
     例: "frontend,,electron" → ["frontend", "electron"]
```

相关源文件：`packages/ee-bin/src/lib/helpers.ts` 中 `formatCmds()` 函数

### utils.loadConfig 配置加载

```
loadConfig(binFile?)
  │
  ├─ 确定配置路径：binPath = binFile || './cmd/bin.js'
  │
  ├─ loadFile(binPath) → userConfig
  │     ├─ .json5/.json → JsonLib.parse(文件内容)
  │     ├─ .js/.cjs → require(binPath)
  │     │     ├─ ESM default 提取 → mod.default ?? mod
  │     │     └─ 配置工厂模式 → is.function(result) && !is.class(result) → result()
  │     └─ null/undefined 导出 → 警告 + 返回 {}
  │
  └─ extend(true, { ...defaultConfig }, userConfig) → BinConfig
     用户配置深度覆盖默认配置
```

相关源文件：`packages/ee-bin/src/lib/utils.ts` 中 `loadConfig()` 和 `loadFile()` 函数

### utils.loadFile 文件读取

```
loadFile(filepath)
  │
  ├─ 校验文件存在：fs.existsSync(configFile)
  │     不存在 → 抛出错误（包含路径提示）
  │
  ├─ .json5 / .json 格式
  │     fs.readFileSync → JsonLib.parse → 返回对象
  │     （JSON5 支持注释、尾逗号等扩展语法）
  │
  └─ .js / .cjs 格式
      require(configFile) → mod
      ├─ ESM default 提取：mod.default != null ? mod.default : mod
      └─ 配置工厂模式检测：
          is.function(result) && !is.class(result) → result()（调用工厂函数）
          is.class(result) → result（类直接作为配置，不调用）
      ├─ null/undefined 导出 → 警告 + 返回 {}
      └─ 返回 result || {}
```

工厂模式设计：配置文件可以导出一个普通函数，函数被调用后返回配置对象。但类导出不被调用——类本身就是配置类定义，不是工厂函数。`is.function + !is.class` 双条件确保只调用真正的工厂函数。

相关源文件：`packages/ee-bin/src/lib/utils.ts` 中 `loadFile()` 函数

### utils 其他工具函数

```
rm(name)
  └─ fs.existsSync(name) → 不存在则 return
  └─ fs.rmSync(name, { recursive: true, force: true })

readJsonSync(filepath, encoding)
  └─ fs.existsSync(filepath) → 不存在则抛出错误
  └─ JSON.parse(fs.readFileSync(filepath, { encoding }))

writeJsonSync(filepath, str, options?)
  ├─ fs.mkdirSync(dirname(filepath), { recursive: true })（自动创建目录）
  ├─ typeof str === 'object' → JSON.stringify(str, replacer, space) + '\n'
  └─ typeof str !== 'object' → String(str)
  └─ fs.writeFileSync(filepath, data)
  默认 space=2（2 空格缩进）

toArray(value)
  ├─ string → [value]
  ├─ Array → value
  └─ undefined → []
```

相关源文件：`packages/ee-bin/src/lib/utils.ts` 中 `rm()`、`readJsonSync()`、`writeJsonSync()`、`toArray()` 函数

## 关键设计决策

### 1. 自定义 extend 替代 lodash.merge

选择自实现而非使用 lodash.merge，三个核心原因：

- **数组行为差异**：lodash.merge 逐索引合并数组（`[a] + [b] = [a+b]`），但配置合并场景下用户配置应全量覆盖默认数组，而非逐元素合并
- **安全防护**：lodash 无内置原型链污染防护，需额外配置；自实现直接过滤 `__proto__/constructor/prototype` 键
- **体积考量**：ee-bin 只需深合并这一个功能，引入整个 lodash 包不值得

### 2. 轻量替代库策略

每个替代只保留 ee-bin 实际需要的子集：

| 替代目标 | 保留功能 | 舍弃功能 |
|----------|----------|----------|
| is-type-of | `is.function`, `is.class` | 其他 30+ 类型检测 |
| chalk | 8 种 ANSI 颜色 | 256 色、RGB、主题、链式调用 |
| fs-extra | `copyDirSync` | 其他 20+ 文件操作 |
| debug | `createDebug`（带缓存） | 日志格式化、时间戳、颜色命名空间 |

### 3. createDebug 缓存优化

原始 debug npm 包每次日志调用都解析 `DEBUG` 环境变量字符串。自实现通过缓存避免重复解析：

- 缓存上次 `DEBUG` 值，只在值变化时重新计算启用状态
- 支持 `*` 通配符和 `:*` 前缀通配（如 `ee-bin:*` 匹配所有 ee-bin 子命名空间）
- 长运行进程中 `DEBUG` 环境变量通常不变，缓存命中率极高

### 4. loadFile 配置工厂模式

`.js` 配置文件可导出工厂函数而非静态对象，支持动态配置生成：

```js
// 静态导出
module.exports = { dev: { ... } };

// 工厂函数导出（可根据环境变量动态生成配置）
module.exports = function() {
  return { dev: { port: process.env.PORT || 8080 } };
};
```

`is.class` 检测防止误调用类导出——类导出本身就是配置类定义，不应作为函数调用。

### 5. writeJsonSync 自动目录创建

写入 JSON 文件前自动创建目标目录（`fs.mkdirSync(dirname, { recursive: true })`），避免因目录不存在导致写入失败。这对 incrUpdater 等模块特别重要——输出目录可能首次使用时不存在。

### 6. 空导出值警告而非报错

当配置文件导出 `null` 或 `undefined` 时，`loadFile` 输出警告而非抛出错误。原因：

- 配置文件可能有意导出空值（禁用某功能）
- 警告提示用户检查导出内容，但不阻断构建流程
- 最终返回 `{}`，与 `extend()` 合合时等同于无自定义配置

## 模块关系

```
extend.ts（独立模块，无外部依赖）
  └─ 被 utils.ts 引用 → extend(true, defaultConfig, userConfig)

helpers.ts（独立模块，仅依赖 Node.js fs/path）
  ├─ 被 move.ts 引用 → chalk, copyDirSync, formatCmds
  ├─ 被 iconGen.ts 引用 → chalk
  ├─ 被 incrUpdater.ts 引用 → chalk
  ├─ 被 utils.ts 引用 → createDebug, chalk, is
  └─ 被 serve.ts 引用 → chalk, createDebug, formatCmds

utils.ts（依赖 extend.ts + helpers.ts + bin_default.ts + json5）
  ├─ 引入 extend.ts → extend()（配置合并）
  ├─ 引入 helpers.ts → createDebug, chalk, is
  ├─ 引入 bin_default.ts → defaultConfig（默认配置基础）
  ├─ 引入 json5 → JSON5 解析器
  └─ 被以下模块引用
      ├─ move.ts → loadConfig, rm
      ├─ incrUpdater.ts → loadConfig, writeJsonSync
      ├─ encrypt.ts → loadConfig, rm, readJsonSync, writeJsonSync
      └─ serve.ts → loadConfig

三层依赖关系：
  ┌─────────────┐
  │  extend.ts   │  ← 最底层，纯函数无依赖
  ├─────────────┤
  │ helpers.ts   │  ← 轻量替代库，仅 Node.js 内置依赖
  ├─────────────┤
  │  utils.ts    │  ← 核心工具，依赖 extend + helpers + json5
  └─────────────┘
  各功能模块（move/iconGen/incrUpdater/serve/encrypt）依赖上述三层
```

**配置加载完整调用链**：

```
任何 CLI 命令 → loadConfig(binFile?)
  → loadFile('./cmd/bin.js')
    → .js → require() → ESM default 提取 → 工厂函数检测 → 配置对象
  → extend(true, { ...defaultConfig }, userConfig) → BinConfig
    → isPlainObject() → 递归深合并
    → __proto__/constructor/prototype 过滤
    → 数组全量覆盖
```
