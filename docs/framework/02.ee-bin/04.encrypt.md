# 加密系统（confusion/bytecode/strict 三种模式）

## 概述

ee-bin 的加密系统提供三种代码保护策略：confusion（混淆）、bytecode（字节码编译）、strict（混淆+字节码组合），以及 none（不加密）。该系统独立处理前端和 Electron 两个目标，每个目标有自己的加密策略和配置。核心设计包括：入口文件的特殊处理（bytenode loader shell）、特定文件强制混淆（BrowserWindow preload）、前端 bytecode 不兼容性处理，以及字节码编译的安全机制（防止编译失败导致代码丢失）。

**源文件**: `packages/ee-bin/src/tools/encrypt.ts`

**配置默认值**: `packages/ee-bin/src/config/bin_default.ts`（encrypt 配置段）

**类型定义**: `packages/ee-bin/src/types/config.ts`（EncryptConfig, ConfusionOptions, BytecodeOptions 接口）

## 核心流程

### 加密总体流程

```
encrypt(options)  ← CLI encrypt 命令入口
  ├── 创建 Encrypt({ target: 'electron' }) → electronEpt
  │     └── await electronEpt.encrypt()
  ├── 创建 Encrypt({ target: 'frontend' }) → frontendEpt
  │     └── await frontendEpt.encrypt()
  └── 两个实例独立处理，各有自己的加密策略
```

### encrypt() 方法流程

**源码**: `Encrypt.encrypt()` (第 150-189 行)

```
encrypt()
  ├── 1. 类型校验:
  │     ├── type 不在 EncryptTypes ('bytecode', 'confusion', 'strict') → 跳过返回
  │     └── target='frontend' 且 type='bytecode'/'strict' → 打印提示，跳过返回
  │           └── 原因: 浏览器渲染进程 V8 版本与编译时 V8 不同，字节码不兼容

  │     ├── 跳过非文件（目录等）

  │     ├── specFiles 中的文件 → generate(fullpath, 'confusion')
  │     │     └── 强制使用混淆（如 preload/bridge.js 不能用 bytecode）
  │     └── 其余文件 → generate(fullpath)
  │           └── 使用全局加密类型
  └── 4. 打印结束信息
```

### generate() 单文件加密

**源码**: `Encrypt.generate()` (第 200-218 行)

```
generate(curPath, type?)
  ├── encryptType = type || this.type  ← 支持类型覆盖（specFiles 强制混淆）
  ├── strict: generateJSConfuseFile → generateBytecodeFile  ← 先混淆再字节码
  ├── bytecode: generateBytecodeFile  ← 仅字节码
  └── confusion: generateJSConfuseFile  ← 仅混淆
```

**strict 模式的双重保护**: 先混淆再编译字节码。混淆后的代码已经难以阅读，再编译为字节码使源码完全不可逆。这是最强的保护策略。

### generateEntryLoader() 入口文件处理

**源码**: `Encrypt.generateEntryLoader()` (第 220-239 行)

```
generateEntryLoader(curPath)
  ├── strict: generateJSConfuseFile → generateBytecodeFile(deleteSource=false)
  │     ├── 先混淆代码
  │     └── 编译为 .jsc，但不删除源文件（deleteSource=false）
  ├── bytecode: generateBytecodeFile(deleteSource=false)
  │     └── 编译为 .jsc，不删除源文件
  ├── confusion: generate(curPath) → 正常混淆，返回
  │
  ├── 编译完成后:
  │     ├── jscName = basename(curPath) 去扩展名 + '.jsc'
  │     │     └── 例如: main.js → main.jsc
  │     └── 用 bytenode loader shell 替换原始 .js:
  │           │   require('bytenode');
  │           │   require('./main.jsc');
  │           │
  │           └── fs.writeFileSync(curPath, loaderCode)
```

**为什么入口文件需要 loader shell**:
- Electron/Node.js 运行时要求入口文件是字面的 `.js` 文件（不能是 `.jsc`）
- `package.json` 的 `main` 字段必须指向 `.js` 文件
- loader shell 是一个极小的 JS 文件，仅加载 bytenode 和 `.jsc` 文件

**为什么 deleteSource=false**:
- 在编译为 .jsc 后，原始 .js 文件不能删除，因为它需要被替换为 loader shell
- 如果先删除再写入 loader shell，中间步骤失败会导致入口文件丢失

**loader shell 示例**（假设入口文件是 `main.js`）：

```javascript
require('bytenode');
require('./main.jsc');

```

运行时流程：
```
Electron 启动 → package.json main = 'main.js'
  → 加载 main.js (loader shell)
    → require('bytenode') → 注册 .jsc 加载器
    → require('./main.jsc') → 执行字节码
```

### generateJSConfuseFile() 混淆实现

**源码**: `Encrypt.generateJSConfuseFile()` (第 247-263 行)

```
generateJSConfuseFile(file)
  ├── 合并混淆选项:
  │     ├── 默认: { compact: true, stringArray: true, stringArrayThreshold: 1 }
  │     ├── 用户: this.cOpt → Object.assign 覆盖默认值
  │     └── 结果: 用户未设置的字段使用默认值
  ├── 读取源文件: fs.readFileSync(file, 'utf8')
  ├── 执行混淆:
  │     └── withSuppressedObfuscatorAd(this.silent, () =>
  │           JavaScriptObfuscator.obfuscate(code, opt))
  │           ├── silent=true → 临时拦截 console.log 中的广告行
  │           └── silent=false → 正常执行，可能看到 "Pro" 广告
  └── 写回原文件: fs.writeFileSync(file, result.getObfuscatedCode(), 'utf8')
      └── 原位置替换（in-place replacement），不创建副本文件
```

**withSuppressedObfuscatorAd 机制**:

**源码**: `withSuppressedObfuscatorAd()` (第 53-72 行)

```
withSuppressedObfuscatorAd(silent, fn)
  ├── silent=false → 直接执行 fn()，不做任何拦截
  ├── silent=true → 拦截 console.log 中的广告行:
  │     ├── 保存原始 console.log → original
  │     ├── 替换 console.log 为自定义函数:
  │     │     ├── 检查首个参数是否包含 'JavaScript Obfuscator Pro' 或 'obfuscator.io'
  │     │     ├── 包含 → return (不输出，拦截广告)
  │     │     └── 不包含 → original.apply(console, args) (正常输出)
  │     ├── 执行 fn() → 混淆操作
  │     └── finally → console.log = original (恢复原始函数)
  └── 返回 fn() 的结果
```

javascript-obfuscator v5.x 在 stdout 是 TTY 且非 CI 环境时，会通过 `console.log` 打印 "JavaScript Obfuscator Pro" 推荐广告。没有官方选项关闭此广告，`withSuppressedObfuscatorAd` 通过临时替换 `console.log` 来过滤广告行。

### generateBytecodeFile() 字节码编译

**源码**: `Encrypt.generateBytecodeFile()` (第 274-295 行)

```
generateBytecodeFile(curPath, deleteSource=true)
  ├── 扩展名检查: ext !== '.js' → 返回（不处理 .json 等非 JS 文件）
  ├── 构建 bytenode 编译选项:
  │     ├── 默认: { filename: curPath, output: jscFile, electron: true }
  │     ├── 用户: this.bOpt → Object.assign 覆盖默认值
  │     └── electron: true → 编译为兼容 Electron 的 V8 字节码
  ├── await bytenode.compileFile(opt) → 编译 .js → .jsc
  ├── 安全验证:
  │     ├── fs.existsSync(jscFile) → .jsc 文件存在:
  │     │     └── deleteSource=true → fs.unlinkSync(curPath) 删除源文件
  │     │         └── 使用 unlinkSync 而非 rmSync，避免递归删除风险
  │     └── !fs.existsSync(jscFile) → .jsc 不存在:
  │           └── throw Error → 保留源文件，防止代码丢失
  └── 编译完成
```

**安全机制详解**:

字节码编译可能失败（如 bytenode 版本不兼容、Electron 版本问题等）。安全机制确保：
1. 编译成功 → 验证 `.jsc` 存在 → 删除源 `.js`（正常流程）
2. 编译失败 → `.jsc` 不存在 → 抛出错误 + 保留源 `.js`（防止代码丢失）
3. 使用 `fs.unlinkSync` 删除单个文件，而非 `fs.rmSync`（避免递归删除风险）

**electron: true 的意义**: bytenode 的 `electron: true` 选项启用 Electron V8 字节码兼容性。Electron 的 V8 版本可能与系统 Node.js 不同，直接用 Node.js 编译的字节码在 Electron 中无法执行。

### _getCodeFiles() 懒初始化文件列表

**源码**: `Encrypt._getCodeFiles()` (第 137-142 行)

```
_getCodeFiles()
  ├── codefiles !== undefined → 返回缓存（避免重复扫描）
  ├── patterns === null → 返回 undefined（无法执行加密）
  └── 首次调用: globbySync(patterns, { cwd: basePath }) → 存入 this.codefiles
      └── 返回文件列表
```

懒初始化设计：`codefiles` 不在构造函数中计算，而是在 `encrypt()` 方法首次调用 `_getCodeFiles()` 时才扫描。原因：
- `type='none'` 时不需要扫描文件（加密被跳过），避免浪费 I/O
- 扫描结果缓存在 `this.codefiles`，后续调用直接返回缓存

## 加密模式详解

### confusion（混淆）模式

| 特性 | 说明 |
|------|------|
| 工具 | javascript-obfuscator |
| 保护强度 | 中等（代码可读但难以理解） |
| 可逆性 | 低（理论上可通过手动分析还原，但成本极高） |
| 适用目标 | frontend + electron |
| 性能影响 | 中等（字符串数组解码有运行时开销） |

混淆默认选项（与用户配置通过 `Object.assign` 合并）：

| 选项 | 默认值 | 说明 |
|------|--------|------|
| `compact` | `true` | 移除换行，压缩代码 |
| `stringArray` | `true` | 将字符串提取到数组中 |
| `stringArrayThreshold` | `1` | 所有字符串都放入数组（0~1） |

**前端与 Electron 混淆策略差异**:

| 配置项 | 前端默认 | Electron 默认 | 原因 |
|--------|---------|-------------|------|
| `stringArrayEncoding` | `['none']` | `['rc4']` | 前端对性能敏感，不做解码开销；Electron 主进程可容忍 |
| `deadCodeInjection` | `false` | `false` | 死代码注入显著增加文件大小 |
| `target` | `'browser'` | `'node'` | 运行时环境不同 |

### bytecode（字节码）模式

| 特性 | 说明 |
|------|------|
| 工具 | bytenode |
| 保护强度 | 高（源码完全不可读） |
| 可逆性 | 极低（V8 字节码无法反编译为原始 JS） |
| 适用目标 | 仅 electron（前端不支持） |
| 性能影响 | 无（V8 直接执行字节码，无需解析 JS） |

**前端不支持 bytecode 的原因**:
- 编译时使用 Node.js/Electron 的 V8 引擎编译字节码
- 浏览器渲染进程使用不同版本的 V8 引擎
- V8 字节码格式与引擎版本绑定，跨版本不兼容
- 因此前端 bytecode 编译结果在浏览器中无法执行

**bytecode 编译选项**:

| 选项 | 默认值 | 说明 |
|------|--------|------|
| `filename` | 自动设置（curPath） | 源文件路径 |
| `output` | 自动设置（curPath + 'c'） | 输出 .jsc 文件路径 |
| `electron` | `true` | 编译为兼容 Electron 的 V8 字节码 |

### strict（严格）模式

| 特性 | 说明 |
|------|------|
| 组合 | confusion + bytecode |
| 保护强度 | 最高（双重保护） |
| 执行流程 | 先混淆 → 再编译字节码 |
| 适用目标 | 仅 electron |

strict 模式的处理顺序：
1. `generateJSConfuseFile(curPath)` → 将代码混淆为难以阅读的形式
2. `generateBytecodeFile(curPath)` → 将混淆后的代码编译为字节码

混淆后的代码被编译进字节码，提供两层保护：
- 即使能反编译字节码，得到的也是混淆后的代码
- 混淆 + 字节码的组合使逆向工程成本极高

## 特殊文件处理策略

### entryFiles（入口文件）

**示例**: `./public/electron/main.js`

入口文件的特殊处理原因：
- Electron/Node.js 运行时要求字面的 `.js` 入口文件
- `package.json` 的 `main` 字段必须指向 `.js`
- bytecode/strict 模式下：编译为 `main.jsc`，然后用 bytenode loader shell 替换 `main.js`

```
原始: main.js (包含业务代码)
  → 编译: main.jsc (V8 字节码)
  → 替换: main.js (极小 loader shell: require('bytenode'); require('./main.jsc');)
```

confusion 模式下入口文件正常混淆，不需要 loader shell。

### specificFiles（特定文件）

**示例**: `./public/electron/preload/bridge.js`

特定文件强制使用 confusion 的原因：
- `preload/bridge.js` 是 BrowserWindow 的 preload 脚本
- Electron 通过文件路径直接加载 preload 脚本
- preload 脚本必须在可读格式（JS）中，bytecode 格式不适用
- 即使全局加密类型是 bytecode/strict，特定文件仍强制使用 confusion

### 前端加密限制

前端加密的限制：
- **不支持 bytecode**: 浏览器渲染进程 V8 版本与编译时不同
- **仅支持 confusion**: 前端代码只能使用 javascript-obfuscator 混淆
- **性能敏感**: 前端 `stringArrayEncoding` 默认 `['none']`，避免解码开销

## 关键设计决策

### 1. 两个独立 Encrypt 实例

`encrypt()` 入口函数创建两个 Encrypt 实例（electron + frontend），各自独立处理。原因：
- 两个目标的加密策略完全不同（前端不支持 bytecode）
- 两个目标的文件列表不同（electron 在 `./public/electron/`，前端在 `./public/dist/`）
- 两个目标的混淆选项不同（`target: 'node'` vs `'browser'`）
- 独立实例避免了条件分支的复杂性

### 2. 懒初始化文件列表

`_getCodeFiles()` 使用懒初始化而非在构造函数中扫描。原因：
- `type='none'` 时加密被跳过，不需要扫描文件
- 避免构造函数中浪费 I/O（globby 扫描可能很慢）
- 扫描结果缓存，后续调用不重复扫描

### 3. withSuppressedObfuscatorAd 广告拦截

javascript-obfuscator v5.x 会打印 "JavaScript Obfuscator Pro" 推荐广告，没有官方关闭选项。`withSuppressedObfuscatorAd` 通过临时替换 `console.log` 实现过滤：
- 只拦截包含特定关键词的行，其他输出正常
- 使用 try/finally 确保原始 `console.log` 总是被恢复
- `silent=false` 时不做任何拦截，保持默认行为

### 4. generateBytecodeFile 安全机制

字节码编译的安全机制防止代码丢失：
- 编译前不做任何文件操作
- 编译后验证 `.jsc` 文件是否存在
- `.jsc` 存在 → 删除源 `.js`（正常流程）
- `.jsc` 不存在 → 抛出错误 + 保留源 `.js`（防止代码丢失）
- 使用 `fs.unlinkSync` 删除单个文件而非 `fs.rmSync`，避免误删目录

### 5. Object.assign 合而非深合并

混淆和字节码选项使用 `Object.assign` 合而非深合并（`extend`）。原因：
- `ConfusionOptions` 和 `BytecodeOptions` 都是扁平结构（无嵌套对象）
- `Object.assign` 对扁平结构的覆盖行为正确
- 用户配置的每个字段直接覆盖对应默认值

### 6. 前端 bytecode 跳过而非报错

当前端配置 `type='bytecode'` 或 `type='strict'` 时，不抛出错误而是打印提示并跳过。原因：
- 这是技术限制而非配置错误
- 用户可能同时配置前端和 Electron，前端自动跳过 bytecode 不影响 Electron 加密
- 抛出错误会中断整个加密流程，包括本应成功的 Electron 加密

## API 导出

| 导出项 | 类型 | 说明 |
|------|------|------|
| `encrypt` | `(options: EncryptOptions) => Promise<void>` | 加密入口函数，处理 electron + frontend |
| `cleanEncrypt` | `(options: { dir?: string \| string[] }) => void` | 清除加密输出目录 |

**EncryptOptions 接口**:

| 字段 | 类型 | 说明 |
|------|------|------|
| `config` | `string?` | 自定义 bin.js 配置文件路径 |
| `out` | `string?` | 加密输出目录覆盖 |
| `target` | `string?` | 加密目标（`'electron'` 或 `'frontend'`） |

**Encrypt 类关键方法**:

| 方法 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `encrypt()` | 无 | `Promise<void>` | 执行加密流程 |
| `generate(curPath, type?)` | `string, string?` | `Promise<void>` | 单文件加密 |
| `generateEntryLoader(curPath)` | `string` | `Promise<void>` | 入口文件特殊处理 |
| `generateJSConfuseFile(file)` | `string` | `void` | 混淆单文件 |
| `generateBytecodeFile(curPath, deleteSource)` | `string, boolean` | `Promise<void>` | 字节码编译 |
| `_getCodeFiles()` | 无 (private) | `string[] \| undefined` | 懒获取文件列表 |

## 模块关系

```
tools/encrypt.ts
  ├── 内部依赖:
  │     ├── lib/helpers.ts → chalk (彩色输出)
  │     └── lib/utils.ts → loadConfig, toArray (配置加载, 数组归一化)
  ├── 外部依赖:
  │     ├── bytenode → compileFile (字节码编译引擎)
  │     ├── javascript-obfuscator → obfuscate (代码混淆引擎)
  │     └── globbySync → 文件扫描
  ├── 类型依赖:
  │     └── types/config.ts → EncryptConfig, ConfusionOptions, BytecodeOptions, EncryptOptions
  └── CLI 路由:
        ├── index.ts → encrypt 命令 → encrypt(options)
        └── index.ts → clean 命令 → cleanEncrypt(options)

加密模块与其他 ee-bin 模块的关系:
  ├── 无直接依赖 serve.ts (加密是独立操作，不涉及进程管理)
  ├── 无直接依赖 move.ts (加密处理的是构建输出，不是资源移动)
  └── 无直接依赖 bundle_registry_plugin.ts (加密在打包后执行，处理打包产物)
```
