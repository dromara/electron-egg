构建模式负责将前端和 Electron 代码打包为可分发产物。`ee-bin build` 是发布流程的核心命令，通过 `--cmds` 选项灵活组合构建步骤。

## 完整流程

`ServeProcess.build()` 的执行流程：

<Steps>
### 设置环境变量

设置 `NODE_ENV`，默认为 `prod`。可通过 `--env` 选项覆盖。

```js
process.env.NODE_ENV = env || 'prod';
```

设为 `prod` 时，esbuild sourcemap 自动关闭，生成更小的输出文件。

### 加载配置

从 `./cmd/bin.js` 加载用户配置，深度合并默认配置。

### 检查 --cmds 参数

`--cmds` 必须指定。未指定时，ee-bin 打印警告并退出：

```bash
Warning Please modify the build property in the bin file
```

### 处理 electron 命令

如果 `--cmds` 包含 `electron`：

1. 执行 `this.bundle()` 打包 Electron 代码
2. 从命令列表中移除 `electron`（它只触发打包，不启动子进程）
3. 切换 `package.json` 的 `main` 字段到打包输出

```js
if (commands.includes('electron')) {
  await this.bundle(binCfg.build.electron);
  // 移除 electron，不再传给 multiExec
  const index = commands.indexOf('electron');
  commands.splice(index, 1);
  cmds = commands.join(',');
  this._switchPkgMain();
}
```

### 执行剩余命令

调用 `this.multiExec()` 执行剩余命令（如 `frontend`、`win64`、`mac` 等）。每个命令创建独立子进程。

### 恢复 main 字段

构建完成后，调用 `_restorePkgMain()` 将 `package.json` 的 `main` 字段恢复到原始值。防止配置文件被永久修改。
</Steps>

<Note>
`electron` 命令仅触发 esbuild 打包，不会启动 Electron 进程，因此从命令列表中移除后不传给 `multiExec`。而 `frontend`、`win64`、`mac` 等命令则是启动子进程执行构建。
</Note>

## --cmds 组合

`--cmds` 用逗号分隔多个构建步骤，支持以下组合：

<Tabs>
<Tab title="前端 + Electron">
```bash
ee-bin build --cmds frontend,electron
```
先构建前端（`npm run build`），再打包 Electron 代码。这是最基本的构建组合，通常在 `pnpm build` 内部调用。
</Tab>
<Tab title="Windows 64 位">
```bash
ee-bin build --cmds frontend,electron,win64
```
构建前端 + 打包 Electron + 生成 Windows 64 位安装包。`win64` 使用 `electron-builder` 配置 `./cmd/builder.json`。
</Tab>
<Tab title="macOS">
```bash
ee-bin build --cmds frontend,electron,mac
```
构建 + 生成 macOS 安装包。使用 `./cmd/builder-mac.json` 配置。
</Tab>
<Tab title="macOS ARM64">
```bash
ee-bin build --cmds frontend,electron,mac_arm64
```
构建 + 生成 macOS ARM64 安装包。使用 `./cmd/builder-mac-arm64.json` 配置。
</Tab>
<Tab title="Linux">
```bash
ee-bin build --cmds frontend,electron,linux
```
构建 + 生成 Linux deb 包。使用 `./cmd/builder-linux.json` 配置。
</Tab>
</Tabs>

内置的平台打包命令：

| 命令 | electron-builder 配置 | 目标平台 |
|------|----------------------|----------|
| `win32` | `builder.json` `-w=nsis --ia32` | Windows 32 位 |
| `win64` | `builder.json` `-w=nsis --x64` | Windows 64 位 |
| `win_e` | `builder.json` `-w=portable --x64` | Windows 便携版 |
| `win_7z` | `builder.json` `-w=7z --x64` | Windows 7z 压缩版 |
| `mac` | `builder-mac.json` `-m` | macOS x64 |
| `mac_arm64` | `builder-mac-arm64.json` `-m --arm64` | macOS ARM64 |
| `linux` | `builder-linux.json` `-l=deb --x64` | Linux x64 |
| `linux_arm64` | `builder-linux.json` `-l=deb --arm64` | Linux ARM64 |

## Bundle 与 Copy 模式

`build.electron.bundleType` 决定 Electron 代码的打包方式：

### Bundle 模式

`bundleType: 'bundle'`（默认）使用 esbuild + `bundleRegistryPlugin` 将 Electron 代码打包为单文件：

1. 清除输出目录 `public/electron/`
2. 调用 `_bundleWithRegistry()` 执行 esbuild 构建
3. 虚拟入口 `app:bundle-entry` 加载配置注册 → 控制器注册 → main.js
4. 打包结果为 `public/electron/main.js` 单文件
5. 复制不可打包的文件（jobs 目录、preload/bridge.js、用户指定 copy 目标）

### Copy 模式

`bundleType: 'copy'` 直接复制 `electron/` 目录到 `public/electron/`，不执行 esbuild 打包：

```js
if (bundleConfig.bundleType === 'copy') {
  copyDirSync(path.join(cwd, ELECTRON_DIR), outdir);
}
```

适用于不需要打包的场景（如调试、或业务代码本身就是多文件结构）。

<Warning>
Copy 模式不会生成 `global.__EE_CONTROLLER_REGISTRY__` 和 `global.__EE_CONFIG_REGISTRY__`。运行时 ee-core 会退回到 globby 文件系统扫描 + `require()` 方式加载控制器和配置，与 bundle 模式的行为不同。
</Warning>

## package.json main 字段切换

打包后 Electron 需要从 `public/electron/main.js` 启动，而不是 `electron/main.js`。ee-bin 通过切换 `package.json` 的 `main` 字段实现：

### _switchPkgMain()

打包完成后将 `main` 字段切换到 `./public/electron/main.js`：

```js
private _switchPkgMain(): void {
  const pkgPath = path.join(process.cwd(), PKG_PATH);
  const pkg = readJsonSync(pkgPath);
  const bundleMainPath = BUNDLE_DIR + '/main.js';

  if (pkg.main !== bundleMainPath) {
    this.originalPkgMain = pkg.main as string | undefined;  // 保存原始值
    pkg.main = bundleMainPath;                                // 切换
    writeJsonSync(pkgPath, pkg);
  }
}
```

### _restorePkgMain()

构建完成后恢复 `main` 字段到原始值：

```js
private _restorePkgMain(): void {
  if (this.originalPkgMain !== undefined) {
    const pkgPath = path.join(process.cwd(), PKG_PATH);
    const pkg = readJsonSync(pkgPath);
    pkg.main = this.originalPkgMain;
    writeJsonSync(pkgPath, pkg);
    this.originalPkgMain = undefined;
  }
}
```

<Note>
dev 模式下 `main` 字段在 `SIGINT/SIGTERM` 时恢复；build 模式下在 `multiExec` 完成后立即恢复。两种模式都确保 `package.json` 不会被永久修改。
</Note>

## --env 选项

`--env` 控制 `NODE_ENV` 值，影响 esbuild sourcemap 和代码压缩行为：

```bash
# 生产构建（默认）
ee-bin build --cmds frontend,electron --env prod

# 开发构建（启用 inline sourcemap）
ee-bin build --cmds frontend,electron --env dev
```

| 环境值 | sourcemap 行为 | 适用场景 |
|--------|---------------|----------|
| `prod` | 自动关闭 | 正式发布 |
| `dev` / `local` | inline sourcemap | 调试构建 |

## 信号处理

构建模式注册了 `SIGINT` 和 `SIGTERM` 信号处理器：

1. 遍历 `execProcess` 表，使用 `tree-kill` 杀死所有子进程
2. 恢复 `package.json` 的 `main` 字段
3. 等待 500ms
4. `process.exit(0)` 退出

```js
process.on('SIGINT', () => {
  console.log('[ee-bin] Received SIGINT. Closing processes...');
  this._closeProcess();
});
```

这确保即使构建被中断，配置文件也会被正确恢复，不会留下脏状态。

## 构建配置示例

```js
// cmd/bin.js
module.exports = {
  build: {
    frontend: {
      directory: './frontend',
      cmd: 'npm',
      args: ['run', 'build'],
    },
    electron: {
      bundleType: 'bundle',
      external: [],
      sourcemap: false,
      minify: true,
      drop: ['console'],
    },
    win64: {
      cmd: 'electron-builder',
      directory: './',
      args: ['--config=./cmd/builder.json', '-w=nsis', '--x64'],
    },
  },
};
```
