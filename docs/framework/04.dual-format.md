# 双格式输出机制（CJS + ESM）

## 概述

ee-core 和 ee-bin 都输出双格式：CommonJS (CJS) 和 ECMAScript Modules (ESM)。同一份 TypeScript 源码通过两次编译（不同的 `module` 设置）生成两种格式，由 `package.json` 的 `exports` map 提供条件化路径解析。

## 编译策略

### ESM 编译（主输出）

**配置文件**: `packages/ee-core/tsconfig.json` / `packages/ee-bin/tsconfig.json`

```
target: ES2020
module: NodeNext
moduleResolution: NodeNext
outDir: ./dist/esm/
declaration: true        → 生成 .d.ts 类型声明
declarationMap: true      → 生成 .d.ts.map（IDE跳转）
sourceMap: true           → 生成 .js.map
```

ESM 编译是**权威输出**：类型声明、source map、declaration map 都从此生成。输出文件使用 ESM 语法（`export`/`import`），NodeNext 要求源码中 import 路径必须带 `.js` 扩展名。

### CJS 编译（兼容输出）

**配置文件**: `packages/ee-core/tsconfig.cjs.json` / `packages/ee-bin/tsconfig.cjs.json`

```
extends: tsconfig.json    → 继承所有基础设置
module: CommonJS          → 输出 require/module.exports 语法
moduleResolution: node    → Node.js 传统模块解析
outDir: ./dist/cjs/
declaration: false        → 不生成 .d.ts（类型由 ESM 输出提供）
declarationMap: false     → 不生成 .d.ts.map
```

CJS 编译是**次要输出**：不生成类型声明文件，两种格式共用 ESM 的 `.d.ts`。这种设计避免了类型声明重复维护。

### 构建顺序

```bash
npm run build → clean → build:cjs → build:esm → gen-exports
```

CJS 先编译，ESM 后编译（ESM 生成类型声明供两者共用），最后 `gen-exports` 脚本自动生成 `package.json` 的 `exports` map。

## exports Map 设计

### ee-core 的 exports 结构

```json
{
  "exports": {
    ".": {
      "import": { "types": "./dist/esm/index.d.ts", "default": "./dist/esm/index.js" },
      "require": { "types": "./dist/esm/index.d.ts", "default": "./dist/cjs/index.js" }
    },
    "./log": {
      "import": { "types": "./dist/esm/log/index.d.ts", "default": "./dist/esm/log/index.js" },
      "require": { "types": "./dist/esm/log/index.d.ts", "default": "./dist/cjs/log/index.js" }
    },
    // ... 19+ 子路径模块，每个都有 import/require 条目
  }
}
```

**关键设计**：
- `import` 条件（ESM 消费者）指向 `dist/esm/`
- `require` 条件（CJS 消费者）指向 `dist/cjs/`
- 两者的 `types` 都指向 `dist/esm/` 的 `.d.ts`（单一类型来源）
- `exports` map 由 `scripts/gen-exports.js` 自动生成（扫描 `src/` 子目录），勿手动编辑

### ee-bin 的 exports 结构

```json
{
  "exports": {
    ".": {
      "import": { "types": "./dist/esm/index.d.ts", "default": "./dist/esm/index.js" },
      "require": { "types": "./dist/esm/index.d.ts", "default": "./dist/cjs/index.js" }
    }
  }
}
```

ee-bin 只有一个入口点（CLI 工具），不需要子路径 exports。

## 源码中的 .js 扩展名要求

ee-core 和 ee-bin 的 TypeScript 源码中所有本地 import 必须带 `.js` 扩展名：

```typescript
// 正确（NodeNext 要求）
import { loadConfig } from '../config/index.js';
import { logger } from '../log/index.js';

// 错误（NodeNext 不允许省略扩展名）
import { loadConfig } from '../config/index';
```

`.js` 扩展名在 TypeScript 编译时被保留，Node.js 运行时根据 `exports` map 解析到实际的 `.js` 文件（ESM 或 CJS）。这种写法对两种输出格式都有效。

## package.json 其他字段

| 字段 | ee-core | ee-bin | 说明 |
|------|---------|--------|------|
| `main` | `./dist/cjs/index.js` | `./dist/esm/index.js` | CJS 默认入口（ee-bin 用 ESM 因为 CLI 需要最新特性） |
| `module` | `./dist/esm/index.js` | `./dist/esm/index.js` | ESM 默认入口（兼容旧 bundler） |
| `types` | `./dist/esm/index.d.ts` | `./dist/esm/index.d.ts` | 类型声明入口 |
| `bin` | — | `{ "ee-bin": "./dist/esm/index.js" }` | CLI 命令映射（带 shebang） |

## ee-bin 的 CLI 入口

ee-bin 的入口文件 `src/index.ts` 开头有 shebang `#!/usr/bin/env node`，编译后保留在 `dist/esm/index.js`。`build:chmod` 步骤确保两个 CLI 文件可执行：

```bash
npm run build:chmod → chmod +x dist/esm/index.js dist/cjs/index.js
```

## 互操作边界

当 CJS 项目使用 ee-core（`require('ee-core')`）时：
- Node.js 通过 `exports` map 的 `require` 条件加载 `dist/cjs/index.js`
- 类型声明由 `dist/esm/index.d.ts` 提供

当 ESM 项目使用 ee-core（`import ... from 'ee-core'`）时：
- Node.js 通过 `exports` map 的 `import` 条件加载 `dist/esm/index.js`
- 类型声明同样由 `dist/esm/index.d.ts` 提供

ee-core 内部处理 ESM 互操作的关键位置：
- `FileLoader.parseFromRegistry()`: 检查 `__esModule` 标记，解包 `.default`
- `ConfigLoader._loadConfig()`: 处理 esbuild 生成的 `{ __esModule: true, default: fn }` 模式
- `core/utils/index.ts` 的 `loadFile()`: CJS require 后检查 `__esModule` + `default`
- ee-bin 的 `lib/utils.ts` 的 `loadFile()`: 同样处理 ESM default 导出
