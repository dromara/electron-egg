# 05. SocketIO + HTTP + IPC 三通道服务

## 概述

Socket 模块是 ee-core 框架的通信枢纽，提供三种通信通道服务，覆盖所有进程间通信场景：

- **SocketServer**：基于 SocketIO 的 WebSocket/HTTP polling 通信，供第三方软件（Go/Python 后端、远程控制面板等）与主进程交互
- **HttpServer**：基于 Koa 的 HTTP/HTTPS RESTful API 服务，供浏览器端或其他 HTTP 客户端调用主进程控制器方法
- **IpcServer**：基于 Electron ipcMain 的 IPC 通道服务，供渲染进程通过 ipcRenderer 与主进程交互

三种服务共享同一套路由机制（`resolveControllerFn`），将通信路径映射到控制器方法，实现统一的请求分发模型。

源文件位于 `packages/ee-core/src/socket/` 目录下，入口为 `index.ts`。

## 核心流程

### 三通道初始化流程

源文件：`packages/ee-core/src/socket/index.ts`

```
Application.run() → loadSocket()
  │
  ├─ await createSocketServer()     1. 创建 SocketIO 服务（异步，需获取端口）
  ├─ await createHttpServer()       2. 创建 HTTP 服务（异步，需获取端口）
  └─ createIpcServer()              3. 创建 IPC 服务（同步，注册通道即可）
```

初始化顺序的设计考量：
- SocketIO 先启动，因为它提供最基础的双向通信能力，Go/Python 后端可能依赖此通道
- HTTP 其次启动，RESTful API 是对外暴露的标准接口
- IPC 最后创建，因为它是 Electron 内部通道，不需要端口分配

**Instance 存储结构**：
```ts
const Instance = {
  socketServer: SocketServer | null,
  httpServer: HttpServer | null,
  ipcServer: IpcServer | null,
}
```

每个服务通过对应的 `create*()` 工厂函数创建，通过 `get*()` 获取实例。

### 共享路由机制：resolveControllerFn

源文件：`packages/ee-core/src/socket/utils.ts`

三种服务共享同一个控制器函数解析逻辑，将通信路径字符串映射到控制器对象树中的方法：

```
resolveControllerFn(controller, cmd, separator)
  │
  ├─ 防御检查：cmd 非 string → return null
  ├─ cmd.split(separator) → 将路径拆分为层级数组
  │   例：'controller/user/add' + separator='/' → ['controller', 'user', 'add']
  ├─ 逐层遍历控制器对象树：
  │   obj = controller → obj = obj['controller'] → obj = obj['user'] → obj = obj['add']
  │   任一层不存在 → 记录错误日志，return null
  └─ 最终节点类型检查：
      ├─ isFunction(obj) → return obj（找到目标方法）
      └─ 非 function → 记录错误日志，return null
```

此机制实现了"通信路径 → 控制器方法"的自动映射，三种通道使用统一的命名规则。

### SocketServer 通信流程

源文件：`packages/ee-core/src/socket/socketServer.ts`

```
SocketServer.create()
  │
  ├─ new SocketServer()          构造实例
  │   ├─ 读取 config.socketServer 配置
  │   └─ 读取 channelSeparator（默认 '/'）
  │
  ├─ await init()
  │   ├─ [enable=false] → 直接返回，不启动
  │   ├─ await getPort({ port }) → 获取可用端口
  │   ├─ process.env.EE_SOCKET_PORT = String(port) → 写入环境变量
  │   ├─ new Server(port, config) → 创建 SocketIO Server
  │   └─ connect() → 注册连接和消息处理
  │
  └─ return instance

connect() 连接处理流程：
  │
  ├─ io.on('connection', (socket) => ...)
  │   ├─ 读取 channel 配置（默认 SocketIO.partySoftware = 'socket-channel'）
  │   └─ socket.on(channel, async (message, callback) => ...)
  │       ├─ 解析 message: { cmd, args }
  │       ├─ resolveControllerFn(controller, cmd, channelSeparator) → 定位方法
  │       ├─ [方法不存在] → throw Error('function not exists')
  │       ├─ fn.call(controller, args) → 执行控制器方法
  │       ├─ callback(result) → 将结果返回给客户端
  │       └─ [异常] → callback({ __EE_ERROR__: true, message })
```

**消息格式**：
```ts
// 客户端发送
{ cmd: 'controller/user/add', args: { name: 'test' } }

// 服务端正常返回
callback(控制器方法的返回值)

// 服务端异常返回
callback({ __EE_ERROR__: true, message: '错误信息' })
```

### HttpServer 通信流程

源文件：`packages/ee-core/src/socket/httpServer.ts`

```
HttpServer.create()
  │
  ├─ new HttpServer()
  │   ├─ 读取 config.httpServer 配置
  │   └─ 读取 channelSeparator
  │
  ├─ await init()
  │   ├─ [enable=false] → 直接返回
  │   ├─ await getPort() → 获取端口
  │   ├─ process.env.EE_HTTP_PORT = String(port)
  │   └─ _create() → 构建 Koa 应用
  │
  └─ return instance

_create() Koa 应用构建流程：
  │
  ├─ 构建 Koa 应用
  ├─ 中间件加载链（按顺序）：
  │   1. errorHandler → 全局错误处理
  │   2. preMiddleware → 用户前置中间件
  │   3. cors → CORS 跨域中间件
  │   4. koaBody → 请求体解析中间件
  │   5. _dispatch → 核心路由分发中间件
  │   6. postMiddleware → 用户后置中间件
  │
  ├─ HTTPS 支持：
  │   ├─ [https.enable=true] → 读取 SSL key/cert 文件
  │   ├─ https.createServer(sslOptions, koaApp.callback())
  │   └─ 否则 → koaApp.listen()
  │
  └─ 注册 server error 事件处理

_dispatch() 核心路由分发流程：
  │
  ├─ 获取 controller 和 filterRequest 配置
  ├─ uriPath = ctx.request.path（如 '/controller/user/add'）
  ├─ params = ctx.request.query（GET 请求参数）
  ├─ body = ctx.request.body（POST 请求参数）
  │
  ├─ 1. 移除前导 '/' → 'controller/user/add'
  ├─ 2. filterRequest.uris 检查：
  │   ├─ [在过滤列表中] → 返回 filterRequest.returnData，不走控制器
  │   └─ [不在列表中] → 继续路由
  ├─ 3. 自动补 'controller' 前缀：
  │   ├─ uriPath.slice(0,10) !== 'controller' → 添加前缀
  │   └─ 已有前缀 → 不添加
  ├─ 4. 路径转换：
  │   'controller/user/add' → '/' 分隔 → cmd = 'controller/user/add'
  ├─ 5. 参数选择：
  │   ├─ GET → args = params（query 参数）
  │   └─ POST → args = body（body 参数）
  ├─ 6. resolveControllerFn(controller, cmd, channelSeparator) → fn
  ├─ 7. fn.call(controller, args, ctx) → 执行
  │   ├─ [成功] → ctx.response.body = result
  │   └─ [异常] → ctx.response.status = 500, body = { error: 'Internal Server Error' }
  │              ctx.__ee_error = err（供下游错误上报中间件使用）
  └─ await next()
```

**HTTP 路由映射示例**：
```
GET  /user/add?name=test     → controller.user.add({ name: 'test' }, ctx)
POST /controller/user/add    → controller.user.add(body参数, ctx)
GET  /favicon.ico            → filterRequest.returnData（不走控制器）
```

### IpcServer 通信流程

源文件：`packages/ee-core/src/socket/ipcServer.ts`

```
new IpcServer()
  │
  ├─ 读取 channelSeparator（默认 '/'）
  ├─ directory = 'controller'
  ├─ init()
  │   ├─ getController() → 获取控制器对象树
  │   └─ loop(controller, 'controller') → 递归遍历
  │
  └─ 构造完成

loop(obj, pathname) 递归遍历流程：
  │
  ├─ Object.keys(obj) → 获取所有键
  ├─ 过滤 constructor 键（跳过）
  ├─ 对每个 key：
  │   ├─ subObj = obj[key]
  │   ├─ propertyChain = pathname + '.' + key（如 'controller.user'）
  │   │
  │   ├─ [subObj[EXPORTS] === true] → 叶子节点
  │   │   └─ register(subObj, propertyChain) → 注册 IPC 通道
  │   │
  │   └─ [typeof subObj === 'object'] → 中间节点
  │       └─ loop(subObj, propertyChain) → 继续递归

register(exportObj, propertyChain) IPC 通道注册：
  │
  ├─ Object.keys(exportObj) → 遍历每个方法名
  ├─ 对每个方法 key：
  │   ├─ tmpChannel = propertyChain + '.' + key（如 'controller.user.add')
  │   ├─ channel = tmpChannel.replace('.', channelSeparator)（如 'controller/user/add')
  │   │
  │   ├─ ipcMain.on(channel, async (event, params) => ...)  [send/on 模型]
  │   │   ├─ resolveControllerFn() → fn
  │   │   ├─ fn.call(controller, params, event) → result
  │   │   ├─ event.returnValue = result → 同步返回给渲染进程
  │   │   └─ [异常] → event.returnValue = { __EE_ERROR__: true, message }
  │   │
  │   └─ ipcMain.handle(channel, async (event, params) => ...)  [invoke/handle 模型]
  │       ├─ resolveControllerFn() → fn
  │       └─ return fn.call(controller, params, event) → Promise 返回
```

**IPC 两种通信模型对比**：

| 模型 | 渲染进程调用 | 主进程注册 | 返回方式 | 异步支持 |
|------|-------------|-----------|---------|---------|
| send/on | `ipcRenderer.sendSync()` | `ipcMain.on()` | `event.returnValue` | 支持（Electron 嵌套消息循环） |
| invoke/handle | `ipcRenderer.invoke()` | `ipcMain.handle()` | Promise | 推荐，原生异步 |

**通道命名规则**：
```
控制器路径：controller.user.add
→ 替换 '.' 为 channelSeparator（默认 '/'）
→ IPC 通道名：controller/user/add
```

## 关键设计决策

### 1. 工厂模式 vs 构造函数模式

SocketServer 和 HttpServer 使用 `static async create()` 工厂模式，因为初始化需要异步获取端口（`getPort()`）。IpcServer 使用普通构造函数，因为 IPC 注册不需要异步操作（端口由 Electron 管理）。

### 2. 统一路由机制（resolveControllerFn）

三种通信通道共享 `resolveControllerFn()` 路由函数，这意味着：
- 控制器方法只需定义一次，三种通道自动可访问
- 通道命名规则统一：`controller/{name}/{method}`，用 `channelSeparator` 分隔
- 新增控制器方法时无需手动注册路由，FileLoader 自动完成对象树构建

### 3. __EE_ERROR__ 错误标记

SocketServer 和 IpcServer（send/on 模型）在异常时返回 `{ __EE_ERROR__: true, message }` 而非直接抛出。这是因为：
- SocketIO callback 只能传值，不能传异常对象
- IPC sendSync 的 `event.returnValue` 只能传可序列化数据
- 客户端通过检查 `__EE_ERROR__` 标记区分"正常返回 undefined"和"方法执行异常"

HttpServer 则使用标准 HTTP 500 响应 + `ctx.__ee_error` 附加原始错误，供下游中间件（如错误上报）使用。

### 4. IPC 双模型注册

IpcServer 为每个方法同时注册 `ipcMain.on` 和 `ipcMain.handle`，原因：
- `send/on` 模型支持同步调用（渲染进程使用 `sendSync()`），兼容旧代码
- `invoke/handle` 模型是异步调用（渲染进程使用 `invoke()`），是推荐方式
- 两个模型注册同一通道名，渲染进程可选择调用方式

### 5. EXPORTS 符号标记区分叶子/中间节点

IpcServer 的 `loop()` 递归遍历通过 `EXPORTS` 符号（由 FileLoader 在加载控制器时设置）区分叶子节点和中间节点。这避免了"判断对象是否是控制器方法集"的歧义——中间节点（目录级）和叶子节点（方法集）都是对象，只有 `EXPORTS` 标记能可靠区分。

### 6. filterRequest 机制

HttpServer 的 `filterRequest` 配置允许指定不需要走控制器路由的 URI（如 `favicon.ico`），直接返回预定义数据。这避免了无意义的控制器调用和 500 错误。

### 7. HTTP 中间件链的开放设计

HttpServer 的中间件链分为六个环节，其中 `preMiddleware` 和 `postMiddleware` 是用户自定义中间件的插入点。中间件支持工厂函数模式（`() => Koa.Middleware`），方便注入依赖。

## 模块关系

```
socket/index.ts
  ├─ socket/socketServer.ts
  │     ├─ 引用 config → getConfig（端口和 SocketIO 配置）
  │     ├─ 引用 const/channel → SocketIO.partySoftware（默认通道名）
  │     ├─ 引用 controller → getController（获取控制器对象树）
  │     ├─ 引用 utils/port → getPort（端口分配）
  │     ├─ 引用 socket/utils → resolveControllerFn（路由解析）
  │     └─ 引用 socket.io（外部依赖）
  │
  ├─ socket/httpServer.ts
  │     ├─ 引用 config → getConfig（HTTP 配置）
  │     ├─ 引用 controller → getController
  │     ├─ 引用 utils/port → getPort
  │     ├─ 引用 socket/utils → resolveControllerFn
  │     ├─ 引用 ps → getBaseDir（SSL 证书文件路径）
  │     ├─ 引用 log → coreLogger
  │     └─ 引用 koa, koa2-cors, koa-body（外部依赖）
  │
  ├─ socket/ipcServer.ts
  │     ├─ 引用 config → getConfig（channelSeparator）
  │     ├─ 引用 controller → getController
  │     ├─ 引用 core/loader/file_loader → EXPORTS（叶子节点标记）
  │     ├─ 引用 socket/utils → resolveControllerFn
  │     └─ 引用 electron → ipcMain
  │
  └─ socket/utils.ts
        ├─ 引用 utils/type_check → isFunction
        └─ 引用 log → coreLogger

上游调用者：
  Application.run() → loadSocket()

下游依赖：
  - config 模块：端口、通道配置、cors 配置等
  - controller 模块：提供控制器对象树
  - core/loader/file_loader 模块：EXPORTS 符号标记（IPC 叶子节点识别）
  - utils/port 模块：动态端口分配
