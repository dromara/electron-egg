# 06. Job 系统（ChildJob / ChildPoolJob / LoadBalancer）

## 概述

Job 系统是 ee-core 框架的子进程任务管理模块，提供两类子进程管理器和一套负载均衡算法库：

- **ChildJob**：单任务子进程管理器，每次执行创建一个独立子进程，适用于一次性、独立的任务场景
- **ChildPoolJob**：子进程池管理器，维护一组常驻子进程并通过负载均衡算法分配任务，适用于高频、复用型任务场景
- **LoadBalancer**：负载均衡引擎，提供 8 种算法（轮询、加权、随机、指定、最小连接数及其加权变体），由 ChildPoolJob 内部使用
- **registry**：子进程管理器注册表，在应用退出时统一终止所有子进程

核心设计思想：利用 `child_process.fork()` 创建 Node.js 子进程，通过 IPC 通道发送命令和接收结果。子进程内运行 ChildApp 应用，接收 `run` 命令并加载/执行任务文件。类型任务实例化后缓存复用，函数型任务每次直接调用。

源文件位于 `packages/ee-core/src/jobs/` 目录下。

## 核心流程

### registry 注册与退出清理

源文件：`packages/ee-core/src/jobs/registry.ts`

```
registerJobManager(job)       注册 Killable 实例到 registeredJobs 数组
killAllJobs()                 在 before-quit 事件中调用
  │
  ├─ 遍历 registeredJobs
  │   ├─ [有 killAll 方法] → job.killAll()
  │   └─ [有 getPids + kill 方法] →逐个 kill(pid)
  │
  └─ [异常] → 收集到 failed 数组，最后汇总报错
```

**Killable 接口定义**：
```ts
type Killable = { killAll: () => void } | { getPids: () => string[]; kill: (pid: string | number) => void }
```

两种接口适配不同管理器：
- ChildPoolJob 有 `killAll()` 方法，直接调用
- ChildJob 有 `getPids()` + `kill()` 方法，需逐个终止

### ChildJob 单任务流程

源文件：`packages/ee-core/src/jobs/child/index.ts`

```
new ChildJob()
  │
  ├─ this.jobs = {}               子进程映射（pid → JobProcess）
  ├─ this.config = getConfig().jobs || { messageLog: false }
  ├─ _initEvents()                 注册自动清理事件
  │   ├─ on(childProcessExit) → delete this.jobs[pid]
  │   └─ on(childProcessError) → delete this.jobs[pid]

exec(filepath, params, opt)
  │
  ├─ getFullpath(filepath) → jobPath    解析任务文件完整路径
  ├─ createProcess(opt) → proc          创建子进程
  ├─ proc.dispatch('run', jobPath, params)  发送运行命令
  └─ return proc

createProcess(opt)
  │
  ├─ 合并 processArgs: { type: 'childJob' }
  ├─ new JobProcess(this, options, this.config) → proc
  ├─ [proc.pid 存在] → this.jobs[pid] = proc
  └─ return proc

getPids() → Object.keys(this.jobs)    获取所有子进程 PID
kill(pid) → this.jobs[pid].kill()     终止指定子进程
```

### JobProcess 子进程管理

源文件：`packages/ee-core/src/jobs/child/jobProcess.ts`

```
new JobProcess(host, opt, config)
  │
  ├─ 定位 app.js 入口：
  │   currentFilePath = __filename
  │   appPath = path.join(dirname(currentFilePath), 'app.js')
  │   │  └─ 即 packages/ee-core/src/jobs/child/app.ts 编译后的 app.js
  │
  ├─ 配置 cwd：
  │   ├─ [未打包] → cwd = getBaseDir()
  │   └─ [已打包] → cwd = path.join(getBaseDir(), '..')
  │       │  └─ 打包后需跳出 app.asar，子进程才能访问 node_modules
  │
  ├─ 合并默认配置：
  │   defaultOptions = {
  │     processArgs: { type: 'childJob', eeConfig: getConfig() },
  │     processOptions: { cwd, env: allEnv(), stdio: ['ignore', 'pipe', 'pipe', 'ipc'] }
  │   }
  │
  ├─ 序列化参数：
  │   this.args.push(JSON.stringify(options.processArgs))
  │   │  └─ 子进程通过 process.argv[2] 接收此参数
  │
  ├─ fork(appPath, this.args, options.processOptions)
  │   this.child = fork(...)
  │   this.pid = this.child.pid
  │
  └─ _init() 注册事件监听
```

**_init() 事件监听**：
```
child.on('message', (m: ProcessMessage))
  │
  ├─ [m.channel === Processes.showException] → coreLogger.error(m.data)
  │   │  └─ 子进程异常消息，记录到日志
  │
  └─ [m.channel === Processes.sendToMain] → _eventEmit(m)
      │  └─ 业务消息，根据 eventReceiver 分发
      ├─ [eventReceiver === 'forkProcess'] → this.emitter.emit(event, data)
      │   │  └─ 仅发送到当前进程级 emitter（进程内监听）
      ├─ [eventReceiver === 'childJob'] → this.host.emit(event, data)
      │   │  └─ 仅发送到宿主 emitter（ChildJob/ChildPoolJob 级监听）
      └─ [其他/all] → 双重发送：host + emitter

child.stdout.on('data') → coreLogger.info   捕获子进程 stdout
child.stderr.on('data') → coreLogger.error  捕获子进程 stderr

child.on('exit', (code, signal)) → host.emit(Events.childProcessExit, { pid })
child.on('error', (err))          → host.emit(Events.childProcessError, { pid })
```

**消息类型定义**：

```ts
// 主进程 → 子进程
interface JobMessage {
  mid: string;        // 消息 ID（随机字符串）
  cmd: string;        // 命令（'run'）
  jobPath?: string;   // 任务文件路径
  jobParams?: unknown[];  // 任务构造参数
  jobFunc?: string;   // 实例方法名
  jobFuncParams?: unknown[]; // 方法参数
}

// 子进程 → 主进程
interface ProcessMessage {
  channel: string;    // 通道名（Processes.showException / Processes.sendToMain）
  eventReceiver: string;  // 接收者类型（Receiver.childJob / Receiver.forkProcess / all）
  event?: string;
  data?: unknown;
}
```

**对外操作方法**：

```
dispatch(cmd, jobPath, ...params)
  │
  ├─ mid = getRandomString()
  ├─ 构建 JobMessage { mid, cmd, jobPath, jobParams: params }
  └─ this.child.send(msg)

callFunc(jobPath, funcName, ...params)
  │
  ├─ fullPath = getFullpath(jobPath)  解析完整路径
  ├─ 构建 JobMessage { mid, cmd:'run', jobPath: fullPath, jobFunc: funcName, jobFuncParams: params }
  └─ this.child.send(msg)

kill(timeout = 1000)
  │
  ├─ this.child.kill('SIGINT')   先尝试优雅退出
  └─ setTimeout(timeout) → [未成功] this.child.kill('SIGKILL')  强制终止
```

### ChildApp 子进程应用

源文件：`packages/ee-core/src/jobs/child/app.ts`

```
模块初始化（子进程启动时立即执行）：
  │
  ├─ 解析 process.argv[2] 获取配置参数
  │   ├─ args.eeConfig → setConfig(args.eeConfig)
  │   │   │  └─ 子进程通过 IPC 参数接收配置，不从文件系统加载
  │   │   │     └─ 避免打包模式下无法读取配置文件的问题
  │   └─ [解析失败] → 使用默认配置
  │
  └─ loadException() → 子进程独立加载异常处理

new ChildApp()
  │
  ├─ this.jobMap = new Map()  任务实例缓存
  ├─ _initEvents()
  │   ├─ process.on('message', _handleMessage)
  │   └─ process.once('exit', (code) → 记录日志)

_handleMessage(m: JobMessage)
  │
  ├─ commands.indexOf(m.cmd) === -1 → return（忽略未知命令）
  └─ [m.cmd === 'run'] → this.run(m)

run(msg)
  │
  ├─ mod = requireFile(jobPath)  加载任务文件
  │
  ├─ [isClass(mod) || isBytecodeClass(mod)] → 类/字节码类
  │   ├─ [jobMap 未缓存] → new mod(...jobParams)，jobMap.set(jobPath, instance)
  │   ├─ [jobMap 已缓存] → jobMap.get(jobPath)
  │   ├─ [jobFunc 存在且为方法] → instance[jobFunc](...jobFuncParams)
  │   └─ [无 jobFunc] → instance.handle(...jobParams)（默认入口方法）
  │
  ├─ [isFunction(mod)] → 纯函数
  │   └─ mod(jobParams) → 直接调用
  │
  └─ [异常] → process.send({ channel: Processes.showException, data: String(e) })
      │  └─ 错误回传主进程，不会静默丢失
```

**类型任务 vs 函数型任务的行为差异**：

| 类型 | 首次执行 | 后续执行 | 指定方法 | 默认方法 |
|------|---------|---------|---------|---------|
| 类/字节码类 | `new mod()` 实例化并缓存 | 从 `jobMap` 取缓存实例 | `instance[jobFunc](params)` | `instance.handle(params)` |
| 纯函数 | 直接调用 | 直接调用 | 不适用 | `mod(params)` |

### ChildPoolJob 子进程池流程

源文件：`packages/ee-core/src/jobs/child-pool/index.ts`

```
new ChildPoolJob(opt = {})
  │
  ├─ 默认配置：
  │   min = 3, max = 6, strategy = 'polling'
  │   weights = [1, 1, 1, 1, 1, 1]  默认权重
  │
  ├─ 创建 LoadBalancer：
  │   lbOpt = { algorithm: 'polling', targets: [] }
  │   this.LB = new LoadBalancer(lbOpt)
  │
  ├─ getConfig().jobs → config
  └─ _initEvents()  注册自动清理事件
      ├─ on(childProcessExit) → _removeChild(pid)
      └─ on(childProcessError) → _removeChild(pid)

create(number = 3) → 创建子进程
  │
  ├─ [number < 0 或 > max] → throw Error
  ├─ [当前进程数 > max] → throw Error
  ├─ [number + 当前数 > max] → number = max - 当前数（限制上限）
  │
  ├─ 创建 number 个 JobProcess：
  │   options = { processArgs: { type: 'childPoolJob' } }
  │   for (i = 1; i <= number; i++):
  │     task = new JobProcess(this, options, this.config)
  │     _childCreated(task) → 注册到池和负载均衡器
  │
  └─ return Object.keys(this.children)  返回所有 PID 列表

_childCreated(childProcess)
  │
  ├─ this.children[pid] = childProcess  加入池
  ├─ LB.add({ id: pid, weight: weights[length-1] || 1 })  加入负载均衡器

run(filepath, params) → 执行任务
  │
  ├─ jobPath = getFullpath(filepath)  解析路径
  ├─ childProcess = await getChild()  从池中选择进程
  ├─ childProcess.dispatch('run', jobPath, params)  发送命令
  └─ return childProcess

getChild() → 选择子进程
  │
  ├─ [池为空] → create(1)，自动创建一个进程
  ├─ [池不为空] → LB.pickOne()，负载均衡器选择
  │   ├─ picked.id → this.children[picked.id]
  │   └─ [选中进程不存在] → throw Error

getBoundChild(boundId) → 绑定式获取
  │
  ├─ boundMap.get(boundId) → boundPid
  │   ├─ [children[boundPid] 存在] → return proc
  │   └─ [不存在] → 重新获取并绑定
  │       ├─ proc = getChild()
  │       └─ boundMap.set(boundId, proc.pid)

killAll(type = 'parallel')
  │
  ├─ [type === 'parallel'] → 所有进程同时 kill()
  └─ [type === 'sequence'] → 每秒 kill 一个（优雅退出）
```

**boundMap 绑定机制**：将特定标识（如用户 ID、会话 ID）绑定到固定子进程，确保同一标识的任务始终在同一个进程执行。绑定的进程退出后自动重新分配。

### LoadBalancer 负载均衡流程

源文件：`packages/ee-core/src/jobs/load-balancer/index.ts`

```
new LoadBalancer({ targets, algorithm })
  │
  ├─ this.targets = targets                    目标列表 [{id, weight}]
  ├─ this.algorithm = algorithm || 'polling'   算法名称
  ├─ this.params = {                           运行时参数
  │     currentIndex: 0,                       轮询当前索引
  │     weightIndex: 0,                        权重索引
  │     weightTotal: 0,                        权重总和
  │     connectionsMap: {},                     连接数映射
  │     cpuOccupancyMap: {},                   CPU占用映射
  │     memoryOccupancyMap: {}                 内存占用映射
  │   }
  ├─ this.scheduler = new Scheduler(algorithm) 算法调度器
  ├─ this.memoParams = memorizedParams()       算法参数映射
  └─ calculateWeightIndex()                    计算权重总和

pickOne(...params)
  │
  ├─ memoFn = memoParams[algorithm]  获取当前算法的参数函数
  ├─ memoFn(...params) → 算法参数数组
  └─ scheduler.calculate(targets, 算法参数) → 选出的目标
```

**memorizedParams() 算法参数映射**：

```ts
{
  polling:        () => [currentIndex, params]
  weights:        () => [weightTotal, params]
  random:         () => []
  specify:        (id) => [id]
  weightsPolling: () => [weightIndex, weightTotal, params]
  weightsRandom:  () => [weightTotal]
  minimumConnection:         () => [connectionsMap]
  weightsMinimumConnection:  () => [weightTotal, connectionsMap, params]
}
```

### Scheduler 算法调度器

源文件：`packages/ee-core/src/jobs/load-balancer/scheduler.ts`

```ts
class Scheduler {
  algorithm: string;

  calculate(tasks, ...params) → algorithms[algorithm](tasks, ...params)
  setAlgorithm(algorithm) → 切换算法（验证是否在 AlgorithmType 中）
}
```

Scheduler 是算法查找的中间层，将 `calculate()` 调用分派到对应的算法函数。

### 8 种负载均衡算法

源文件：`packages/ee-core/src/jobs/load-balancer/algorithm/`

| 算法 | 文件 | 参数 | 核心逻辑 |
|------|------|------|---------|
| polling | `polling.ts` | currentIndex, params | 按序号依次选择，currentIndex 循环递增 |
| weights | `weights.ts` | weightTotal, params | weight + random*weightTotal 取最大值 |
| random | `random.ts` | 无 | 随机选择一个目标 |
| specify | `specify.ts` | id | 按 id 查找指定目标 |
| minimumConnection | `minimumConnection.ts` | connectionsMap | 选连接数最少的目标 |
| weightsPolling | `weightsPolling.ts` | weightIndex, weightTotal, params | 加权轮询 |
| weightsRandom | `weightsRandom.ts` | weightTotal | 加权随机 |
| weightsMinimumConnection | `weightsMinimumConnection.ts` | weightTotal, connectionsMap, params | 加权最小连接数 |

**轮询算法详解**（`polling.ts`）：
```ts
function polling(tasks, currentIndex, context):
  task = tasks[currentIndex]
  context.currentIndex++ → currentIndex %= tasks.length  循环递增
  return task
```

**加权算法详解**（`weights.ts`）：
```ts
function weights(tasks, weightTotal, context):
  对每个 task: sum = (task.weight || 0) + random() * weightTotal
  取 sum 最大的 task → return
  weightIndex 循环递增
```

**最小连接数详解**（`minimumConnection.ts`）：
```ts
function minimumConnection(tasks, conMap):
  找 conMap[id] 最小（或为0）的 task → return
```

**指定ID算法详解**（`specify.ts`）：
```ts
function specify(tasks, id):
  遍历 tasks 找 id 匹配的 → return
```

## 关键设计决策

### 1. 配置通过 IPC 传递而非文件系统加载

子进程通过 `process.argv[2]` 接收序列化的配置（`eeConfig`），而非从文件系统读取配置文件。原因：
- 打包模式下配置文件可能被加密或打包进 asar，子进程无法直接读取
- 配置已在主进程中完成合并，子进程直接接收最终结果避免重复合并
- 子进程是独立 Node.js 环境，无法访问 Electron 主进程的内存状态

### 2. ee-core 作为 esbuild external（不打包进 main.js）

Job 系统依赖 ee-core 的 `app.js` 作为子进程入口文件。此文件必须存在于磁盘上（`child_process.fork()` 要求真实的文件路径），所以 ee-core 必须作为 esbuild 的 external，保留在 `node_modules` 中。这也解释了打包后 `cwd` 需要跳出 `app.asar` 的设计。

### 3. 类型任务的实例缓存（jobMap）

ChildApp 的 `jobMap` 缓存类实例，避免重复创建。这是性能优化——同一任务文件反复调用时，实例只创建一次，后续调用直接使用缓存实例的方法。但这也意味着实例状态会跨调用保持，业务代码需要注意状态管理。

### 4. jobFunc + jobFuncParams 的方法调用机制

`JobMessage` 支持 `jobFunc` 和 `jobFuncParams` 字段，允许主进程指定调用缓存实例的哪个方法。这比只支持 `handle()` 默认方法更灵活——业务类可以定义多个方法，主进程可选择性调用。

### 5. 双重事件分发（forkProcess / childJob / all）

`_eventEmit()` 根据子进程消息的 `eventReceiver` 字段决定分发目标：
- `forkProcess`：进程级 emitter，允许代码监听特定子进程的事件
- `childJob`：宿主级 emitter，允许代码监听所有子进程的同类事件
- `all`/其他：双重分发，两边都能收到

这提供了细粒度的消息路由能力——业务代码可以选择监听特定进程或全局事件。

### 6. kill 的优雅退出策略

`JobProcess.kill()` 先发 SIGINT（优雅退出信号），等待 timeout（默认 1 秒）后检查进程是否已退出。如果未退出则发 SIGKILL 强制终止。这给子进程处理退出逻辑的时间，避免数据丢失。

### 7. boundMap 会话绑定机制

ChildPoolJob 的 `boundMap` 支持将标识绑定到固定子进程，确保同一标识（如用户 ID）的任务始终在同一进程执行。如果绑定进程已退出，自动重新分配。这适合有状态的任务场景（如用户会话数据只在一个进程中维护）。

### 8. 进程池的 min/max 限制

ChildPoolJob 的 `min=3, max=6` 默认值平衡了并发能力和资源消耗：
- 最少 3 个进程提供基本并发能力
- 最多 6 个进程避免过度占用系统资源
- `create()` 调用时自动调整数量不超过 max
