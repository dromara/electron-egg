# FileLoader 核心机制

## 概述

FileLoader 是 ee-core 框架的通用文件加载器，负责扫描指定目录下的脚本文件，将导出内容按目录结构组织为嵌套属性对象。它是控制器加载、配置加载等上层模块的基础设施。

FileLoader 提供三种加载方法：`parse()`（同步文件扫描 + `require()`）、`parseFromRegistry()`（从全局注册表读取，打包模式）、`parseAsync()`（异步文件扫描 + `import()`，ESM 开发模式）。内部使用自实现的 `scanDirSync()/scanDirAsync()` 替代 globby 进行目录扫描，扫描的文件扩展名覆盖 `.js/.ts/.mts/.cts/.tsx/.jsx/.mjs/.cjs`。

导出内容经过 `_processExport()` 处理：类/字节码类/生成器函数/异步函数直接返回；普通函数根据 `call` 选项自动调用；`initializer` 回调允许上层模块自定义处理。处理后的导出通过 `_assignToTarget()` 按属性路径构建嵌套对象，并用 `FULLPATH`/`EXPORTS` 两个 Symbol 标记非原始类型导出。

## 核心流程

### 1. FileLoader 构造与选项

**源文件**：`packages/ee-core/src/core/loader/file_loader.ts`

**构造函数**：

```typescript
constructor(options: FileLoaderOptions) {
  if (!options.directory) {
    throw new Error('options.directory is required');
  }
  this.options = Object.assign({}, defaults, options);
}
```

**默认选项** (`defaults`)：

| 选项 | 默认值 | 说明 |
|------|--------|------|
| `caseStyle` | `'camel'` | 属性命名风格：`camel`（驼峰）、`lower`（首字母小写）、`upper`（首字母大写），或自定义函数 |
| `initializer` | `null` | 自定义初始化器，对导出内容做额外处理 |
| `call` | `true` | 是否自动调用函数导出 |
| `inject` | `undefined` | 函数调用时的注入参数 |
| `target` | `null` | 目标对象（已废弃，现在由 `load()` 内部创建） |
| `match` | `undefined` | 文件匹配模式（覆盖默认 `filePatterns()`） |

**两个 Symbol 标记**：

| Symbol | 说明 |
|--------|------|
| `FULLPATH` | `Symbol('LOADER_ITEM_FULLPATH')`，记录文件的完整路径 |
| `EXPORTS` | `Symbol('LOADER_ITEM_EXPORTS')`，标记导出已被 FileLoader 处理 |

**SCRIPT_EXTENSIONS 常量**：

```typescript
const SCRIPT_EXTENSIONS = ['.js', '.ts', '.mts', '.cts', '.tsx', '.jsx', '.mjs', '.cjs'];
```

这是目录扫描时识别脚本文件的扩展名列表，覆盖了 CJS、ESM、TypeScript 等主流格式。

### 2. 目录扫描：替代 globby

**源文件**：`packages/ee-core/src/core/loader/file_loader.ts`（`scanDirSync`/`scanDirAsync` 函数）

#### `scanDirSync(dir, prefix)`

自实现的同步递归目录扫描，替代 globby（避免 ESM-only 依赖在 CJS 模式下的问题）。

**流程**：

1. 检查目录是否存在（`fs.existsSync`），不存在则返回空数组
2. `fs.readdirSync(dir, { withFileTypes: true })` 获取目录条目
3. 对每个条目：
   - 目录 → 递归调用 `scanDirSync(fullPath, relPath)`
   - 文件 → 检查扩展名是否在 `SCRIPT_EXTENSIONS` 中，匹配则收集相对路径
4. 返回相对路径数组（如 `['user.js', 'admin/login.js']`）

#### `scanDirAsync(dir, prefix)`

异步版本，使用 `fs.promises.readdir` 和递归 `await`，流程完全相同。

**与 globby 的关键差异**：

- 不依赖 globby（ESM-only 包，在 CJS 模式下无法直接 `require()`）
- 扩展名过滤更精确（globby 使用 glob 模式 `**/*.js`，FileLoader 使用显式扩展名列表）
- 返回相对路径而非绝对路径，与后续 `getProperties()` 处理对齐

### 3. parse() 同步文件扫描

**源文件**：`packages/ee-core/src/core/loader/file_loader.ts`

**流程**：

```
1. 将 directory 视为数组（支持单个或多个目录）
2. 对每个 directory：
   a. scanDirSync(directory) → 相对路径列表
   b. 对每个相对路径 filepath：
      - fullpath = path.join(directory, filepath)
      - properties = getProperties(filepath, { caseStyle })
      - dirName = directory 最后一级目录名（如 'controller'）
      - exports = loadFile(fullpath)  // 同步 require
      - item = _processExport(exports, fullpath, properties, dirName)
      - 若 item 非 null 则加入 items
3. 返回 items 数组
```

**`getProperties(filepath, { caseStyle })`**（源文件 `packages/ee-core/src/utils/wrap.ts`）：

将相对文件路径转换为属性路径数组。例如：

| filepath | caseStyle | 结果 |
|----------|-----------|------|
| `'example.js'` | `'lower'` | `['example']` |
| `'admin/login.js'` | `'lower'` | `['admin', 'login']` |
| `'user_info.js'` | `'camel'` | `['userInfo']` |
| `'user-info.js'` | `'camel'` | `['userInfo']` |
| `'UserInfo.js'` | `'lower'` | `['userInfo']` |

**`defaultCamelize` 处理规则**：

1. 移除扩展名，按 `/` 分割为路径段
2. 对每个路径段：下划线/连字符后的字母转大写（`foo_bar` → `fooBar`）
3. 根据 `caseStyle` 调整首字母：
   - `lower`：首字母小写（控制器使用）
   - `upper`：首字母大写
   - `camel`：保持原样
4. 路径段只允许 `a-z0-9_-` 字符，否则抛出错误

### 4. parseFromRegistry() 注册表模式

**源文件**：`packages/ee-core/src/core/loader/file_loader.ts`

**流程**：

```
1. 从 options.registry 获取注册表（全局变量如 __EE_CONTROLLER_REGISTRY__）
2. 对每个注册表条目 entry：
   a. exports = entry.module
   b. ESM 互操作：若 __esModule 标记存在，提取 .default 属性
   c. fullpath = entry.fullpath
   d. properties = entry.properties（打包时由 esbuild 插件预计算）
   e. dirName = fullpath 倒数第二级目录名（如 'controller'）
   f. item = _processExport(exports, fullpath, properties, dirName)
   g. 若 item 非 null 则加入 items
3. 返回 items 数组
```

**与 parse() 的关键差异**：

- 无文件系统扫描，直接从注册表读取
- `properties` 已由打包插件预计算（复制了 `defaultCamelize` 的 `caseStyle: 'lower'` 逻辑）
- `dirName` 从 `fullpath` 中提取而非从 `directory` 参数获取
- ESM 互操作处理（`__esModule` + `.default` 提取）

### 5. parseAsync() 异步文件扫描

**源文件**：`packages/ee-core/src/core/loader/file_loader.ts`

流程与 `parse()` 完全对称，唯一区别：

- 使用 `scanDirAsync()` 替代 `scanDirSync()`
- 使用 `loadFileAsync()` 替代 `loadFile()`（动态 `import()` 替代 `require()`）

适用于 ESM 开发模式，控制器、服务等模块可以使用 ESM 格式。

### 6. _processExport() 导出处理

**源文件**：`packages/ee-core/src/core/loader/file_loader.ts`

**流程**：

```
1. 若 exports == null → 返回 null（跳过空导出）
2. 计算 pathName = dirName + '.' + properties.join('.')
3. 若 initializer 存在 → exports = initializer(exports, { pathName, path: fullpath })
4. 若是类或字节码类 → 设置 prototype.pathName 和 prototype.fullPath
5. 分类处理：
   a. 类/生成器函数/异步函数/字节码类 → 直接返回 { fullpath, properties, exports }
   b. 普通函数 + call=true → 自动调用 exports(inject)，返回调用结果
   c. 其他 → 返回 { fullpath, properties, exports }
```

**initializer 的作用**：

上层模块通过 `initializer` 回调对导出内容做自定义处理。控制器加载器使用 initializer 实现 `wrapClass()` 包装（类导出 → 方法映射）。若无 initializer，导出内容不做额外处理。

**类的 pathName/fullPath 设置**：

在类原型上设置 `pathName` 和 `fullPath`，为 IPC 通道路由和调试提供路径元数据。这些信息在控制器中间件函数的 `[FULLPATH]` 标记中也会用到。

**函数自动调用（`call` 选项）**：

当 `call=true` 时，普通函数导出会被自动调用（传入 `inject` 参数）。这在服务加载等场景中使用——服务文件导出初始化函数，自动调用后返回服务实例。类和异步/生成器函数不会被自动调用。

### 7. _assignToTarget() 嵌套对象构建

**源文件**：`packages/ee-core/src/core/loader/file_loader.ts`

**流程**：

```
1. 创建空 target 对象
2. 对每个 LoaderItem：
   a. 按 properties 数组逐层创建嵌套对象
   b. 最后一级属性 → 赋值为 exports
   c. 若 exports 是非原始类型 → 设置 [FULLPATH] 和 [EXPORTS] Symbol
   d. 中间级 → 确保对象存在（current[property] = current[property] || {}）
3. 返回 target
```

**示例**：

```
items = [
  { fullpath: '/app/controller/example.js', properties: ['example'], exports: { test: fn } },
  { fullpath: '/app/controller/admin/login.js', properties: ['admin', 'login'], exports: { auth: fn } },
]

target = {
  example: { test: fn, [FULLPATH]: '/app/controller/example.js', [EXPORTS]: true },
  admin: {
    login: { auth: fn, [FULLPATH]: '/app/controller/admin/login.js', [EXPORTS]: true }
  }
}
```

注意：`dirName` 前缀不在 `_assignToTarget()` 中处理——它只在 `_processExport()` 中用于构建 `pathName`。最终嵌套对象的外层键名由上层模块（如 ControllerLoader）在合并时添加。

### 8. load() 和 loadAsync() 入口方法

**源文件**：`packages/ee-core/src/core/loader/file_loader.ts`

#### `load()`

```typescript
load(): Record<string, unknown> {
  const items = this.options.registry ? this.parseFromRegistry() : this.parse();
  return this._assignToTarget(items);
}
```

- 若 `registry` 存在（打包模式） → `parseFromRegistry()`
- 否则（开发模式） → `parse()`

#### `loadAsync()`

```typescript
async loadAsync(): Promise<Record<string, unknown>> {
  const items = await this.parseAsync();
  return this._assignToTarget(items);
}
```

异步模式始终使用文件系统扫描（不支持注册表的异步读取——注册表在打包模式下已经预加载完毕）。

### 9. 底层工具函数

**源文件**：`packages/ee-core/src/core/utils/index.ts`

#### `loadFile(filepath)` — 同步文件加载

```
1. 获取文件扩展名
2. 非JS扩展名（不在 Module._extensions 中）→ fs.readFileSync 返回 Buffer
3. JS/CJS/JSC扩展名 → requireMod(filepath) 加载模块
4. ESM 互操作：若 obj.__esModule → 提取 obj.default
5. 返回模块导出或 Buffer
6. 加载失败 → 抛出包含文件路径的错误信息
```

`requireMod` 通过 `createRequire()` 创建，兼容 CJS 和 ESM 环境。`Module._extensions` 包含 Node.js 原生支持的扩展名以及 bytenode 注册的 `.jsc`。

#### `loadFileAsync(filepath)` — 异步文件加载

流程与 `loadFile` 对称，使用 `import()` 替代 `require()`，`fs.promises.readFile` 替代 `fs.readFileSync`。额外支持 `.mjs` 扩展名（即使不在 `Module._extensions` 中也通过 `import()` 加载）。

#### `callFn(fn, args, ctx)` — 函数调用

```typescript
async function callFn(fn, args, ctx) {
  args = args || [];
  if (!isFunction(fn)) return undefined;
  return ctx ? fn.call(ctx, ...args) : fn(...args);
}
```

若提供 `ctx`，使用 `fn.call(ctx, ...args)` 绑定上下文；否则直接调用 `fn(...args)`。注意该函数标记为 `async`，但实际调用可能是同步的——这是为了兼容异步方法返回 Promise 的场景。

#### `isBytecodeClass(exports)` — 字节码类检测

```typescript
function isBytecodeClass(exports) {
  if (!exports) return false;
  return String(exports).indexOf('[class') !== -1;
}
```

bytenode 编译的 `.jsc` 文件中的类在 `toString()` 时显示为 `[class XXX]`，通过检测此特征识别字节码类。

#### `getResolvedFilename(filepath, baseDir)` — 相对路径计算

将绝对路径转换为基于 baseDir 的相对路径，使用正斜杠分隔。

#### `filePatterns()` — 文件匹配模式

返回 `['**/*.js', '**/*.jsc']`，用于 globby 风格的文件过滤（当前已被 `scanDirSync/Async` 替代，但保留接口兼容）。

## 关键设计决策

1. **自实现目录扫描替代 globby**：globby 是 ESM-only 包，在 CJS 模式下无法直接 `require()`。FileLoader 自实现 `scanDirSync/scanDirAsync`，使用 `fs.readdir` + 扩展名过滤，消除了对 globby 的依赖。扫描的扩展名列表 `SCRIPT_EXTENSIONS` 覆盖了 `.js/.ts/.mts/.cts/.tsx/.jsx/.mjs/.cjs`，比 globby 的 glob 模式更精确。

2. **Symbol 标记而非属性注入**：使用 `FULLPATH` 和 `EXPORTS` 两个 Symbol 标记导出，而非直接在对象上添加字符串属性。Symbol 不会与业务代码的属性名冲突，也不会被 `JSON.stringify` 序列化，是元数据的理想载体。

3. **三种加载方法对应三种场景**：`parse()` 对应 CJS 开发模式（同步 `require`）；`parseFromRegistry()` 对应打包模式（无文件系统依赖）；`parseAsync()` 对应 ESM 开发模式（异步 `import`）。三种方法产出相同的 `LoaderItem[]` 结构，通过 `_processExport()` 和 `_assignToTarget()` 统一处理。

4. **initializer 回调实现上层定制**：FileLoader 不直接处理控制器包装、服务初始化等业务逻辑，而是通过 `initializer` 回调将定制点暴露给上层模块。这使 FileLoader 保持通用性，同时允许 ControllerLoader、ServiceLoader 等模块注入各自的处理逻辑。

5. **pathName 和 fullPath 元数据**：在类原型上设置 `pathName`（如 `controller.example`）和 `fullPath`（如 `/app/controller/example.js`），这些信息服务于 IPC 通道路由、调试日志和错误追踪。中间件函数上的 `[FULLPATH]` 标记（如 `/app/controller/example.js#ExampleController.test()`）进一步细化到方法级别。

6. **函数自动调用（call 选项）**：`call=true` 时普通函数导出被自动调用，传入 `inject` 参数。这在服务文件（导出初始化函数）场景中非常有用——函数被调用后返回服务实例，而非函数本身。类、生成器函数、异步函数不受此影响，直接返回。

7. **ESM 互操作一致性**：`parseFromRegistry()` 和 `core/utils/loadFile()` 都使用相同的 ESM 互操作逻辑：检查 `__esModule` 标记，存在则提取 `.default`。这确保了打包模式和开发模式对 ESM 模块的处理结果一致。

8. **属性命名风格可配置**：`caseStyle` 支持 `lower`/`upper`/`camel` 三种预设风格和自定义函数。控制器使用 `lower`（与打包插件的 `caseStyle: 'lower'` 一致），其他模块可以使用不同风格。自定义函数模式允许完全控制属性路径转换逻辑。

## 模块关系

```
core/loader/file_loader.ts (FileLoader)
  ├── 依赖 → utils/type_check (isClass, isFunction, isPrimitive, isGeneratorFunction, isAsyncFunction)
  ├── 依赖 → core/utils (loadFile, loadFileAsync, isBytecodeClass)
  ├── 依赖 → utils/wrap (getProperties)
  ├── 依赖 → types (FileLoaderOptions)
  └── 被依赖 → controller/controller_loader.ts (核心使用方)
  └── 被依赖 → 其他需要文件扫描的加载器

core/utils/index.ts
  ├── 依赖 → utils/type_check (isFunction)
  ├── 依赖 → bytenode (注册 .jsc 扩展名)
  ├── 依赖 → module (createRequire, Module._extensions)
  └── 被依赖 → core/loader/file_loader.ts (loadFile, loadFileAsync, isBytecodeClass)
  └── 被依赖 → controller/controller_loader.ts (callFn, isBytecodeClass)
  └── 被依赖 → config/config_loader.ts (通过 loader/index.ts 间接使用 loadFile)

utils/wrap.ts
  ├── 依赖 → utils/type_check (isFunction)
  └── 被依赖 → core/loader/file_loader.ts (getProperties)
```

FileLoader 是框架的基础设施层——控制器加载器、配置加载器（通过 `loadFile`）等都依赖它。它的通用设计（initializer 回调、caseStyle 可配置、三种加载方法）使得不同上层模块可以共享同一套文件扫描和组织逻辑，只需注入各自的定制参数。
