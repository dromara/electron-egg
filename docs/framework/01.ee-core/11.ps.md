# 进程状态与路径

## 概述

ee-core 的进程状态与路径模块（ps）提供环境检测、进程类型判断和路径获取的基础能力，是框架各模块获取运行时环境信息的统一入口。模块遵循"集中管理、单一来源"的设计原则——所有路径和环境信息均从 `process.env` 读取，这些环境变量由 `boot.ts` 在框架启动最早阶段集中设置。

核心能力：
- **环境检测**：开发/生产环境判断、进程类型判断（主进程/渲染进程/子进程）
- **路径获取**：20+ 个路径函数，覆盖项目根目录、electron 目录、日志目录、数据目录、打包后资源目录等
- **进程工具**：退出进程、格式化 IPC 消息、命令行参数获取

## 核心流程

### 环境变量设置流程

`boot.ts` 在框架启动最早阶段将关键信息写入 `process.env`，ps 模块所有函数从此读取：

```
boot.ts init()
  │
  ├─ 设置 EE_ENV (dev/local/prod)
  ├─ 设置 EE_APP_NAME (应用名称)
  ├─ 设置 EE_APP_VERSION (应用版本)
  ├─ 设置 EE_BASE_DIR (项目根目录)
  ├─ 设置 EE_ELECTRON_DIR (electron 目录)
  ├─ 设置 EE_USER_HOME (用户主目录)
  ├─ 设置 EE_APP_DATA (应用数据目录)
  ├─ 设置 EE_APP_USER_DATA (Electron appUserData 目录)
  ├─ 设置 EE_EXEC_DIR (可执行文件目录)
  ├─ 设置 EE_IS_PACKAGED (是否已打包: 'true'/'false')
  ├─ 设置 EE_SOCKET_PORT (Socket 端口)
  └─ 设置 EE_HTTP_PORT (HTTP 端口)
```

相关源文件：`packages/ee-core/src/ps/index.ts` 中各环境变量读取函数

### 进程类型判断流程

Electron 应用中有三种进程类型，ps 模块通过 `process.type` 和环境变量区分：

```
processType()
  │
  ├─ process.type === 'browser' → 主进程（Electron 主进程）
  ├─ process.type === 'renderer' → 渲染进程（浏览器窗口）
  ├─ ELECTRON_RUN_AS_NODE === 1 → 子进程（child_process.fork 创建）
  └─ 其他 → 未知类型
```

相关源文件：`packages/ee-core/src/ps/index.ts` 中 `isMain()`、`isRenderer()`、`isForkedChild()`、`processType()` 函数

### 路径获取逻辑

不同路径函数根据环境（开发/生产）和平台（Windows/macOS/Linux/OpenHarmony）返回不同值：

```
开发环境 vs 生产环境路径对照：

getDataDir():
  dev  → {baseDir}/data
  prod → {userHome}/.{appName}/data
  prod + openharmony → {appUserData}/{appName}/data

getLogDir():
  dev  → {baseDir}/logs
  prod → {userHome}/.{appName}/logs
  prod + openharmony → {appUserData}/{appName}/logs

getExtraResourcesDir():
  prod + Windows/Linux → {execDir}/resources/extraResources
  prod + macOS         → {execDir}/../Resources/extraResources
  dev                   → {execDir}/build/extraResources

getRootDir():
  dev  → getBaseDir() (项目根目录)
  prod → getAppUserDataDir() (应用数据目录)

getUserHomeDir():
  默认 → process.env.EE_USER_HOME
  openharmony → getAppUserDataDir() (权限限制)
```

相关源文件：`packages/ee-core/src/ps/index.ts` 中各路径获取函数

### 子进程类型判断流程

通过解析 `process.argv[2]` 的 JSON 参数判断子进程类型：

```
isChildJob():
  JSON.parse(argv[2]).type === 'childJob' → true

isChildPoolJob():
  JSON.parse(argv[2]).type === 'childPoolJob' → true

exitChildJob(code):
  JSON.parse(argv[2]).type === 'childJob' → process.exit(code)
  解析失败 → process.exit(code)（安全退出）
```

相关源文件：`packages/ee-core/src/ps/index.ts` 中 `isChildJob()`、`isChildPoolJob()`、`exitChildJob()` 函数

## 关键设计决策

### 1. 所有路径信息从 process.env 读取而非实时计算

路径信息由 `boot.ts` 在启动最早阶段一次性写入 `process.env`，后续所有模块通过 ps 模块读取。这种设计：
- **避免重复计算**：打包后路径在不同平台有不同逻辑，集中设置避免各模块重复判断
- **确保一致性**：所有模块获取的路径信息来自同一来源
- **支持子进程**：fork 创建的子进程继承父进程的 env，自动获得正确的路径信息

### 2. OpenHarmony 特殊路径处理

OpenHarmony 系统中用户主目录可能不可写，因此：
- `getDataDir()` 和 `getLogDir()` 在 OpenHarmony 生产环境使用 `{appUserData}/{appName}/` 而非 `{userHome}/.{appName}/`
- `getUserHomeDir()` 在 OpenHarmony 返回 `appUserData` 而非真实用户主目录
- `getExtraResourcesDir()` 有 OpenHarmony 专属路径逻辑

### 3. macOS 打包后路径差异处理

macOS 应用包结构中，可执行文件位于 `Contents/MacOS/`，而资源位于 `Contents/Resources/`，因此 `getExtraResourcesDir()` 在 macOS 使用 `../Resources/extraResources` 而非 `resources/extraResources`。

### 4. 开发环境与生产环境路径分离

开发环境使用项目根目录下的子路径（方便直接访问源码和配置），生产环境使用用户主目录下的隐藏目录（符合桌面应用数据存储惯例）。这种分离确保：
- 开发时日志/数据在项目目录内可见
- 生产时数据/日志持久化到用户目录，不受应用更新影响

### 5. 子进程参数通过 argv[2] JSON 传递

`child_process.fork()` 创建的子进程通过 `argv[2]` 接收 JSON 格式的启动参数，包含 `type` 字段标识子进程类型（childJob/childPoolJob）。这比通过 env 传递更灵活，因为同一 env 可能被多个 fork 子进程共享。

## 模块关系

```
ps/index.ts
  ├─ imports → utils/is.ts (is.openharmony, is.macOS 平台判断)
  ├─ imports → path (路径拼接)
  │
  ├─ 被导入 → app/boot.ts (设置 process.env 的源头)
  ├─ 被导入 → log/logger.ts (getLogDir 日志目录、isDev 环境判断)
  ├─ 被导入 → config/configLoader.ts (getElectronDir 配置加载路径)
  ├─ 袍导入 → controller/controllerLoader.ts (getElectronDir 控制器扫描路径)
  ├─ 袍导入 → cross/crossProcess.ts (getExtraResourcesDir, isPackaged, isDev, getBaseDir)
  ├─ 袍导入 → electron/app/index.ts (isPackaged, getBaseDir, getExecDir)
  ├─ 袍导入 → storage/sqliteStorage.ts (getDataDir 数据目录)
  ├─ 袍导入 → socket/socketServer.ts (getSocketPort, getHttpPort)
  ├─ 袍导入 → jobs/childJob.ts (isForkedChild, getElectronDir)
  ├─ 袍导入 → exception/index.ts (isForkedChild, isRenderer, isDev, isMain)
  └─ 袍导入 → loader/index.ts (getElectronDir 模块路径解析)

  ps 模块是 ee-core 中被导入最广泛的模块之一，几乎所有模块都需要路径或环境信息。
```
