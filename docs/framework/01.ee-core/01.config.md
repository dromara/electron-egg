# 配置加载系统

## 概述

ee-core 的配置加载系统采用三层合并策略：框架内置默认配置 → 业务默认配置 (`config.default`) → 环境特定配置 (`config.{env}`)。系统支持两种加载模式——打包模式从全局注册表读取（无文件系统依赖），开发模式从文件系统加载。配置文件支持函数导出，接收 `appInfo` 参数实现动态配置。

配置加载在启动流程中紧随异常处理之后执行，是日志、通信服务、Electron 功能等后续模块的基础数据源。

## 核心流程

### 1. 配置模块入口

**源文件**：`packages/ee-core/src/config/index.ts`

模块级状态：`config` 变量持有运行时配置（初始为 `null`）。

**四个导出函数**：

| 函数 | 说明 |
|------|------|
| `loadConfig()` | 创建 `ConfigLoader`，调用 `load()`，将结果赋值给 `config`。在 `boot.ts` 的 `init()` 中被调用 |
| `getConfig()` | 惰性加载：若 `config` 为 `null` 则自动触发 `loadConfig()`。通常在框架启动后使用 |
| `setConfig(cfg)` | 直接设置配置对象。用于子进程接收主进程传递的配置（打包输出目录可能缺少配置文件） |
| `getAppInfo()` | 从 `process.env` 构建 `AppInfo` 对象（name、baseDir、electronDir、env、root） |

**`getAppInfo()` 返回结构**：

```typescript
{
  name: process.env.EE_APP_NAME,
  baseDir: process.env.EE_BASE_DIR,
  electronDir: process.env.EE_ELECTRON_DIR,
  env: process.env.EE_ENV,       // 默认 'prod'
  root: process.env.EE_BASE_DIR,
}
```

### 2. ConfigLoader 加载流程

**源文件**：`packages/ee-core/src/config/config_loader.ts`

**`ConfigLoader.load()` 完整流程**：

```
1. timing.start('Load Config')
2. _AppConfig()                — 加载业务配置（config.default + config.{env}）
3. defaultConfig()             — 加载框架内置默认配置
4. extend(true, defaultConf, appConfig) — 深度合并：业务配置覆盖默认配置
5. timing.end('Load Config')
6. 返回合并后的 Config 对象
```

**`_AppConfig()` 流程**：

按照固定顺序加载两个配置文件名：

```typescript
const names = ['config.default', `config.${env()}`];
```

即先加载 `config.default`，再加载 `config.local`（开发环境）或 `config.prod`（生产环境）。每个文件通过 `_loadConfig()` 加载，使用 `extend(true, target, config)` 深度合并到同一个目标对象。

**`_loadConfig()` 双模式机制**：

#### 打包模式（Bundle Mode）

当 `globalThis.__EE_CONFIG_REGISTRY__` 存在时：

1. 按 `filename` 查找注册表条目（`find(e => e.filename === filename)`）
2. 若未找到，返回 `null`（静默跳过）
3. ESM 互操作处理：检查 `__esModule` 标记，若有则提取 `.default` 属性（esbuild 打包会将配置模块包装为 `{ __esModule: true, default: fn }`）
4. 函数导出处理：若 `mod` 是函数，传入 `appInfo` 参数调用，返回动态配置
5. 对象导出：直接返回

#### 开发模式（Dev Mode）

当注册表不存在时：

1. 构建文件路径：`path.join(dirpath, 'config', filename)`，如 `{electronDir}/config/config.default`
2. 调用 `loadFile(filepath, appInfo)` 加载（来自 `loader/index.ts`）
3. 若文件不存在或加载失败，**静默跳过**（返回 `null`），不抛出异常。这是因为子进程在打包输出目录中可能缺少配置文件
4. `loadFile()` 本身对配置文件的函数导出会自动处理：传入 `appInfo` 调用函数并返回结果

**注册表条目结构** (`ConfigRegistryEntry`)：

```typescript
interface ConfigRegistryEntry {
  filename: string;   // 如 'config.default', 'config.prod'
  module: unknown;    // 配置模块（对象或函数）
}
```

## 配置合并链

```
ee-core defaultConfig()
  └── 深度合并 ← config.default（业务默认配置）
        └── 深度合并 ← config.local / config.prod（环境特定配置）
```

**合并规则**：

- 使用 `extend(true, target, source)` 深度合并
- 后加载的配置覆盖同名属性
- 数组不做合并，直接替换
- 对象类型递归合并

**示例**：

```javascript
// ee-core 默认：windowsOption.width = 980
// config.default：windowsOption.width = 1200
// config.prod：windowsOption.height = 800
// 最终结果：windowsOption = { width: 1200, height: 800, ...其他默认值保留 }
```

## 关键设计决策

1. **三层合并而非两层**：框架默认 → 业务默认 → 环境特定。这确保了即使业务没有覆盖某个配置项，框架也能提供合理的默认值。开发者只需要在 `config.default` 中覆盖自己关心的项，其余自动继承框架默认。

2. **配置文件支持函数导出**：配置文件可以导出函数，接收 `appInfo` 参数（包含 name、baseDir、electronDir、env 等）。这允许配置根据运行环境动态生成，如开发环境启用 DevTools、生产环境关闭。

3. **打包模式与开发模式双路径**：打包模式下配置由 esbuild 插件预注册到 `globalThis.__EE_CONFIG_REGISTRY__`，运行时无需文件系统扫描。开发模式下从文件系统加载。两种模式的 `_loadConfig()` 实现完全独立，但产出相同结构的配置对象。

4. **ESM 互操作处理**：esbuild 打包会将 ESM 配置模块包装为 `{ __esModule: true, default: fn }`。`_loadConfig()` 只在 `__esModule` 标记存在时提取 `.default`，避免误拆 CJS 模块的正常导出。这与 `FileLoader.parseFromRegistry()` 的处理逻辑一致。

5. **子进程配置传递**：`setConfig(cfg)` 允许主进程直接向子进程传递配置对象，而非让子进程从文件系统重新加载。这是因为子进程（`child_process.fork`）的工作目录可能是打包输出目录，其中可能缺少原始配置文件。

6. **静默跳过缺失配置文件**：开发模式下 `_loadConfig()` 对文件不存在的情况返回 `null` 而非抛出异常。这保证了即使没有 `config.local.js`（很多项目不需要），框架也能正常启动。

7. **Timing 计量**：`ConfigLoader` 使用 `Timing` 类记录配置加载耗时，便于性能分析。

## API 导出

### config/index.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `loadConfig()` | 函数 | 加载并合并配置，赋值给模块级 `config` |
| `getConfig()` | 函数 | 惰性获取配置（未加载时自动触发 `loadConfig()`） |
| `setConfig(cfg)` | 函数 | 直接设置配置（子进程使用） |
| `getAppInfo()` | 函数 | 从 `process.env` 构建 `AppInfo` |

### config/config_loader.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `ConfigLoader` | 类 | 配置加载器 |
| `ConfigLoader.load()` | 方法 | 执行完整加载流程，返回合并后的 `Config` |
| `ConfigLoader.timing` | 属性 | `Timing` 实例，记录加载耗时 |

### config/default_config.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `defaultConfig()` | 函数 | 返回完整的框架默认 `Config` 对象 |

## 模块关系

```
config/index.ts
  ├── 依赖 → config/config_loader.ts (ConfigLoader)
  ├── 依赖 → types (AppInfo, Config 类型)
  └── 被依赖 → app/boot.ts (loadConfig 调用)
  └── 被依赖 → controller/controller_loader.ts (getConfig 间接使用)
  └── 被依赖 → exception/index.ts (getConfig 读取退出配置)
  └── 被依赖 → 所有需要配置的模块

config/config_loader.ts
  ├── 依赖 → ps (appName, env, getElectronDir, getBaseDir, getRootDir)
  ├── 依赖 → utils/extend (extend 深度合并)
  ├── 依赖 → loader (loadFile 文件加载)
  ├── 依赖 → utils/type_check (isFunction)
  ├── 依赖 → core/utils/timing (Timing)
  └── 依赖 → config/default_config.ts (defaultConfig)
  └── 读取 → globalThis.__EE_CONFIG_REGISTRY__ (打包模式注册表)

config/default_config.ts
  ├── 依赖 → ps (isDev, getBaseDir, getLogDir)
  └── 依赖 → const/channel (SocketIO 常量)
```

配置模块是框架的基础数据源——几乎所有后续模块（日志、通信、Electron、控制器等）都通过 `getConfig()` 获取运行参数。配置的加载时机（`init()` 中紧跟 `loadException()`）确保了后续模块启动时配置已经就绪。
