# 加载器工具

## 概述

ee-core 的加载器工具模块（loader）提供文件加载、执行和路径解析的基础能力，是控制器加载、配置加载和任务执行的核心依赖。模块围绕"文件导出类型决定加载行为"的设计理念，提供五种差异化函数：

- **loadFile**：加载文件，普通函数导出自动执行（用于配置文件）
- **requireFile**：加载文件，不做任何处理（用于子进程任务加载）
- **execFile**：加载并运行文件，类导出实例化、函数导出执行（用于控制器/任务）
- **resolveModule**：解析模块绝对路径，支持字节码(.jsc)回退
- **getFullpath**：获取 electron 目录下文件的绝对路径

## 核心流程

### loadFile 流程

```
loadFile(filepath, ...inject)
  │
  ├─ 1. 路径处理
  │     非绝对路径 → 拼接 getElectronDir()
  │     绝对路径 → 直接使用
  │
  ├─ 2. resolveModule(fullpath)
  │     尝试 require.resolve()
  │     失败 → 后缀规则回退 (.js→.jsc, .default/.prod→.jsc)
  │     返回 undefined 表示无法解析
  │
  ├─ 3. 存在性检查
  │     fullpath 或 resolveModule 结果不存在 → fs.existsSync 检查
  │     不存在 → 抛出异常
  │
  ├─ 4. coreLoadFile(fullpath)
  │     加载模块导出内容
  │
  └─ 5. 导出类型判断
      ├─ isFunction(ret) && !isClass(ret) && !isBytecodeClass(ret)
      │   → 普通函数：自动执行 ret(...inject)，返回执行结果
      │   （典型场景：config.default.js(appInfo) 接收 appInfo 参数）
      └─ 其他（类、对象、字节码类）
          → 返回原始导出，不做处理
```

相关源文件：`packages/ee-core/src/loader/index.ts` 中 `loadFile()` 函数

### requireFile 流程

```
requireFile(filepath)
  │
  └─ coreLoadFile(filepath) → 直接返回模块导出内容
     不做路径解析、不做自动执行、不做类型判断
```

相关源文件：`packages/ee-core/src/loader/index.ts` 中 `requireFile()` 函数

### execFile 流程

```
execFile(filepath, ...inject)
  │
  ├─ coreLoadFile(filepath) → 加载模块导出内容
  │
  └─ 导出类型判断
      ├─ isClass(ret) || isBytecodeClass(ret)
      │   → 类/字节码类：new Constructor(...inject)，返回实例
      │   （典型场景：控制器类实例化，注入 appInfo）
      ├─ isFunction(ret)
      │   → 普通函数：ret(...inject)，返回执行结果
      │   （典型场景：任务函数执行）
      └─ 其他
          → 直接返回原始导出
```

相关源文件：`packages/ee-core/src/loader/index.ts` 中 `execFile()` 函数

### resolveModule 流程

```
resolveModule(filepath)
  │
  ├─ 1. 尝试 require.resolve(filepath)
  │     成功 → 返回绝对路径
  │     失败 ↓
  │
  ├─ 2. 后缀规则回退
  │     filepath.endsWith('.default') 或 '.prod'
  │       → 尝试 filepath + '.jsc' (字节码版本)
  │     filepath.endsWith('.js')
  │       → 尝试 filepath + 'c' (.js → .jsc)
  │
  ├─ 3. 文件系统直接检查
  │     fullpath 不存在 → 尝试 filepath + '.js'
  │     仍不存在 → 尝试 filepath + '.jsc'
  │
  └─ 4. 最终判断
      filepath 存在 → 返回 filepath
      fullpath 存在 → 返回 fullpath
      都不存在 → console.warn + 返回 undefined
```

相关源文件：`packages/ee-core/src/loader/index.ts` 中 `resolveModule()` 函数

### getFullpath 流程

```
getFullpath(filepath)
  │
  ├─ 非绝对路径 → 拼接 getElectronDir()
  │
  ├─ resolveModule(filepath) → 尝试解析
  │
  └─ 结果不存在 → 抛出异常
     结果存在 → 返回绝对路径
```

相关源文件：`packages/ee-core/src/loader/index.ts` 中 `getFullpath()` 函数

## 关键设计决策

### 1. 三种加载函数差异化行为

ee-core 明确区分了三种加载场景：

| 函数 | 路径解析 | 自动执行 | 典型场景 |
|------|----------|----------|----------|
| `loadFile` | 相对路径基于 electronDir | 普通函数自动执行 | 配置文件加载 |
| `requireFile` | 无（需传绝对路径） | 无 | 子进程任务加载 |
| `execFile` | 无（需传绝对路径） | 类实例化/函数执行 | 控制器实例化 |

差异化行为避免了"一刀切"的问题——配置文件需要自动执行以传入 appInfo，而子进程任务需要拿到原始模块引用。

### 2. 字节码(.jsc)回退机制

`resolveModule()` 的后缀规则回退是加密场景的关键支持。生产环境中，代码可能被编译为字节码(.jsc)，文件命名规则如 `config.default.jsc` 替代 `config.default.js`。加载器在 `require.resolve()` 失败后按规则查找字节码版本，确保加密打包后的应用仍能正常加载模块。

### 3. isClass + isBytecodeClass 双重类检测

`execFile()` 和 `loadFile()` 同时使用 `isClass()` 和 `isBytecodeClass()` 检测类导出。`isClass()` 通过检查 `prototype` 的 `writable` 属性识别 ES6 class；`isBytecodeClass()` 由 `core/utils` 提供，识别经过字节码编译的类。双重检测确保加密和未加密场景都能正确区分类和函数。

### 4. 模块不可解析时返回 undefined 而非抛异常

`resolveModule()` 在无法解析模块时返回 `undefined` 并输出 `console.warn`，而非抛出异常。这是因为在控制器/配置的扫描流程中，某些文件可能有意不存在（如生产环境没有 `config.local.js`），调用方需要根据返回值决定后续行为。

### 5. coreLoadFile 作为底层抽象

所有加载函数最终调用 `core/utils` 中的 `coreLoadFile()`，这是框架最底层的文件加载抽象。它处理实际的 require/import 逻辑，包括字节码文件的加载。上层函数围绕此底层添加路径解析和类型判断逻辑，形成了清晰的分层。

## API 导出

### `packages/ee-core/src/loader/index.ts`

| 导出 | 类型 | 说明 |
|------|------|------|
| `loadFile(filepath, ...inject)` | function | 加载文件，普通函数导出自动执行 |
| `requireFile(filepath)` | function | 加载文件，返回原始导出 |
| `execFile(filepath, ...inject)` | function | 加载并运行文件，类实例化/函数执行 |
| `resolveModule(filepath)` | function | 解析模块绝对路径，支持字节码回退 |
| `getFullpath(filepath)` | function | 获取 electron 目录下文件的绝对路径 |

**函数签名与行为对比**：

```typescript
// loadFile: 相对路径 → 解析 → 存在性检查 → 加载 → 类型判断 → 自动执行
loadFile(filepath: string, ...inject: unknown[]): unknown

// requireFile: 绝对路径 → 直接加载，不做任何处理
requireFile(filepath: string): unknown

// execFile: 绝对路径 → 加载 → 类实例化或函数执行
execFile(filepath: string, ...inject: unknown[]): unknown

// resolveModule: 尝试 require.resolve → 后缀回退 → 文件系统检查 → 返回路径或 undefined
resolveModule(filepath: string): string | undefined

// getFullpath: 相对路径 → 解析 → 抛异常如果不存在
getFullpath(filepath: string): string
```

**典型使用示例**：

```typescript
// 配置加载：loadFile 自动执行 config.default.js(appInfo)
const config = loadFile('config/config.default', appInfo);

// 子进程任务加载：requireFile 获取原始模块引用
const taskModule = requireFile('/path/to/jobs/task.js');

// 控制器实例化：execFile 创建控制器类实例
const controller = execFile('/path/to/controller/example.js', appInfo);

// 模块路径解析：resolveModule 支持字节码回退
const modulePath = resolveModule('config/config.default'); // → .jsc if encrypted

// 文件路径确认：getFullpath 确保文件存在
const fullPath = getFullpath('controller/example'); // throws if not found
```

## 模块关系

```
loader/index.ts
  ├─ imports → utils/type_check.ts (isFunction, isClass 类型判断)
  ├─ imports → core/utils/index.ts (coreLoadFile, isBytecodeClass 底层加载)
  ├─ imports → ps/index.ts (getElectronDir 获取 electron 目录)
  ├─ imports → fs (fs.existsSync 存在性检查)
  ├─ imports → path (path.isAbsolute, path.join 路径处理)
  │
  ├─ 被导入 → config/configLoader.ts (loadFile 加载配置文件)
  ├─ 被导入 → controller/controllerLoader.ts (execFile 实例化控制器)
  ├─ 被导入 → jobs/childJob.ts (requireFile 加载子进程任务模块)
  ├─ 被导入 → jobs/childPoolJob.ts (requireFile 加载子进程池任务模块)
  └─ 被导入 → core/fileLoader.ts (resolveModule 解析模块路径)

  依赖链：
  loader/index.ts → core/utils/index.ts → 底层 require/import 实现
  loader/index.ts → utils/type_check.ts → JavaScript 类型判断
  loader/index.ts → ps/index.ts → 环境路径获取
```

### 与控制器加载的关系

```
controller/controllerLoader.ts
  │
  ├─ 扫描模式（非打包）:
  │     globby 扫描 electron/controller/ 目录
  │     → getFullpath() 确认文件存在
  │     → execFile(filepath, appInfo) 实例化控制器
  │
  └─ 注册表模式（打包）:
      global.__EE_CONTROLLER_REGISTRY__ 中的路径
      → requireFile(filepath) 获取模块引用
      → execFile() 实例化控制器
```

### 与配置加载的关系

```
config/configLoader.ts
  │
  ├─ 非打包模式:
  │     loadFile('config/config.default', appInfo) → 自动执行，注入 appInfo
  │     loadFile('config/config.local', appInfo) → 同上
  │     extend() 合并多层配置
  │
  └─ 打包模式:
      global.__EE_CONFIG_REGISTRY__ 中的函数
      → 直接调用，注入 appInfo
```
