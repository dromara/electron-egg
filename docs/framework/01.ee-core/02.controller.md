# 控制器加载系统

## 概述

控制器是 ee-core 框架业务逻辑的核心载体。每个前端 IPC 请求最终路由到对应控制器的方法。控制器加载系统负责扫描 `electron/controller/` 目录下的所有文件，将导出的类转换为方法映射对象，将导出的函数直接注册。

系统支持同步加载（CJS 项目使用 `require()`）和异步加载（ESM 项目使用 `import()`），并支持打包模式从全局注册表读取和开发模式从文件系统扫描两种路径。类控制器通过 `wrapClass()` 和 `methodToMiddleware()` 两层包装，确保并发安全——每次 IPC 调用创建新的控制器实例。

## 核心流程

### 1. 控制器模块入口

**源文件**：`packages/ee-core/src/controller/index.ts`

模块级状态：

- `controllers`：已加载的控制器方法映射（初始 `null`）
- `loading`：异步加载进行中标志（防止同步/异步并发竞争）

**四个导出函数**：

| 函数 | 说明 |
|------|------|
| `loadController()` | 同步加载：创建 `ControllerLoader`，调用 `load()`。若 `loading=true` 则抛出错误 |
| `loadControllerAsync()` | 异步加载：设置 `loading=true`，创建 `ControllerLoader`，调用 `loadAsync()`，完成后在 `finally` 中清除标志 |
| `getControllers()` | 惰性获取：若未加载则自动触发 `loadController()`；若异步加载中则抛出错误 |
| `getController()` | `getControllers()` 的别名 |

**并发安全设计**：

- `loading` 标志防止同步和异步加载同时进行
- 异步加载期间调用同步方法会抛出明确错误信息
- `loadControllerAsync()` 使用 `try/finally` 确保标志始终被清除

### 2. ControllerLoader 加载流程

**源文件**：`packages/ee-core/src/controller/controller_loader.ts`

#### `load()` 方法（同步）

```
1. timing.start('Load Controller')
2. 检查 globalThis.__EE_CONTROLLER_REGISTRY__（打包模式）
3. 构建 FileLoader 选项：
   - caseStyle: 'lower'
   - directory: path.join(getElectronDir(), 'controller')
   - registry: 若存在则传入
   - initializer: 类/字节码类 → 设置 pathName/fullPath → wrapClass()
4. 创建 FileLoader 并调用 load()
5. timing.end('Load Controller')
6. 返回控制器方法映射
```

#### `loadAsync()` 方法（异步）

流程与 `load()` 相同，唯一区别是使用 `FileLoader.loadAsync()`（异步文件扫描 + `import()`）。

#### initializer 回调逻辑

```typescript
initializer: (obj, options) => {
  if (isClass(obj) || isBytecodeClass(obj)) {
    // 设置路径信息到类原型
    obj.prototype.pathName = options.pathName;
    obj.prototype.fullPath = options.path;
    // 类控制器 → wrapClass() 包装
    return wrapClass(obj);
  }
  // 函数控制器 → 直接返回，不做包装
  return obj;
}
```

**分类处理**：

- **类导出**（`isClass()` 或 `isBytecodeClass()` 检测为真）：设置 `pathName`/`fullPath` 到原型，通过 `wrapClass()` 包装为方法映射
- **函数导出**：不做任何包装，直接注册。FileLoader 的 `call` 选项默认为 `true`，会将函数导出自动调用（但函数返回的对象会被保留）

### 3. wrapClass() 类包装

**源文件**：`packages/ee-core/src/controller/controller_loader.ts`（`wrapClass` 函数）

`wrapClass(Controller)` 遍历类原型链，收集所有方法并映射为中间件函数。

**执行流程**：

1. 从 `Controller.prototype` 开始，沿原型链向上遍历（直到 `Object.prototype`）
2. 对每一层，通过 `Object.getOwnPropertyNames(proto)` 获取所有自有属性名
3. 过滤规则：
   - 跳过 `constructor`
   - 跳过 getter/setter（通过 `Object.getOwnPropertyDescriptor` 检测）
   - 跳过非函数属性（`isFunction(d?.value)` 检测）
   - 子类已定义的方法不被父类同名方法覆盖（`hasOwnProperty` 检查：`ret` 中已有则跳过）
4. 每个有效方法名映射为 `methodToMiddleware(Controller, key)` 的返回值
5. 在中间件函数上标记 `[FULLPATH]`：`{proto.fullPath}#{Controller.name}.{key}()`

**返回结构**：

```javascript
// 输入：class UserController extends BaseController { add() {} delete() {} list() {} }
// 输出：
{
  add: classControllerMiddleware,     // 每次调用创建新实例
  delete: classControllerMiddleware,  // 每次调用创建新实例
  list: classControllerMiddleware,    // 每次调用创建新实例
  // BaseController 的方法如果未被 UserController 覆盖，也会被收集
}
```

### 4. methodToMiddleware() 中间件函数

**源文件**：`packages/ee-core/src/controller/controller_loader.ts`（`methodToMiddleware` 函数）

```typescript
function methodToMiddleware(Controller, key) {
  return function classControllerMiddleware(...args) {
    const controller = new Controller();  // 每次调用创建新实例
    const fn = controller[key];
    if (!fn) return undefined;
    return callFn(fn, args, controller);  // 以 controller 为上下文调用方法
  };
}
```

**核心设计：每次 IPC 调用创建新的控制器实例**。

这是并发安全的根本保证。当多个 IPC 请求同时到达时，每个请求拥有独立的控制器实例，避免了共享状态导致的竞争条件。代价是轻微的实例创建开销，但对于桌面应用的 IPC 请求频率来说完全可以忽略。

`callFn(fn, args, controller)` 来自 `core/utils/index.ts`，以 `controller` 作为 `this` 上下文调用方法，确保方法内部 `this` 指向正确的实例。

### 5. IPC 通道命名机制

控制器文件路径到 IPC 通道的转换由 FileLoader 和 `utils/wrap.ts` 共同完成。

**转换链路**：

```
文件路径: controller/example.js
  → FileLoader 扫描: 相对路径 'example.js'
  → getProperties('example.js', { caseStyle: 'lower' }): ['example']
  → 目录名前缀: 'controller'
  → pathName: 'controller.example'
  → 属性映射: target.controller.example = { test: fn, ... }
  → IPC 通道: 'controller/example/test'（使用 mainServer.channelSeparator '/' 分隔）
```

**多层目录示例**：

```
文件路径: controller/admin/login.js
  → 相对路径: 'admin/login.js'
  → getProperties: ['admin', 'login']
  → pathName: 'controller.admin.login'
  → 属性映射: target.controller.admin.login = { auth: fn }
  → IPC 通道: 'controller/admin/login/auth'
```

**`caseStyle: 'lower'` 的作用**（源文件 `packages/ee-core/src/utils/wrap.ts`）：

`defaultCamelize(filepath, 'lower')` 将路径段的第一个字母转为小写。例如 `UserInfo.js` → `userInfo`，`User-Info.js` → `userInfo`。这与控制器注册插件的 `caseStyle: 'lower'` 保持一致，确保打包模式和开发模式产出相同的属性路径。

### 6. 控制器导出结构示例

```
controller/
  example.js          → { controller: { example: { test: fn, upload: fn } } }
  user.js             → { controller: { user: { add: fn, delete: fn } } }
  admin/login.js      → { controller: { admin: { login: { auth: fn } } } }
```

其中 `fn` 是 `methodToMiddleware` 返回的中间件函数，每次调用都会创建新的控制器实例。

## 关键设计决策

1. **每次调用创建新实例（并发安全）**：这是控制器系统最重要的设计决策。`methodToMiddleware` 不缓存控制器实例，每次 IPC 请求都 `new Controller()`。这确保了多个并发请求之间没有状态共享，从根本上消除了竞争条件。虽然增加了实例创建开销，但桌面应用 IPC 请求频率远低于 Web 服务器的 HTTP 请求频率，开销完全可接受。

2. **原型链遍历收集方法（继承支持）**：`wrapClass` 遍历完整原型链，允许控制器类继承基类方法。但子类同名方法不会被父类覆盖（`hasOwnProperty` 检查），保证了方法覆盖的正确性。getter/setter 和非函数属性被跳过，只收集真正的方法。

3. **类与函数导出的不同处理**：类导出通过 `wrapClass` + `methodToMiddleware` 包装为并发安全的中间件；函数导出不做包装，直接注册。这意味着函数导出没有并发安全保护，适用于无状态的辅助函数。

4. **pathName/fullPath 元数据**：在类原型和中间件函数上设置 `pathName`/`fullPath`，为 IPC 路由、调试日志和错误追踪提供路径信息。IPC 通道名由 pathName 转换而来（使用 `channelSeparator` 替换 `.`）。

5. **打包模式与开发模式一致性**：两种模式使用相同的 `initializer` 和 `wrapClass` 逻辑，只是文件扫描来源不同。`caseStyle: 'lower'` 在两种模式下都一致使用，确保打包产出和开发运行产出相同的控制器映射结构。

6. **loading 标志防止并发竞争**：同步和异步加载不能同时进行。`loading` 标志在异步加载开始时设置，结束时清除。同步方法在标志为 true 时抛出明确错误，避免不确定的行为。

7. **字节码类支持**：`isBytecodeClass()` 通过 `String(exports).indexOf('[class')` 检测 bytenode 编译的 `.jsc` 文件中的类。字节码类与普通类享有相同的 `wrapClass` 处理，确保加密控制器也能正常工作。

## API 导出

### controller/index.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `loadController()` | 函数 | 同步加载控制器，返回方法映射 |
| `loadControllerAsync()` | 异步函数 | 异步加载控制器，返回方法映射 |
| `getControllers()` | 函数 | 惰性获取控制器映射 |
| `getController()` | 函数 | `getControllers()` 别名 |

### controller/controller_loader.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `ControllerLoader` | 类 | 控制器加载器 |
| `ControllerLoader.load()` | 方法 | 同步加载，返回方法映射 |
| `ControllerLoader.loadAsync()` | 异步方法 | 异步加载，返回方法映射 |
| `ControllerLoader.timing` | 属性 | `Timing` 实例 |

注意：`wrapClass()` 和 `methodToMiddleware()` 是模块内部函数，不对外导出。

## 模块关系

```
controller/index.ts
  ├── 依赖 → controller/controller_loader.ts (ControllerLoader)
  └── 被依赖 → app/application.ts (loadController / loadControllerAsync)

controller/controller_loader.ts
  ├── 依赖 → utils/type_check (isClass, isFunction)
  ├── 依赖 → ps (getElectronDir)
  ├── 依赖 → core/utils/timing (Timing)
  ├── 依赖 → core/loader/file_loader (FileLoader, FULLPATH)
  ├── 依赖 → core/utils (isBytecodeClass, callFn)
  ├── 依赖 → types (RegistryEntry)
  └── 读取 → globalThis.__EE_CONTROLLER_REGISTRY__ (打包模式注册表)

wrapClass 函数
  ├── 依赖 → utils/type_check (isFunction)
  ├── 依赖 → core/loader/file_loader (FULLPATH symbol)

methodToMiddleware 函数
  ├── 依赖 → core/utils (callFn)

IPC 通道路由
  ├── 控制器方法映射 → cross/ipc (IPC 通信)
  ├── pathName → mainServer.channelSeparator → IPC 通道名
```

控制器加载系统是框架启动的核心环节之一（`Application.run()` 的第一步），加载产出的方法映射直接服务于 IPC 通信系统。控制器与 `cross/ipc` 的关系是：控制器提供方法映射，IPC 系统根据通道名从映射中查找对应方法并调用。
