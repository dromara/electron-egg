# 07. SqliteStorage（better-sqlite3 封装）

## 概述

SqliteStorage 是 ee-core 框架的本地数据库存储模块，基于 better-sqlite3 封装，提供同步的 SQLite 数据库操作能力。模块的核心设计是自动推断存储模式——根据数据库名称的不同格式（纯文件名、相对路径、绝对路径、内存模式），自动决定文件存放位置和目录创建策略，同时内置路径遍历安全防护。

better-sqlite3 选择同步 API 设计（而非 async），原因：
- Electron 主进程不需要担心数据库操作阻塞 UI（UI 在渲染进程）
- 同步 API 更简单，避免了异步回调/Promise 的复杂性
- better-sqlite3 的同步操作经过高度优化，性能接近异步方案

源文件：`packages/ee-core/src/storage/sqliteStorage.ts`

## 核心流程

### SqliteStorage 构造流程

源文件：`packages/ee-core/src/storage/sqliteStorage.ts`

```
new SqliteStorage(name, opt = {})
  │
  ├─ 1. 安全验证
  │   ├─ assert(name) → 名称非空检查
  │   │   │  └─ [空名称] → throw: "db name Cannot be empty"
  │   ├─ assert(!name.includes('..')) → 路径遍历防护
  │   │   │  └─ [含 '..'] → throw: "db name contains path traversal"
  │   │   │     └─ 防止恶意输入如 '../../etc/passwd.db' 访问意外目录
  │
  ├─ 2. 推断存储模式
  │   this.mode = getMode(name)
  │   │
  │   ├─ ':memory:' → mode = 'memory'
  │   │   │  └─ 内存数据库，不持久化到磁盘
  │   │
  │   ├─ 包含路径分隔符且为绝对路径 → mode = 'absolute'
  │   │   │  └─ 如 '/tmp/data/app.db'、'C:\data\app.db'
  │   │
  │   ├─ 包含路径分隔符但非绝对路径 → mode = 'relative'
  │   │   │  └─ 如 'sub/app.db'、'data/test.db'
  │   │
  │   └─ 不包含路径分隔符 → mode = 'onlyName'
  │       └─ 如 'app.db'、'mydata.db'
  │
  ├─ 3. 创建数据库目录
  │   this.dbDir = _createDatabaseDir()
  │   │
  │   ├─ [absolute 模式] → dbDir = path.dirname(name)
  │   │   │  └─ 使用名称中的目录部分，如 '/tmp/data' 从 '/tmp/data/app.db'
  │   │
  │   ├─ [其他模式] → dbDir = path.join(getDataDir(), 'db')
  │   │   │  └─ 默认存放在 {dataDir}/db/ 目录下
  │   │
  │   └─ [目录不存在] → mkdir(dbDir, { mode: 0o755 })
  │       │  └─ 自动创建，权限 0o755（owner 读写执行，其他读执行）
  │
  ├─ 4. 提取文件名
  │   this.fileName = _formatFileName(name)
  │   │
  │   ├─ [memory 模式] → fileName = ':memory:'（保持原名）
  │   └─ [文件模式] → fileName = path.basename(name)
  │       │  └─ 如 '/tmp/data/app.db' → 'app.db'
  │       │  └─ 如 'sub/app.db' → 'app.db'
  │       │  └─ 如 'app.db' → 'app.db'
  │
  └─ 5. 打开数据库连接
      this.db = _initDB(opt)
      │
      ├─ 合并默认选项：options = Object.assign({ timeout: 5000 }, opt)
      │   │  └─ 默认超时 5000ms（better-sqlite3 的 busy timeout）
      │
      ├─ [memory 模式] → dbPath = ':memory:'
      │   └─ new Database(':memory:', options)
      │   │  └─ better-sqlite3 自动在内存中创建数据库
      │
      ├─ [文件模式] → dbPath = getFilePath()（dbDir + fileName）
      │   ├─ new Database(dbPath, options)
      │   │  └─ better-sqlite3 自动创建数据库文件（如不存在）
      │   └─ assert(fs.existsSync(dbPath))
      │       │  └─ 验证文件创建成功
      │       └─ [文件不存在] → throw: "error: db {dbPath} not exists"
      │           └─ 可能原因：目录权限不足、磁盘空间不足等
```

### 四种存储模式详解

| 模式 | 名称格式 | dbDir | fileName | 数据库路径示例 | 适用场景 |
|------|---------|-------|----------|--------------|---------|
| memory | `:memory:` | 不使用 | `:memory:` | 内存中 | 临时数据、缓存、测试 |
| onlyName | `app.db` | `{dataDir}/db` | `app.db` | `{dataDir}/db/app.db` | 最常见，单文件数据库 |
| relative | `sub/app.db` | `{dataDir}/db` | `app.db` | `{dataDir}/db/sub/app.db` | 分类存储，多数据库场景 |
| absolute | `/tmp/app.db` | `/tmp` | `app.db` | `/tmp/app.db` | 指定特定位置 |

**注意**：relative 模式下 `dbDir` 仍然是 `{dataDir}/db`（不是 `{dataDir}/db/sub`），但 `getFilePath()` 返回 `path.join(dbDir, fileName)` = `{dataDir}/db/app.db`。这意味着 relative 模式目前将文件名扁平化到 `db/` 目录下，子目录信息在 `fileName` 中被 `path.basename()` 丢失。如果业务需要真正的子目录存储，应使用 absolute 模式指定完整路径。

### getDataDir() 数据目录

`getDataDir()` 来自 `packages/ee-core/src/ps/index.ts`，返回应用的数据存储根目录。不同环境下此路径不同：
- 开发环境：项目根目录下的 `data/` 目录
- 生产打包：Electron 的 `userData` 目录（macOS: `~/Library/Application Support/{appName}`）

所有非 absolute 模式的数据库文件默认存放在 `{dataDir}/db/` 下，确保数据目录与应用绑定。

### getFilePath() 完整路径计算

```ts
getFilePath(): string
  return path.join(this.dbDir, this.fileName)
```

简单拼接 dbDir 和 fileName。对于文件模式，这是数据库文件的绝对路径；对于 memory 模式，此路径无实际意义。

## 关键设计决策

### 1. 路径遍历安全防护

构造函数中使用 `assert(!name.includes('..'))` 阻止路径遍历攻击。这是关键安全措施——如果允许 `..`，攻击者可通过名称如 `../../etc/shadow.db` 将数据库文件创建在意外位置，甚至覆盖系统文件。assert 在生产环境下默认不抛出（Node.js 可配置），但 ee-core 选择保留此检查以确保安全。

### 2. 同步 API 的选择

better-sqlite3 全部 API 都是同步的（`new Database()`、`.prepare()`、`.run()` 等），这与 node-sqlite3（异步 callback/Promise）截然不同。在 Electron 主进程环境中，同步操作不会阻塞 UI，且简化了代码逻辑。SqliteStorage 不需要额外适配异步模式。

### 3. 自动创建目录

`_createDatabaseDir()` 在发现目录不存在时自动创建（权限 0o755）。这是便利性设计——业务代码不需要手动确保目录存在。但这也意味着首次创建可能因权限问题失败，`_initDB()` 的 `assert(fs.existsSync(dbPath))` 检查捕获了此类问题。

### 4. 默认 timeout: 5000ms

better-sqlite3 的 `timeout` 选项对应 SQLite 的 `busy_timeout`——当数据库被其他连接锁定时，等待最多 5000ms 再报 SQLITE_BUSY 错误。5 秒的默认值适合大多数场景（Electron 主进程通常只有一个数据库连接，但在多窗口或多进程场景可能并发访问）。

### 5. 文件创建后验证（existsSync 检查）

`_initDB()` 在文件模式下创建数据库后，立即用 `fs.existsSync(dbPath)` 验证文件是否存在。这是防护性设计——正常情况下 better-sqlite3 会自动创建文件，但如果目录权限不足或路径异常，文件可能不会被创建。assert 在此提供明确的错误信息而非让业务代码遇到不可解释的数据库错误。

### 6. onlyName 模式的简洁性

最常见的使用方式是只传文件名（如 `'app.db'`），文件自动存放在 `{dataDir}/db/` 下。这是最简单的模式，覆盖了 80% 的使用场景。relative 和 absolute 模式为特殊需求提供灵活性。

### 7. memory 模式的零磁盘依赖

`:memory:` 模式完全在内存中运行，不涉及任何文件操作。`_createDatabaseDir()` 和 `_formatFileName()` 对 memory 模式有特殊处理，避免了无意义的目录创建和文件名提取。

## API 导出

### storage/sqliteStorage.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `SqliteStorage` | Class | SQLite 存储类 |

**SqliteStorage 类成员**：

| 属性/方法 | 类型 | 说明 |
|-----------|------|------|
| `name` | `string` | 数据库名称（原始输入值） |
| `mode` | `string` | 存储模式：memory / onlyName / relative / absolute |
| `dbDir` | `string` | 数据库文件所在目录的绝对路径 |
| `fileName` | `string` | 数据库文件名（仅文件名部分） |
| `db` | `Database.Database` | better-sqlite3 数据库实例 |
| `constructor(name, opt)` | `(string, Record<string, unknown>)` | 创建实例，自动推断模式和初始化 |
| `getMode(name)` | `function → string` | 从名称推断存储模式 |
| `getDbDir()` | `function → string` | 获取数据库目录路径 |
| `getFilePath()` | `function → string` | 获取数据库文件完整路径 |
| `_initDB(opt)` | `function → Database.Database` | 打开数据库连接（内部方法） |
| `_createDatabaseDir()` | `function → string` | 创建数据库目录（内部方法） |
| `_formatFileName(name)` | `function → string` | 提取文件名（内部方法） |

**使用示例**：

```ts
// 最常用：纯文件名模式
const db = new SqliteStorage('myapp.db');
// 数据库路径：{dataDir}/db/myapp.db

// 内存数据库
const memDb = new SqliteStorage(':memory:');
// 不持久化，进程退出后数据丢失

// 绝对路径模式
const absDb = new SqliteStorage('/tmp/test.db');
// 数据库路径：/tmp/test.db

// 使用 better-sqlite3 API 操作数据库
const stmt = db.db.prepare('CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY, name TEXT)');
stmt.run();

const insert = db.db.prepare('INSERT INTO users (name) VALUES (?)');
insert.run('Alice');

const select = db.db.prepare('SELECT * FROM users');
const rows = select.all();
```

## 模块关系

```
storage/sqliteStorage.ts
  ├─ 引用 better-sqlite3 → Database（外部依赖，需 re-sqlite 为 Electron 重编译）
  ├─ 引用 utils/helper → mkdir（目录创建工具）
  └─ 引用 ps → getDataDir（获取应用数据目录）

上游调用者：
  - 业务代码 → new SqliteStorage(name, opt) 创建数据库实例

下游依赖：
  - ps 模块：提供 getDataDir()（数据根目录）
  - utils/helper 模块：提供 mkdir()（带权限的目录创建）
  - better-sqlite3（外部 npm 包）：核心 SQLite 操作能力
    │  └─ 需要通过 pnpm re-sqlite 为 Electron 重新编译原生模块
    │  └─ 原因：better-sqlite3 使用 C++ binding，Electron 的 Node.js 版本可能与系统不同

相关模块：
  - electron/app → before-quit 时如果数据库需要关闭，业务代码自行处理
  - config 模块 → 如需配置数据库路径，可从 config 中读取
