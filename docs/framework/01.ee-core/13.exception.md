# 异常处理

## 概述

ee-core 的异常处理模块（exception）注册全局 `uncaughtException` 和 `unhandledRejection` 监听器，捕获未被 try/catch 或 .catch() 处理的异常和 Promise 拒绝。模块在框架启动流程中最先被 `boot.ts` 调用（`loadException()`），确保后续所有流程中的异常都能被捕获、记录并按配置决定是否退出进程。

核心设计原则：
- **优先记录**：异常首先通过 coreLogger 记录到框架核心日志，确保异常信息不丢失
- **安全降级**：日志系统不可用时（如子进程中 pino transport 不可用），回退到 console.error
- **进程分类退出**：主进程、子进程、渲染进程分别有独立的退出配置
- **开发辅助**：开发环境下 fork 子进程的异常额外发送到终端，便于调试

## 核心流程

### 启动流程

异常处理在框架启动流程中最先初始化，确保后续所有模块的异常都能被捕获：

```
Boot → loadException → loadConfig → loadLog → ...
         ↑ 最先加载，确保后续异常可捕获
```

1. `boot.ts` 的 `init()` 调用 `loadException()`
2. `loadException()` 注册 `uncaughtException` 和 `unhandledRejection` 两个全局监听器

相关源文件：`packages/ee-core/src/exception/index.ts` 中 `loadException()` 函数

### uncaughtException 处理流程

```
process.on('uncaughtException', err)
  │
  ├─ 1. 错误标准化
  │     err instanceof Error → 直接使用
  │     非 Error → new Error(String(err))
  │     error.name === 'Error' → 重命名为 'unhandledExceptionError'
  │     （便于在日志中区分未被捕获的异常和普通 Error）
  │
  ├─ 2. 日志记录
  │     尝试 coreLogger.error(error)
  │     失败 → console.error('[ee-core] uncaughtException:', error)
  │     （pino transport 在子进程中可能不可用）
  │
  ├─ 3. 开发环境错误发送
  │     isForkedChild() && isDev() → childMessage.sendErrorToTerminal(error)
  │     （子进程异常信息发送到主进程终端，便于调试）
  │
  └─ 4. 退出决策
      _exit() → 根据 exception 配置决定是否退出
```

相关源文件：`packages/ee-core/src/exception/index.ts` 中 `uncaughtExceptionHandler()` 和 `_exit()` 函数

### unhandledRejection 处理流程

```
process.on('unhandledRejection', err)
  │
  ├─ 1. 错误标准化
  │     err instanceof Error → 直接使用
  │     非 Error：
  │       new Error(String(err))
  │       err 是对象 → 提取 name, message, stack 属性
  │       （支持非 Error 类型的 Promise 拒绝值）
  │     error.name === 'Error' → 重命名为 'unhandledRejectionError'
  │
  ├─ 2. 日志记录
  │     尝试 coreLogger.error(error)
  │     失败 → console.error('[ee-core] unhandledRejection:', error)
  │
  ├─ 3. 开发环境错误发送
  │     isForkedChild() && isDev() → childMessage.sendErrorToTerminal(error)
  │
  └─ 4. 退出决策
      _exit() → 根据 exception 配置决定是否退出
```

相关源文件：`packages/ee-core/src/exception/index.ts` 中 `unhandledRejectionHandler()` 函数

### 退出决策流程

```
_exit()
  │
  ├─ 获取 exception 配置
  │     getConfig().exception → { mainExit, childExit, rendererExit }
  │
  ├─ 进程类型匹配
  │     isMain() && mainExit === true → _delayExit()
  │     isForkedChild() && childExit === true → _delayExit()
  │     isRenderer() && rendererExit === true → _delayExit()
  │     其他 → 不退出（异常被记录但进程继续运行）
  │
  └─ _delayExit()
     setTimeout(() => process.exit(), 1500)
     延迟 1500ms 等待日志写入等异步操作完成
```

相关源文件：`packages/ee-core/src/exception/index.ts` 中 `_exit()` 和 `_delayExit()` 函数

### uncaughtExceptionMonitor 流程

```
process.on('uncaughtExceptionMonitor', err)
  │
  └─ coreLogger.error('uncaughtExceptionMonitor:', error)
     仅用于监控和上报，不影响后续异常处理流程
     （此事件在 uncaughtException 触发之前被触发）
```

相关源文件：`packages/ee-core/src/exception/index.ts` 中 `uncaughtExceptionMonitorHandler()` 函数

## 关键设计决策

### 1. loadException 最先加载

异常处理在启动流程中排在第一位（`loadException → loadConfig → loadLog`），确保后续所有模块初始化和运行时的异常都能被捕获。如果异常处理在 config 或 log 之后才加载，那么 config 加载失败时的异常就无法被正确处理。

### 2. coreLogger 优先 + console 降级

异常记录首先尝试 `coreLogger.error()`（写入框架核心日志文件），如果 coreLogger 失败则回退到 `console.error()`。这种降级设计是因为：
- 在 fork 子进程中，pino 的 transport worker 可能尚未初始化或不可用
- 在异常处理阶段，日志系统本身可能处于异常状态
- console.error 确保异常信息永远不会被静默吞掉

### 3. 非 Error 类型的 Promise 拒接处理

`unhandledRejection` 处理器不仅处理 Error 类型拒绝值，还处理非 Error 类型（如字符串、数字、普通对象）。对于对象类型的拒绝值，会尝试提取 `name`、`message`、`stack` 属性到标准化 Error 对象中，确保日志中保留尽可能多的错误信息。

### 4. 错误名称重命名

当 Error 的 `name` 属性为默认值 `'Error'` 时，处理器将其重命名为 `'unhandledExceptionError'` 或 `'unhandledRejectionError'`。这使得在日志搜索和过滤时能快速区分未被捕获的异常与被正常记录的 error 级别日志。

### 5. 延迟退出（1500ms）

`_delayExit()` 使用 1500ms 延迟而非立即退出，等待以下操作完成：
- pino 异步日志写入（pino 通过 worker thread 写入，需要时间）
- childMessage 错误信息发送（IPC 通信需要时间）
- 其他可能的异步清理操作

立即退出可能导致最后的日志条目丢失，延迟确保了日志完整性。

### 6. 进程分类退出配置

异常退出行为按进程类型分别配置：
- `mainExit`：主进程异常后是否退出（生产环境通常 true，避免半死状态）
- `childExit`：子进程异常后是否退出（通常 true，子进程无法恢复）
- `rendererExit`：渲染进程异常后是否退出（通常 false，避免因单个窗口崩溃关闭整个应用）

这种分类配置避免了"一刀切"——渲染进程的异常不应导致整个应用退出。

### 7. 开发环境子进程错误终端发送

开发环境中 fork 子进程的异常通过 `childMessage.sendErrorToTerminal()` 发送到主进程终端。这解决了子进程异常在终端中不易观察的问题——子进程的 stdout/stderr 可能被配置为 `ignore`，异常信息只写入日志文件，开发者不易发现。

## API 导出

### `packages/ee-core/src/exception/index.ts`

| 导出 | 类型 | 说明 |
|------|------|------|
| `loadException()` | function | 注册全局异常处理器，由 boot.ts 调用 |
| `uncaughtExceptionMonitorHandler()` | function | 注册异常监控处理器（仅记录，不影响处理流程） |

**内部函数（不导出）**：

| 函数 | 说明 |
|------|------|
| `uncaughtExceptionHandler()` | 注册 uncaughtException 监听器 |
| `unhandledRejectionHandler()` | 注册 unhandledRejection 监听器 |
| `_devError(err)` | 开发环境子进程错误终端发送 |
| `_exit()` | 根据配置决定进程退出 |
| `_delayExit()` | 延迟 1500ms 后退出进程 |

## 模块关系

```
exception/index.ts
  ├─ imports → log/index.ts (coreLogger 异常日志记录)
  ├─ imports → ps/index.ts (isForkedChild, isRenderer, isDev, isMain 进程判断)
  ├─ imports → config/index.ts (getConfig 获取异常退出配置)
  ├─ imports → message/index.ts (childMessage.sendErrorToTerminal 开发环境终端发送)
  │
  ├─ 被导入 → app/boot.ts (loadException 最先调用)
  └─ 袍导入 → app/application.ts (可选：uncaughtExceptionMonitorHandler 监控)

  依赖链：
  exception → log → config → ps → boot (环境设置)
  exception → message → ipc → electron (子进程通信)

  启动顺序：
  boot.ts init() → loadException() → loadConfig() → loadLog() → ...
                     ↑ 最先加载，确保后续异常可捕获
```

### 配置项

异常退出配置位于 `config.default.js` 的 `exception` 节：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `mainExit` | boolean | true | 主进程异常后是否退出 |
| `childExit` | boolean | true | 子进程异常后是否退出 |
| `rendererExit` | boolean | false | 渲染进程异常后是否退出 |

**推荐配置策略**：

```javascript
// 开发环境：不退出便于调试
exception: {
  mainExit: false,
  childExit: false,
  rendererExit: false,
}

// 生产环境：主进程和子进程退出，渲染进程不退出
exception: {
  mainExit: true,
  childExit: true,
  rendererExit: false,
}
```
