# 启动流程与生命周期

## 概述

ee-core 框架的启动流程由 `ElectronEgg` 类发起，经过严格有序的阶段初始化，最终通过 `Application` 类完成业务控制器加载、通信服务启动、Electron 主进程功能初始化。整个生命周期由 `EventBus` 管理，提供五个框架核心里程碑事件和自定义事件机制。

启动阶段分为两大环节：

- **基础初始化**（`init()`）：异常处理 → 配置加载 → 目录创建 → 日志初始化
- **业务启动**（`run()`/`runAsync()`）：控制器加载 → 通信服务 → Ready 事件 → Electron 功能

## 核心流程

### 1. ElectronEgg 构造与初始化

**源文件**：`packages/ee-core/src/app/boot.ts`

`ElectronEgg` 是框架入口类，在 `electron/main.js` 中被实例化和启动。

**构造函数执行流程**：

1. 通过 `electronApp.getAppPath()` 获取 `baseDir`（Electron app 根路径）
2. 解析命令行参数：`--env`（环境名，默认 `prod`）和 `--debugger`（调试源码标志）
3. 根据 `debugger` 参数决定 `electronDir`：
   - 调试模式 → 源码目录 `getElectronCodeDir(baseDir)`，即 `{baseDir}/electron`
   - 生产模式 → 打包输出目录 `getBundleDir(baseDir)`，即 `{baseDir}/public/electron`
4. 收集应用信息构建 `options`：`appName`、`userHome`、`appData`、`appUserData`、`appVersion`、`isPackaged`、`execDir`
5. 生产 + 打包模式下，`execDir` 取可执行文件所在目录（`path.dirname(electronApp.getPath('exe'))`）
6. 将关键信息写入 `process.env`（供配置加载、日志等模块读取）：

   | 环境变量 | 值来源 | 说明 |
   |---------|--------|------|
   | `EE_ENV` | `--env` 参数 | 运行环境 |
   | `EE_APP_NAME` | `electronApp.getName()` | 应用名称 |
   | `EE_APP_VERSION` | `electronApp.getVersion()` | 应用版本 |
   | `EE_BASE_DIR` | `electronApp.getAppPath()` | 项目根目录 |
   | `EE_ELECTRON_DIR` | 调试标志决定 | 业务代码目录 |
   | `EE_USER_HOME` | `electronApp.getPath('home')` | 用户主目录 |
   | `EE_APP_DATA` | `electronApp.getPath('appData')` | 应用数据目录 |
   | `EE_APP_USER_DATA` | `electronApp.getPath('userData')` | 用户数据目录 |
   | `EE_EXEC_DIR` | baseDir 或 exe 路径 | 可执行目录 |
   | `EE_IS_PACKAGED` | `isPackaged` | 是否打包 |
   | `EE_SOCKET_PORT` | 空（后续动态写入） | Socket 端口 |
   | `EE_HTTP_PORT` | 空（后续动态写入） | HTTP 端口 |

7. 调用 `init()` 执行基础初始化

**`init()` 顺序（不可变更）**：

```
loadException() → loadConfig() → loadDir() → loadLog()
```

每一步都有严格的依赖关系：异常处理必须最先注册以捕获后续错误；配置是其他模块的基础；目录必须存在才能写日志；日志依赖配置和目录。

### 2. Application 启动编排

**源文件**：`packages/ee-core/src/app/application.ts`

`Application` 类是框架启动流程的编排者，以单例 `app` 导出。

**`run()` 方法流程**：

```
loadController() → loadSocket() → emitLifecycle(Ready) → loadElectron()
```

- `loadController()`：同步加载控制器（使用 `require()`），适用于 CJS 项目
- `loadSocket()`：启动 IPC/HTTP/SocketIO 通信服务
- `emitLifecycle(Ready)`：触发 Ready 生命周期事件
- `loadElectron()`：初始化 Electron 主进程（创建窗口等）

**`runAsync()` 方法流程**：

```
loadControllerAsync() → loadSocket() → emitLifecycle(Ready) → loadElectron()
```

唯一区别是控制器使用 `import()` 异步加载，支持 ESM 模块。

**`register()` 方法**：委托给 `eventBus.register()` 注册生命周期事件钩子。

### 3. EventBus 事件管理

**源文件**：`packages/ee-core/src/app/events.ts`

`EventBus` 管理两类事件，存储在两个独立映射中：

- **生命周期事件** (`lifecycleEvents`)：框架核心里程碑
- **自定义事件** (`eventsMap`)：业务自由使用

**五个生命周期常量**：

| 常量 | 值 | 触发时机 |
|------|----|---------|
| `Ready` | `'ready'` | 控制器和通信服务加载完毕后 |
| `ElectronAppReady` | `'electron-app-ready'` | Electron `app.whenReady()` 完成后 |
| `WindowReady` | `'window-ready'` | 主窗口创建后 |
| `BeforeClose` | `'before-close'` | 窗口关闭前（可用于清理） |
| `Preload` | `'preload'` | 预加载脚本注入时 |

**注册行为**：

- `register()` 注册生命周期事件，`on()` 注册自定义事件
- 同名事件重复注册会覆盖旧处理器，并输出 `warn` 级别日志（而非静默忽略）

**触发行为的关键设计差异**：

- **`emitLifecycle()`**：同步错误直接向上传播（不吞掉）；异步错误重新抛出为 unhandled rejection（由全局异常处理器捕获）。这是有意设计——框架核心里程碑的失败不应被静默处理。
- **`emit()`**：同步异常和异步 rejection 都被 catch 并记录到日志，不传播。自定义事件属于业务逻辑，单个事件失败不应影响其他逻辑。

`eventBus` 以单例导出，全局共享。

### 4. 目录初始化

**源文件**：`packages/ee-core/src/app/dir.ts`

`loadDir()` → `initDir()` 创建三个必要目录：

1. **用户主目录隐藏应用目录** (`getUserHomeHiddenAppDir()`)：`~/.{appName}/`，存储应用级配置和缓存
2. **数据目录** (`getDataDir()`)：存储业务数据（如 SQLite 数据库文件）
3. **日志目录** (`getLogDir()`)：存储运行日志文件

**OpenHarmony 特殊处理**：由于权限问题，OpenHarmony 环境下使用 `getAppCustomDataDir()` 替代用户主目录隐藏目录（路径格式 `{appUserData}/{appName}/`）。

所有目录权限设为 `0o755`（所有者读写执行，其他人读执行）。异常被捕获并输出到 `console.error`（因为此时日志系统可能尚未初始化完成）。

**路径策略**（源文件 `packages/ee-core/src/ps/index.ts`）：

| 目录 | 开发环境 | 生产环境 |
|------|---------|---------|
| 数据目录 | `{baseDir}/data` | `{userHome}/.{appName}/data` |
| 日志目录 | `{baseDir}/logs` | `{userHome}/.{appName}/logs` |
| 隐藏应用目录 | `~/.{appName}/` | `~/.{appName}/` |

### 5. 全局异常处理

**源文件**：`packages/ee-core/src/exception/index.ts`

`loadException()` 注册两个全局监听器：

- **`uncaughtException`**：未捕获异常处理。记录错误日志，开发环境的子进程会通过 `childMessage.sendErrorToTerminal()` 将错误发送到终端。根据 `exception` 配置决定是否退出进程。
- **`unhandledRejection`**：未处理的 Promise 拒绝。对非 Error 类型的 rejection 提取 `name/message/stack` 信息。行为与 `uncaughtException` 类似。

**退出策略**（`exception` 配置项）：

| 配置项 | 默认值 | 说明 |
|-------|--------|------|
| `mainExit` | `false` | 主进程异常是否退出 |
| `childExit` | `false` | 子进程异常是否退出 |
| `rendererExit` | `true` | 渲染进程异常是否退出 |

退出前延迟 1500ms（`_delayExit()`），确保异步日志写入等操作完成。

## 完整启动顺序图

```
Boot (ElectronEgg constructor)
  ├── 收集环境信息、写入 process.env
  └── init()
      ├── loadException()    — 注册全局异常处理（最先执行，确保后续异常可被捕获）
      ├── loadConfig()       — 加载配置（后续模块依赖配置）
      ├── loadDir()          — 创建运行目录（日志依赖目录）
      └── loadLog()          — 初始化日志系统（依赖配置和目录）

用户调用 run() 或 runAsync()
  ├── loadController() / loadControllerAsync()  — 加载业务控制器
  ├── loadSocket()       — 启动通信服务（IPC/HTTP/SocketIO）
  ├── emitLifecycle(Ready) — 触发 Ready 事件
  └── loadElectron()     — 初始化 Electron 主进程
      ├── app.whenReady()
      ├── emitLifecycle(ElectronAppReady)
      ├── 创建主窗口
      ├── emitLifecycle(WindowReady)
      └── 注册 before-close / preload 等事件
```

## 关键设计决策

1. **初始化顺序不可变更**：`loadException` 必须最先执行（后续任何阶段出错都能被捕获）；配置是其他模块的基础数据源；目录必须在日志写入前存在。这个顺序由依赖关系决定，不是随意排列。

2. **electronDir 双路径策略**：调试模式下指向源码目录（支持热修改），生产模式下指向打包输出目录。这使得开发者可以在不重新打包的情况下调试业务代码。

3. **环境变量桥接**：boot.ts 将所有关键路径信息写入 `process.env`，而非在模块间传递参数。这使得任何模块（包括子进程）都可以通过 `process.env` 获取运行环境信息，无需依赖注入链。

4. **生命周期事件 vs 自定义事件的错误处理策略不同**：生命周期事件是框架核心里程碑，失败不应被静默吞掉；自定义事件属于业务逻辑，容错性更高。这体现了"框架级错误必须可见，业务级错误可以容错"的设计理念。

5. **Application 单例模式**：`app` 作为单例导出，确保整个框架只有一个启动流程编排器。`register()` 方法委托给 `eventBus`，保持职责分离。

6. **目录初始化容错**：`initDir()` 自行捕获异常并输出到 console（而非框架日志），因为此时日志系统可能尚未就绪。这是对初始化顺序依赖的现实妥协。

## API 导出

### boot.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `ElectronEgg` | 类 | 框架入口类 |
| `ElectronEgg.options` | 属性 | `ElectronEggOptions` 类型，包含 env、baseDir、electronDir 等运行信息 |
| `ElectronEgg.init()` | 方法 | 基础初始化（异常→配置→目录→日志） |
| `ElectronEgg.register(eventName, handler)` | 方法 | 注册生命周期事件 |
| `ElectronEgg.run()` | 方法 | 同步启动应用 |
| `ElectronEgg.runAsync()` | 方法 | 异步启动应用 |

### application.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `Application` | 类 | 启动流程编排器 |
| `app` | 单例 | Application 全局单例 |
| `app.register(eventName, handler)` | 方法 | 委托 eventBus.register() |
| `app.run()` | 方法 | 同步启动流程 |
| `app.runAsync()` | 方法 | 异步启动流程 |

### events.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `EventBus` | 类 | 事件总线 |
| `eventBus` | 单例 | EventBus 全局单例 |
| `Ready` | 常量 | `'ready'` |
| `ElectronAppReady` | 常量 | `'electron-app-ready'` |
| `WindowReady` | 常量 | `'window-ready'` |
| `BeforeClose` | 常量 | `'before-close'` |
| `Preload` | 常量 | `'preload'` |
| `eventBus.register(eventName, handler)` | 方法 | 注册生命周期事件 |
| `eventBus.emitLifecycle(eventName, ...args)` | 方法 | 触发生命周期事件 |
| `eventBus.on(eventName, handler)` | 方法 | 注册自定义事件 |
| `eventBus.emit(eventName, ...args)` | 方法 | 触发自定义事件 |

### dir.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `loadDir()` | 函数 | 初始化运行目录 |

### exception/index.ts

| 导出 | 类型 | 说明 |
|------|------|------|
| `loadException()` | 函数 | 注册全局异常处理 |
| `uncaughtExceptionMonitorHandler()` | 函数 | 注册异常监控（不影响后续处理流程） |

## 模块关系

```
boot.ts (ElectronEgg)
  ├── 依赖 → electron/app (获取 Electron app 路径/名称)
  ├── 依赖 → ps (解析命令行参数、路径工具)
  ├── 依赖 → exception (loadException)
  ├── 依赖 → config (loadConfig)
  ├── 依赖 → dir (loadDir)
  ├── 依赖 → log (loadLog)
  └── 依赖 → application (app 单例，register/run/runAsync)

application.ts (Application)
  ├── 依赖 → controller (loadController / loadControllerAsync)
  ├── 依赖 → socket (loadSocket)
  ├── 依赖 → electron (loadElectron)
  └── 依赖 → events (eventBus, Ready)

events.ts (EventBus)
  └── 依赖 → log (coreLogger，用于 warn/error 输出)

dir.ts
  ├── 依赖 → ps (路径获取函数)
  ├── 依赖 → utils/helper (mkdir)
  └── 依赖 → utils/is (openharmony 判断)

exception/index.ts
  ├── 依赖 → log (coreLogger)
  ├── 依赖 → ps (进程类型判断)
  ├── 依赖 → config (getConfig，读取退出配置)
  └── 依赖 → message (childMessage，子进程错误发送)
```

所有模块的路径信息最终来源于 boot.ts 写入的 `process.env`，通过 `ps/index.ts` 的工具函数间接读取。这形成了 boot → process.env → ps → 各模块的间接依赖链。
