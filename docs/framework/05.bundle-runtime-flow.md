# 构建时→运行时完整数据流

## 概述

本文描述 ee-bin 的构建时插件如何生成数据，ee-core 的运行时如何消费这些数据。这是框架最核心的 producer-consumer 契约——构建时扫描目录结构并生成 registry，运行时读取 registry 或回退到 filesystem 扫描，两种模式产生完全一致的数据结构。

## 关键契约点

### 1. computeProperties 必须复现 ee-core 的算法

`packages/ee-bin/src/plugins/bundle_registry_plugin.ts:computeProperties()` 必须产生与 `packages/ee-core/src/utils/wrap.ts:getProperties()` + `defaultCamelize(caseStyle:'lower')` 完全一致的 property path 数组。

差异会导致：bundle 模式下控制器注册到错误的属性路径，IPC 通道名不匹配，前端调用找不到 handler。

### 2. filename 必须匹配 ConfigLoader 的查找逻辑

`packages/ee-bin/src/plugins/bundle_registry_plugin.ts` 生成的 `ConfigEntry.filename` 是不带扩展名的路径（如 `config.default`、`config.prod`），与 `packages/ee-core/src/config/config_loader.ts:_loadConfig()` 的查找逻辑匹配：`globalThis.__EE_CONFIG_REGISTRY__.find(e => e.filename === filename)`。

### 3. 加载顺序不可颠倒

bundle-entry 的 require 顺序：`config-registry → controller-registry → main.js`。这与 ee-core 的生命周期一致（`loadConfig → loadController → loadElectron`）。如果顺序颠倒，控制器加载时配置尚未注入，导致运行时错误。

### 4. banner EE_BUNDLED 是辅助检测信号

`process.env.EE_BUNDLED = "true"` 注入在 bundle 输出的最顶部。虽然运行时主要依赖 `globalThis.__EE_CONTROLLER_REGISTRY__` 和 `__EE_CONFIG_REGISTRY__` 的存在性来判断 bundle 模式，但 `EE_BUNDLED` 可用于其他模块（如 `loader/index.ts` 的 `resolveModule()`）作为辅助判断。

### 5. lazy getter 避免初始化顺序问题

registry 中的 `get module() { return require(relPath); }` 是 lazy getter。这意味着所有控制器模块不会在 registry 加载时立即执行，而是在 `FileLoader.parseFromRegistry()` 实际需要时才被加载。如果改为直接 `module: require(relPath)`，控制器代码会在 main.js 执行前全部运行，可能产生初始化顺序冲突。

## Dev vs Bundle 模式对比

| 维度 | Dev 模式 (filesystem) | Bundle 模式 (registry) |
|------|----------------------|----------------------|
| 触发条件 | `__EE_*_REGISTRY__` 不存在 | `__EE_*_REGISTRY__` 存在 |
| 扫描方式 | `scanDirSync()` 递归 fs.readdir | 直接读取 registry 数组 |
| 文件加载 | `loadFile()` → `require()` | registry lazy getter → `require()` |
| ESM interop | `require()` 返回值解包 | registry module 解包 |
| 配置查找 | `loadFile(filepath, appInfo)` | `registry.find(e => e.filename === name)` |
| 代码位置 | `./electron/` (源码目录) | `./public/electron/main.js` (bundle输出) |
| 调试 | `--debugger=true` → electronDir 指向源码 | 默认指向 bundle 输出 |
| sourcemap | `--env=dev` → inline sourcemap | prod → disabled |

两种模式产生的最终数据结构完全一致——控制器映射对象和配置对象的 key/path/value 完全相同，只是获取方式不同。

## _copyUnbundledFiles 的必要性

某些文件不能被 esbuild 打包进 main.js：

| 文件类型 | 原因 | 处理方式 |
|---------|------|---------|
| `preload/bridge.*` | BrowserWindow preload 必须是独立文件，Electron 直接从磁盘加载 | 逐文件 esbuild 转译 (bundle:false) |
| `jobs/**/*.ts` | `child_process.fork()` 需要磁盘上的真实文件，不能从 bundle 内执行 | 逐文件 esbuild 转译 (bundle:false) |
| 用户自定义 copy | 业务指定的独立资源 | 逐文件转译或原样拷贝 |

`_transpileDir()` 对脚本文件使用 `bundle:false` 的 esbuild 编译，确保 import/require 保持为运行时引用（不被打包解析），同时将 TypeScript 转译为 Node.js 可执行的 JavaScript。非脚本文件（`.json` 等）原样拷贝。
