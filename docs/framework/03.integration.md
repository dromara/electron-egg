# 框架与应用集成

## 概述

本文描述 ee-core/ee-bin 框架与业务应用代码的集成方式。应用代码位于 `electron/` 目录，通过框架提供的 API 实现业务逻辑。框架与应用的边界清晰：框架提供基础设施（生命周期、配置、控制器注册、IPC通信），应用专注于业务实现。

## 应用入口

**源文件**: `electron/main.ts` (19行)

入口代码完全依赖框架驱动：

```typescript
import { ElectronEgg } from 'ee-core';
import { Lifecycle } from './preload/lifecycle';
import { preload } from './preload';

const app = new ElectronEgg();          // 构造器触发 init() → 加载基础模块
const life = new Lifecycle();
app.register("ready", life.ready);      // 注册生命周期钩子
app.register("electron-app-ready", life.electronAppReady);
app.register("window-ready", life.windowReady);
app.register("before-close", life.beforeClose);
app.register("preload", preload);       // 注册预加载钩子
app.run();                              // 启动业务模块 → Electron窗口
```

框架构造器完成所有基础初始化（异常→配置→目录→日志），`run()` 按序启动业务模块。

## 控制器模式

**源目录**: `electron/controller/`

控制器是纯 TypeScript 类，被框架自动发现和注册。框架扫描目录结构，将每个类方法包装为 IPC handler。

### 命名→通道映射

| 文件路径 | 类名 | 方法 | IPC 通道 |
|---------|------|------|---------|
| `controller/example.ts` | ExampleController | `test()` | `controller/example/test` |
| `controller/framework.ts` | FrameworkController | `sqlitedbOperation()` | `controller/framework/sqlitedbOperation` |
| `controller/os/window.ts` | WindowController | `createWindow()` | `controller/os/window/createWindow` |

### 控制器方法签名

```typescript
// IPC 调用：参数 + 可选 event
async someMethod(args: unknown, event?: IpcMainEvent): unknown

// HTTP 调用：参数 + Context
async someMethod(args: unknown, ctx?: Context): unknown
```

框架为每次调用创建新的控制器实例（`methodToMiddleware` 中 `new Controller()`），确保并发安全。

### ee-core API 使用示例

```typescript
// FrameworkController 使用的 API
import { getExtraResourcesDir } from 'ee-core/ps';     // 路径查找
import { logger } from 'ee-core/log';                   // 日志
import { getConfig } from 'ee-core/config';             // 配置读取
import { getMainWindow } from 'ee-core/electron';       // 窗口访问
```

## Service 模式

**源目录**: `electron/service/`

Service 是单例实例，由控制器直接 import 使用，框架不自动管理：

```typescript
// service/database/sqlitedb.ts
import { SqliteStorage } from 'ee-core/storage';
import { getDataDir } from 'ee-core/ps';

export const sqlitedbService = new SqlitedbService();
```

Service 的典型 ee-core API 使用：
- `SqliteStorage` (storage) — 数据库操作
- `ChildJob/ChildPoolJob` (jobs) — 后台任务
- `cross` (cross) — 跨进程通信
- `logger` (log) — 日志
- `is` (utils) — 平台检测

## Preload 模式

**源文件**: `electron/preload/index.ts`

`preload()` 函数注册在 `Preload` 生命周期上，在 Electron app.whenReady() 后、窗口创建前执行：

```typescript
export function preload(): void {
  trayService.create();          // 系统托盘
  securityService.create();       // 安全策略
  autoUpdaterService.create();    // 自动更新
  crossService.createGoServer();  // Go 跨进程服务
  sqlitedbService.init();         // SQLite 初始化
}
```

这是服务启动的统一入口——需要随应用启动的服务都在此初始化。

**Bridge 文件**: `electron/preload/bridge.ts`
```typescript
import { contextBridge, ipcRenderer } from 'electron'
contextBridge.exposeInMainWorld('electron', { ipcRenderer })
```
当 `contextIsolation: true` 时使用此 bridge；当前配置 `nodeIntegration: true`，渲染进程可直接 `require('electron')`。

## 前端 IPC 集成

**源文件**: `frontend/src/api/index.js`

前端定义 IPC 通道映射，格式遵循 `controller/{dir}/{file}/{method}`：

```javascript
const ipcApiRoute = {
  framework: {
    ipcInvokeMsg: 'controller/framework/ipcInvokeMsg',
    sqlitedbOperation: 'controller/framework/sqlitedbOperation',
  },
  os: {
    createWindow: 'controller/os/window/createWindow',
  },
}
```

**IPC 获取方式** (`frontend/src/utils/ipcRenderer.js`):
```javascript
const Renderer = (window.require && window.require('electron')) || window.electron || {};
const ipc = Renderer.ipcRenderer;
```
支持三种场景：nodeIntegration 直接 require、bridge contextBridge、非 Electron 环境降级。

### 前端调用模式

- **invoke/handle** (推荐异步): `ipc.invoke(channel, args)` → Promise
- **send/on** (同步): `ipc.sendSync(channel, args)` → 直接返回值
- **send/listen** (双向): `ipc.send(channel, args)` + `ipc.on(channel, handler)`

## 配置集成

**源目录**: `electron/config/`

三层配置合并：

| 层级 | 文件 | 作用 |
|------|------|------|
| 框架默认 | ee-core `default_config.ts` | 框架内置默认值 |
| 项目默认 | `config.default.ts` | 项目基础配置 |
| 环境覆盖 | `config.local.ts` / `config.prod.ts` | 环境 specific 覆盖 |

合并顺序：框架默认 → config.default → config.{env}，后者覆盖前者。

## 构建配置集成

**源文件**: `cmd/bin.js`

`cmd/bin.js` 是 ee-bin 的项目配置文件，定义整个开发/构建生命周期：

```javascript
module.exports = {
  dev: { frontend: {...}, electron: {...} },
  build: { frontend: {...}, electron: { bundleType, format, ... }, win64: {...}, mac: {...} },
  move: { frontend_dist: { src, dest } },
  encrypt: { frontend: {...}, electron: {...} },
  exec: { go: {...}, python: {...} },
};
```

ee-bin 的 `loadConfig()` 加载此文件并与 `bin_default.ts` 深合并。

## 目录对照：开发 vs 生产

| 目录 | 开发模式 | 生产模式（打包后） |
|------|---------|-----------------|
| 主进程代码 | `./electron/` (TypeScript源码) | `./public/electron/main.js` (bundle输出) |
| 配置文件 | `./electron/config/` | 打包进 main.js (registry模式) |
| 控制器 | `./electron/controller/` | 打包进 main.js (registry模式) |
| 前端 | `http://localhost:8080` (dev server) | `./public/dist/index.html` (静态文件) |
| Job 文件 | `./electron/jobs/` | `./public/electron/jobs/` (逐文件转译) |
| Preload | `./electron/preload/bridge.ts` | `./public/electron/preload/bridge.js` (转译) |

框架通过 `--debugger=true` CLI 参数切换 electronDir 指向（源码目录 vs bundle 输出目录）。
