electron-egg 项目使用 ee-bin 构建工具和 electron-builder 打包平台安装包。本文档涵盖构建模式选择、平台打包命令、打包配置和常见问题。

## 构建模式选择

ee-bin 支持两种构建模式：

<Tabs>
  <Tab title="bundle 模式（推荐）">
    使用 esbuild 将 Electron 主进程代码打包为单一文件，通过 controllerRegistryPlugin 生成控制器注册表，运行时从注册表加载。

    ```javascript
    // cmd/bin.js
    build: {
      electron: {
        bundleType: 'bundle',
      },
    },
    ```

    构建产物结构：

    ```
    public/electron/
    ├── main.js              # 打包后的主进程代码（包含控制器、服务、配置）
    ├── jobs/                # 子进程任务脚本（child_process.fork 要求独立文件）
    │   └── example/
    │       ├── hello.js
    │       └── timer.js
    └── preload/
        └── bridge.js        # BrowserWindow preload 脚本（必须独立文件）
    ```

    <Note>
    bundle 模式下，控制器、服务、配置都被打包到 main.js 中，运行时不需要文件系统扫描。只有 jobs/ 和 preload/bridge.js 需要保持为独立文件。
    </Note>
  </Tab>
  <Tab title="copy 模式">
    直接复制 electron/ 目录到 public/electron/，保留原始文件结构。运行时通过 globby 文件系统扫描加载控制器。

    ```javascript
    // cmd/bin.js
    build: {
      electron: {
        bundleType: 'copy',
      },
    },
    ```

    构建产物结构：

    ```
    public/electron/
    ├── main.js              # 复制的原始 main.js
    ├── controller/          # 控制器文件（运行时 globby 扫描）
    │   └── user.js
    ├── service/             # 服务文件
    │   └── db.js
    ├── config/              # 配置文件
    │   ├── config.default.js
    │   └── config.local.js
    ├── jobs/                # 子进程任务脚本
    │   └── example/
    │       ├── hello.js
    │       └── timer.js
    └── preload/
        └── bridge.js        # BrowserWindow preload 脚本
    ```

    <Warning>
    copy 模式保留了完整文件结构，容易被逆向分析。建议在 production 环境使用 bundle + 加密组合以保护代码。
    </Warning>
  </Tab>
</Tabs>

### 模式对比

| 特性 | bundle | copy |
|------|--------|------|
| 代码保护 | 较好（单一文件） | 较差（文件分散） |
| 启动速度 | 快（无需文件扫描） | 慢（globby 扫描） |
| 代码体积 | 较小（去重合并） | 较大（重复依赖） |
| 调试便利 | 较差（需要 sourcemap） | 较好（原始文件） |
| 加密兼容 | 好（单一文件加密） | 一般（多个文件加密） |

## Bundle 配置详解

```javascript
// cmd/bin.js - 完整 bundle 配置
build: {
  electron: {
    bundleType: 'bundle',
    format: 'cjs',           // 'cjs' | 'esm'（默认 cjs，推荐 Electron）
    external: [],             // 用户自定义外部包
    sourcemap: false,         // 'inline' | 'external' | false（默认自动推断）
    minify: false,            // true | false（生产环境建议 true）
    drop: undefined,          // ['console', 'debugger']（生产环境移除调试语句）
    keepNames: false,         // true | false（minify 时保留函数/类名）
    legalComments: undefined, // 'inline' | 'eof' | 'none'（处理许可证注释）
    define: undefined,        // { 'process.env.X': '"value"' }（编译时常量）
    copy: undefined,          // ['assets', 'data/db.json']（额外复制的文件/目录）
  },
},
```

### 框架管理的外部包

以下包由框架自动管理为 esbuild external，不需要用户手动添加：

```javascript
const frameworkExternal = [
  'ee-core',       // child_process.fork() 需要真实文件
  'ee-bin',        // CLI 工具，不参与运行时
  'electron',      // 原生模块
  'better-sqlite3', // 原生 C++ 模块
  'proxy-agent',   // 网络代理
  'pino-roll',     // 日志文件轮转
  'pino-pretty',   // 日志美化
];
```

### Sourcemap 行为

| 配置值 | 开发环境 | 生产环境 | 说明 |
|-------|---------|---------|------|
| `false`（默认） | inline | off | 自动推断 |
| `'inline'` | inline | inline | 强制内联 sourcemap |
| `'external'` | external | external | 独立 sourcemap 文件 |
| `true` | external | external | 同 `'external'` |

## 平台打包命令

<Tabs>
  <Tab title="Windows">
    ```bash
    # Windows 64 位
    pnpm build-w

    # Windows 32 位（需要额外配置）
    # 在 cmd/bin.js 中配置 arch 为 ia32
    ```
  </Tab>
  <Tab title="macOS">

    pnpm build-m

    # macOS ARM64 (M1/M2/M3)

  </Tab>
  <Tab title="Linux">
    ```bash
    pnpm build-l
    ```
  </Tab>

### 打包流程

```bash
# 完整打包流程
pnpm build            # 1. 构建前端 + Electron + 加密
                      #    → build-frontend → build-electron → encrypt

pnpm build-w          # 2. 打包 Windows 安装包
                      #    → electron-builder → 生成 .exe / .msi
```

<Note>
打包命令会先执行 build 流程（前端构建、Electron 构建、加密），然后调用 electron-builder 生成平台安装包。如果只想打包而不重新构建，需要修改 build 命令配置。
</Note>

## electron-builder 配置

electron-builder 的配置放在 `cmd/` 目录下的 JSON 文件中：

```json
// cmd/builder64.json (Windows 64-bit)
{
  "productName": "MyApp",
  "appId": "com.example.myapp",
  "directories": {
    "output": "build/out"
  },
  "files": [
    "package.json",
    "public/**/*",
    "node_modules/**/*"
  ],
  "extraResources": [
    {
      "from": "go/go_server",
      "to": "go_server"
    }
  ],
  "win": {
    "target": ["nsis"],
    "icon": "build/icons/icon.ico"
  },
  "nsis": {
    "oneClick": false,
    "allowToChangeInstallationDirectory": true,
    "createDesktopShortcut": true
  },
  "mac": {
    "target": ["dmg"],
    "icon": "build/icons/icon.icns"
  },
  "dmg": {
    "contents": [
      { "x": 130, "y": 220 },
      { "x": 410, "y": 220, "type": "link", "path": "/Applications" }
    ]
  },
  "linux": {
    "target": ["AppImage"],
    "icon": "build/icons"
  }
}
```

### extraResources 配置

Cross 模块启动的 Go/Python 后端程序需要通过 `extraResources` 打包：

```json
{
  "extraResources": [
    {
      "from": "go/go_server",
      "to": "go_server"
    },
    {
      "from": "python/python_app",
      "to": "python_app"
    }
  ]
}
```

<Note>
CrossProcess 在生产环境中从 `extraResources` 目录查找可执行文件。extraResources 中的文件会被复制到安装包的 resources 目录，与 app.asar 同级。
</Note>

### files 配置

```json
{
  "files": [
    "package.json",
    "public/**/*",
    "node_modules/**/*",
    "!node_modules/ee-bin/**/*",  // 排除 ee-bin（仅 CLI 工具）
    "!frontend/**/*"              // 排除前端源码（已构建到 public/）
  ]
}
```

## 打包后运行问题

### main.js 路径问题

打包后 package.json 的 `main` 字段需要指向正确的入口文件：

- bundle 模式：`"main": "./public/electron/main.js"`
- copy 模式：`"main": "./public/electron/main.js"`

ee-bin 在 build 时自动切换 package.json 的 main 字段，build 完成后恢复。

<Warning>
如果在 build 过程中中断（Ctrl+C），package.json 的 main 字段可能没有恢复。ee-bin 注册了 SIGINT/SIGTERM 处理器来恢复，但极端情况下可能失败。手动修复方法：

```bash
# 检查并修复 package.json
cat package.json | grep main
# 如果显示 ./public/electron/main.js 而非 ./electron/main.js
# 手动修改回原始值
```
</Warning>

### better-sqlite3 原生模块

打包后 better-sqlite3 需要重新编译以匹配 Electron 的 Node.js 版本：

```bash
# 重新编译 better-sqlite3
pnpm re-sqlite
```

<Note>
如果打包后运行时出现 `better-sqlite3` 相关错误（如 `MODULE_NOT_FOUND` 或 `native module error`），通常是因为原生模块没有针对 Electron 版本重新编译。运行 `pnpm re-sqlite` 即可修复。
</Note>

### CrossProcess 路径问题

打包后 CrossProcess 的可执行文件路径从 `extraResources` 目录查找：

```javascript
// 开发环境：基于项目根目录
cmdPath = path.join(getBaseDir(), 'go/go_server');

// 生产环境：基于 extraResources 目录
cmdPath = path.join(getExtraResourcesDir(), 'go_server');
```

<Warning>
确保 extraResources 配置与 Cross 配置中的文件名一致。如果 extraResources 中打包为 `go_server`，Cross 配置的 `cmd` 应为 `go_server`（不含路径）。
</Warning>

## asar 解包问题

electron-builder 默认将应用代码打包为 app.asar 格式。某些文件需要解包才能正常运行：

### 需要解包的文件

| 文件类型 | 原因 |
|---------|------|
| `node_modules/ee-core/` | child_process.fork() 需要真实文件 |
| `node_modules/better-sqlite3/` | 原生 C++ 模块 |
| `public/electron/jobs/` | child_process.fork() 需要真实文件 |
| `public/electron/preload/bridge.js` | BrowserWindow preload 脚本 |
| `go/go_server` | 外部可执行程序（extraResources） |

### asarUnpack 配置

```json
{
  "asar": true,
  "asarUnpack": [
    "node_modules/ee-core/**/*",
    "node_modules/better-sqlite3/**/*",
    "public/electron/jobs/**/*",
    "public/electron/preload/**/*"
  ]
}
```

<Warning>
child_process.fork() 无法从 asar 内部加载子进程入口文件。ee-core 的 app.js 入口必须在磁盘上作为真实文件存在，否则 fork 会失败。这就是 ee-core 不被打包到 main.js 的根本原因。
</Warning>

### 解包文件路径变化

解包后的文件路径会从 `app.asar/` 变为 `app.asar.unpacked/`：

```javascript
// asar 内：path/to/app.asar/node_modules/ee-core/app.js
// 解包后：path/to/app.asar.unpacked/node_modules/ee-core/app.js
```

Electron 的 `app.getAppPath()` 会自动处理 asar 路径映射，但 `fs.readFileSync()` 等直接文件操作需要注意路径差异。
