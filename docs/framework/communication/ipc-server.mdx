IpcServer 是 electron-egg 框架的核心 IPC 通信服务，负责将控制器方法注册为 Electron IPC 通道处理函数，使渲染进程可以通过 `ipcRenderer` 调用主进程的业务方法。

## 工作原理

IpcServer 通过递归遍历控制器对象树，为每个导出方法注册 IPC 通道。整个流程分为三个阶段：

1. **遍历（loop）**：递归扫描控制器对象树的嵌套结构
2. **检测（EXPORTS）**：通过 Symbol 标记识别叶子节点（方法集合）
3. **注册（register）**：为每个方法注册 `ipcMain.on` 和 `ipcMain.handle`

<Note>
EXPORTS Symbol 标记由 FileLoader 在加载控制器文件时设置。带有 `[EXPORTS] = true` 标记的对象被视为叶子节点（即控制器方法集合），没有标记的对象为中间节点（目录/命名空间），继续递归。
</Note>

### 遍历逻辑

```javascript
// 控制器对象树结构
controller = {
  user: {
    add: fn,       // EXPORTS = true → 叶子节点
    delete: fn,    // EXPORTS = true → 叶子节点
  },
  admin: {
    login: {
      auth: fn,    // EXPORTS = true → 叶子节点
    }
  }
}
```

递归遍历过程中，`loop()` 方法对每个属性进行检查：
- 如果属性对象带有 `EXPORTS` Symbol 标记 → 调用 `register()` 注册所有方法
- 如果属性对象是普通对象 → 继续递归深入

### 通道命名规则

通道名称由控制器属性路径生成，将 `.` 替换为 `channelSeparator`（默认 `/`）：

```
属性路径: controller.user.add
通道名称: controller/user/add
```

可在配置文件中自定义分隔符：

```javascript
// config.default.js
module.exports = {
  mainServer: {
    channelSeparator: '/',  // 默认值，也可改为 '.' 或其他
  },
};
```

## 双通信模型

IpcServer 为每个方法注册两种 IPC 处理器，对应两种通信模型：

<Tabs>
  <Tab title="send/on 同步模型">
    渲染进程使用 `ipcRenderer.sendSync()` 发送消息，主进程通过 `ipcMain.on()` 接收并设置 `event.returnValue` 返回结果。

    ```javascript
    // 主进程注册（IpcServer 自动完成）
    ipcMain.on('controller/user/add', async (event, params) => {
      const result = await fn.call(controller, params, event);
      event.returnValue = result;
    });

    // 渲染进程调用
    const result = ipcRenderer.sendSync('controller/user/add', { name: 'test' });
    ```

    <Note>
    Electron 的 `sendSync` 会进入嵌套消息循环处理微任务，因此 `event.returnValue` 在 `await` 之后设置也能被 `sendSync` 接收到。这意味着同步模型也支持异步控制器方法。
    </Note>
  </Tab>
  <Tab title="invoke/handle 异步模型（推荐）">
    渲染进程使用 `ipcRenderer.invoke()` 发送消息，主进程通过 `ipcMain.handle()` 接收并返回 Promise。

    ```javascript
    // 主进程注册（IpcServer 自动完成）
    ipcMain.handle('controller/user/add', async (event, params) => {
      return await fn.call(controller, params, event);
    });

    // 渲染进程调用
    const result = await ipcRenderer.invoke('controller/user/add', { name: 'test' });
    ```

    <Warning>
    invoke/handle 模型是推荐的使用方式。它不会阻塞渲染进程，支持异步控制器方法，且错误处理更优雅。
    </Warning>
  </Tab>
</Tabs>

## 同步 vs 异步选择指南

| 特性 | send/on (同步) | invoke/handle (异步) |
|------|---------------|---------------------|
| 阻塞渲染进程 | 是 | 否 |
| 支持异步方法 | 是（Electron 嵌套消息循环） | 是（原生 Promise） |
| 返回值 | `event.returnValue` | Promise resolve |
| 错误处理 | `returnValue = undefined` | 可捕获异常 |
| 推荐程度 | 一般 | **推荐** |

<Note>
绝大多数场景应使用 invoke/handle 异步模型，避免阻塞渲染进程导致界面卡顿。仅在必须同步获取结果的场景（如窗口创建前的配置读取）使用 send/on 同步模型。
</Note>

## 渲染进程调用示例

### 通过 preload bridge 调用

```javascript
// preload/bridge.js
const { ipcRenderer } = require('electron');

const api = {
  // 异步调用（推荐）
  async addUser(params) {
    return await ipcRenderer.invoke('controller/user/add', params);
  },
  // 同步调用
  getUserSync(params) {
    return ipcRenderer.sendSync('controller/user/get', params);
  },
};

// 暴露给渲染进程
module.exports = api;
```

### 前端 Vue 组件调用

```javascript
// frontend/src/views/User.vue
import { api } from '@/api/bridge';

export default {
  methods: {
    async handleAdd() {
      const result = await api.addUser({ name: 'Alice', age: 25 });
      console.log('添加用户结果:', result);
    },
  },
};
```

## resolveControllerFn 路径解析

IpcServer 使用 `resolveControllerFn()` 将 IPC 通道路径解析为控制器方法函数。该函数按分隔符分割路径，逐级遍历控制器对象树：

```
通道: controller/user/add
分割: ['controller', 'user', 'add']
遍历: controller → controller.user → controller.user.add → 返回函数
```
