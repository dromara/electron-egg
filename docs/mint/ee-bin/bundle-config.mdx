BundleConfig 控制 esbuild 对 Electron 主进程代码的打包行为。它定义在 `cmd/bin.js` 的 `build.electron` 配置段中，决定了输出格式、外部依赖、调试信息、压缩等关键构建参数。

## 字段总览

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `bundleType` | `'bundle' \| 'copy'` | `'bundle'` | 打包模式 |
| `external` | `string[]` | `[]` | 用户定义的外部包 |
| `sourcemap` | `boolean \| 'inline' \| 'external'` | `false` | Sourcemap 配置 |
| `minify` | `boolean` | `false` | 代码压缩 |
| `drop` | `('console' \| 'debugger')[]` | - | 移除语句 |
| `keepNames` | `boolean` | `false` | 保留函数/类名 |
| `legalComments` | `'inline' \| 'eof' \| 'none'` | - | 许可注释处理 |
| `define` | `Record<string, string>` | - | 编译时常量定义 |
| `copy` | `string[]` | - | 额外复制文件 |
| `format` | `'cjs' \| 'esm'` | `'cjs'` | 输出格式 |

## 字段详解

### bundleType

控制 Electron 代码的打包方式：

<Tabs>
<Tab title="bundle 模式">
使用 esbuild + `bundleRegistryPlugin` 将所有控制器、服务、配置打包为 `public/electron/main.js` 单文件。

打包流程：
1. 清除输出目录 `public/electron/`
2. 执行 esbuild 构建（虚拟入口 `app:bundle-entry`）
3. 加载配置注册 → 控制器注册 → main.js
4. 在输出 banner 中注入 `process.env.EE_BUNDLED = "true"`（ee-core 通过此标记识别 bundle 模式）
5. 复制不可打包的文件（jobs 目录、preload/bridge.js）

```js
electron: {
  bundleType: 'bundle',
}
```
</Tab>
<Tab title="copy 模式">
直接将 `electron/` 目录复制到 `public/electron/`，不执行 esbuild 打包。

<Warning>
Copy 模式不会生成注册表（`global.__EE_CONTROLLER_REGISTRY__` / `global.__EE_CONFIG_REGISTRY__`）。运行时 ee-core 会退回到 globby 文件系统扫描方式加载控制器和配置。
</Warning>

```js
electron: {
  bundleType: 'copy',
}
```
</Tab>
</Tabs>

### external

用户定义的 esbuild 外部包列表。这些包不会被打包进 `main.js`，而是在运行时从 `node_modules` 加载。

<Note>
框架自动管理以下外部包，无需手动添加：

- `ee-core` — `child_process.fork()` 需要其实际文件路径
- `ee-bin` — CLI 工具，不属于运行时
- `electron` — 原生模块，esbuild 无法打包
- `better-sqlite3` — 原生模块
- `proxy-agent` — 依赖原生模块
- `pino-roll` — 依赖 fs 操作，打包后失效
- `pino-pretty` — 依赖 fs 操作，打包后失效

此外，esbuild 配置了 `packages: 'external'`，将所有 `node_modules` 中的包标记为外部。框架显式列表主要用于文档说明和性能优化。
</Note>

用户自定义示例：

```js
electron: {
  external: ['my-native-module', 'another-external-package'],
}
```

### sourcemap

控制是否生成 sourcemap 文件，用于调试时将打包代码映射回源代码：

| 值 | 行为 | 适用场景 |
|----|------|----------|
| `false`（默认） | 自动模式：dev 环境 → inline，prod 环境 → 关闭 | 日常开发 |
| `true` / `'inline'` | 强制生成 inline sourcemap（嵌入输出文件） | 强制调试 |
| `'external'` | 生成独立 `.map` 文件（`main.js.map`） | VS Code 调试 |

自动模式的判断逻辑：

```js
const isDev = process.env.NODE_ENV === 'dev' || process.env.NODE_ENV === 'local';
// false → 自动推断
sourcemap = isDev ? 'inline' : false;
```

<Note>
VS Code 调试配置使用 inline sourcemap，在 `.vscode/launch.json` 的 "Debug Electron" 任务中通过 `--env=dev` 启用。断点设在 `electron/` 源文件中，sourcemap 将 `public/electron/main.js` 映射回源代码。
</Note>

### minify

是否压缩代码（移除空白、缩短变量名等）：

```js
electron: {
  minify: true,  // 生产构建推荐启用
}
```

默认 `false`。生产发布时建议启用以减小输出体积。启用 minify 时，函数和类名会被缩短，如需保留可设置 `keepNames: true`。

### drop

指定要移除的语句类型，用于生产构建去除调试代码：

```js
electron: {
  drop: ['console', 'debugger'],
}
```

| 值 | 效果 |
|----|------|
| `'console'` | 移除所有 `console.*` 调用 |
| `'debugger'` | 移除所有 `debugger` 语句 |

可以只指定其中一个，也可以两个都指定。

### keepNames

在 minify 模式下保留函数和类名：

```js
electron: {
  minify: true,
  keepNames: true,  // 保留函数/类名，便于调试和反射
}
```

默认 `false`。启用后 esbuild 不会缩短函数名和类名，输出体积会略大，但代码可读性和反射能力更好。

### legalComments

控制许可证注释（如 `/* @license */`）的处理方式：

```js
electron: {
  legalComments: 'none',  // 移除所有许可证注释
}
```

| 值 | 行为 |
|----|------|
| `'inline'` | 保留在代码中 |
| `'eof'` | 移到文件末尾 |
| `'none'` | 完全移除 |

不设置时，esbuild 默认行为是保留注释。

### define

定义编译时常量，类似于 C 的 `#define`。esbuild 在打包时将匹配的标识符替换为常量值：

```js
electron: {
  define: {
    'process.env.CUSTOM_VAR': '"hello"',
    'process.env.API_URL': '"https://api.example.com"',
  },
}
```

<Warning>
define 的值必须是字符串表达式。要定义字符串常量，值需要用双引号包裹：`'"value"'`。数字常量直接写数字：`'42'`。
</Warning>

### copy

指定需要从 `electron/` 目录额外复制到 `public/electron/` 的文件或目录。这些文件不参与 esbuild 打包，需要以独立文件形式存在：

```js
electron: {
  copy: ['assets', 'data/db.json', 'templates'],
}
```

框架自动复制的文件（无需手动配置）：

- `jobs/` 目录 — `child_process.fork()` 要求子进程入口为独立文件
- `preload/bridge.js` — BrowserWindow preload 脚本必须为独立文件

copy 目标可以是目录或文件：
- 目录 → 递归复制
- 文件 → 单文件复制，自动创建目标目录

### format

输出格式，决定打包后的模块系统：

<Tabs>
<Tab title="cjs（推荐）">
CommonJS 格式，使用 `require()` / `module.exports`。这是 Electron 主进程的推荐格式，兼容性最好。

```js
electron: {
  format: 'cjs',  // 默认值
}
```

入口文件 `electron/main.js`（JS 文件）自动使用 CJS 格式。
</Tab>
<Tab title="esm">
ESM 格式，使用 `import` / `export`。要求所有业务代码（控制器、服务、配置）都是 ESM 兼容的。

```js
electron: {
  format: 'esm',
}
```

<Warning>
ESM 模式要求：
1. 入口文件使用 `electron/main.ts`（TypeScript）
2. 所有控制器、服务、配置文件必须是 ESM 格式
3. ESM 导入必须包含 `.js` 扩展名（Node.js `--experimental-specifier-resolution=node` 要求）
</Warning>

ee-bin 通过检测入口文件类型自动推断格式偏好：`main.js` → CJS，`main.ts` → ESM。用户可通过 `format` 字段显式覆盖。
</Tab>
</Tabs>

## 完整配置示例

```js
// cmd/bin.js
module.exports = {
  build: {
    frontend: {
      directory: './frontend',
      cmd: 'npm',
      args: ['run', 'build'],
    },
    electron: {
      bundleType: 'bundle',
      external: ['my-native-module'],
      sourcemap: false,            // 自动模式
      minify: true,                // 生产压缩
      drop: ['console', 'debugger'], // 移除调试语句
      keepNames: false,
      legalComments: 'none',
      define: {
        'process.env.APP_VERSION': '"1.0.0"',
      },
      copy: ['assets', 'data/db.json'],
      format: 'cjs',
    },
    win64: {
      cmd: 'electron-builder',
      directory: './',
      args: ['--config=./cmd/builder.json', '-w=nsis', '--x64'],
    },
  },
};
```

## esbuild 内部配置

除了用户可见的 BundleConfig 字段，`_bundleWithRegistry()` 还设置了以下 esbuild 内部参数：

| 参数 | 值 | 说明 |
|------|-----|------|
| `entryPoints` | `['app:bundle-entry']` | 虚拟入口模块 |
| `bundle` | `true` | 启用打包 |
| `platform` | `'node'` | Node.js 平台 |
| `target` | `'node20'` | 目标 Node.js 20 |
| `packages` | `'external'` | 所有 npm 包标记为外部 |
| `banner.js` | `process.env.EE_BUNDLED = "true";` | 注入 bundle 标记 |
| `plugins` | `[bundleRegistryPlugin]` | 控制器和配置注册插件 |

`bundleRegistryPlugin` 在打包时扫描 `electron/controller/` 和 `electron/config/` 目录，生成虚拟模块注入全局注册表。运行时 ee-core 通过 `globalThis.__EE_CONTROLLER_REGISTRY__` 和 `globalThis.__EE_CONFIG_REGISTRY__` 加载模块。
