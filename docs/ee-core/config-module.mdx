# Config 模块

Config 模块负责加载和合并框架默认配置与业务自定义配置，是整个框架运行的基础。所有模块（日志、通信、异常等）都依赖配置系统。

## ConfigLoader 配置加载器

### 三层配置合并

配置合并遵循严格的优先级顺序，后加载的配置覆盖同名属性：

```
defaultConfig (框架内置) → config.default (业务默认) → config.{env} (环境特定)
```

- **defaultConfig**：框架内置的完整默认值，确保所有配置项有合理的初始值
- **config.default**：业务自定义的全局默认配置
- **config.local**（开发环境）或 **config.prod**（生产环境）：环境特定配置

<Note>
合并使用 `extend()` 深度递归合并，嵌套对象不会被直接覆盖而是逐层合并。`__proto__` 属性会被自动跳过以防原型链污染。
</Note>

### 两种加载模式

| 模式 | 触发条件 | 加载方式 |
|------|---------|---------|
| Bundle 模式 | `globalThis.__EE_CONFIG_REGISTRY__` 存在 | 从预注册模块查找配置 |
| Dev 模式 | 注册表不存在 | 从文件系统加载配置文件 |

Bundle 模式下，`controllerRegistryPlugin`（ee-bin esbuild 插件）在构建时扫描 `electron/config/` 目录，将配置模块预注册到全局变量中。

### 配置文件函数导出模式

配置文件支持导出函数，函数接收 `appInfo` 参数，可根据环境动态返回配置：

```javascript
// electron/config/config.default.js
module.exports = (appInfo) => {
  const config = {};

  // 根据 appInfo 动态设置路径
  config.logger = {
    dir: path.join(appInfo.root, 'logs'),
  };

  return config;
};
```

`appInfo` 对象结构：

| 字段 | 说明 |
|------|------|
| `name` | 应用名称 |
| `baseDir` | 项目根目录 |
| `electronDir` | Electron 源码目录 |
| `env` | 运行环境（dev/prod） |
| `root` | 日志和运行数据的根目录 |

## 配置 API

### loadConfig()

在框架启动流程中由 `boot.ts` 的 `init()` 调用，加载并合并配置。

```javascript
import { loadConfig } from 'ee-core/config';
loadConfig();
```

### getConfig()

获取完整的运行时配置对象。如果配置尚未加载，会自动触发加载。

```javascript
import { getConfig } from 'ee-core/config';
const config = getConfig();
console.log(config.logger.level); // 'info'
console.log(config.socketServer.enable); // false
```

### setConfig()

直接设置运行时配置，主要用于子进程场景。子进程通过 IPC 从主进程接收配置，避免在打包输出目录中缺少配置文件导致加载失败。

```javascript
import { setConfig } from 'ee-core/config';
// 子进程接收主进程传递的配置
setConfig(receivedConfig);
```

<Warning>
`setConfig()` 通常只在子进程（ChildJob/ChildPoolJob）中使用，主进程应使用 `loadConfig()` 从文件系统加载。
</Warning>

### getAppInfo()

获取应用基本信息对象：

```javascript
import { getAppInfo } from 'ee-core/config';
const appInfo = getAppInfo();
// { name: 'electron-egg', baseDir: '/path/to/project', electronDir: '/path/to/electron', env: 'dev', root: '/path/to/project' }
```

## 默认配置项参考

### openDevTools

| 值 | 说明 |
|----|------|
| `false` | 不打开 DevTools |
| `true` | 打开 DevTools（底部模式） |
| `{ mode: 'right' }` | 打开 DevTools 并指定位置 |

### singleLock

单实例锁配置。`true` 时只允许运行一个应用实例，后续启动会激活已有实例的窗口。

### windowsOption

主窗口配置，直接传递给 `BrowserWindow` 构造函数：

```javascript
windowsOption: {
  title: 'electron-egg',
  width: 980,
  height: 650,
  minWidth: 400,
  minHeight: 300,
  webPreferences: {
    contextIsolation: false,  // 安全警告：建议新项目设为 true
    nodeIntegration: true,
  },
  frame: true,
  show: false,   // 先加载内容再显示，避免白屏
  icon: path.join(baseDir, 'public/images/logo-32.png'),
}
```

<Warning>
`contextIsolation: false` + `nodeIntegration: true` 允许渲染进程完全访问 Node.js API。任何 XSS 漏洞可能导致远程代码执行。新项目建议设置 `contextIsolation: true` 并使用 preload 脚本。
</Warning>

### logger

日志配置详见 [Log 模块](/ee-core/log-module)。

### socketServer / httpServer / mainServer

通信服务配置详见 [Socket 模块](/ee-core/socket-module) 和 [Electron 模块](/ee-core/electron-module)。

### exception

异常处理配置详见 [App 模块](/ee-core/app-module)。

### jobs

后台任务配置：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `messageLog` | `boolean` | `false` | 是否记录子进程通信消息 |

### cross

跨进程配置详见 [Cross 模块](/ee-core/cross-module)。

## 配置文件编写模式

<CodeGroup>

```javascript [config.default.js]
module.exports = (appInfo) => {
  const config = {};

  // 开发环境配置
  config.logger = {
    level: 'debug',
    prettyPrint: true,
  };

  config.socketServer = {
    enable: true,
    port: 7070,
  };

  config.httpServer = {
    enable: true,
    port: 7071,
  };

  return config;
};
```

```javascript [config.prod.js]
module.exports = (appInfo) => {
  const config = {};

  // 生产环境配置
  config.logger = {
    level: 'info',
    prettyPrint: false,
  };

  config.mainServer = {
    protocol: 'file://',
    indexPath: '/public/dist/index.html',
  };

  return config;
};
```

</CodeGroup>
