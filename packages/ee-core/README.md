# ee-core

Enterprise-grade Electron application framework core — TypeScript edition with dual CJS + ESM module support.

## Overview

`ee-core` provides the runtime foundation for [electron-egg](https://github.com/wallace5303/electron-egg) applications. It handles the complete lifecycle from boot to shutdown: configuration loading, controller registration, IPC communication, window management, logging, storage, background jobs, and cross-process coordination.

The package outputs dual module formats (CJS + ESM) via conditional exports in `package.json`, making it compatible with both `require()` and `import` environments.

## Installation

```bash
npm install ee-core
```

## Subpath Exports

ee-core exposes fine-grained subpath modules. Each can be imported individually:

```ts
// Main entry (boot, app, config, etc.)
import { ElectronEgg } from 'ee-core';

// Specific submodules
import { ControllerLoader } from 'ee-core/controller';
import { Cross, cross } from 'ee-core/cross';
import { extend } from 'ee-core/utils';
import { SqliteStorage } from 'ee-core/storage';
import { ChildJob, ChildPoolJob, LoadBalancer } from 'ee-core/jobs';
import type { Config, DevConfig } from 'ee-core/types';
```

Available subpaths:

| Subpath | Description |
|---|---|
| `ee-core` | Main entry — boot, application, config, events |
| `ee-core/app` | Application lifecycle and event bus |
| `ee-core/config` | Configuration loading and merging |
| `ee-core/controller` | Controller loader and registry |
| `ee-core/core` | FileLoader and core utilities |
| `ee-core/cross` | IPC communication (socket.io, HTTP, Electron IPC) |
| `ee-core/electron` | Electron app/window management |
| `ee-core/exception` | Exception handling |
| `ee-core/html` | HTML static resource path utility |
| `ee-core/jobs` | Background jobs (ChildJob, ChildPoolJob, LoadBalancer) |
| `ee-core/loader` | File/module loading utilities |
| `ee-core/log` | Pino-based logging (pino-roll + pino-pretty) |
| `ee-core/message` | IPC message routing |
| `ee-core/ps` | Process/environment utilities (paths, env detection) |
| `ee-core/socket` | Socket.IO, HTTP, IPC servers |
| `ee-core/storage` | SQLite storage (better-sqlite3) |
| `ee-core/types` | TypeScript type definitions |
| `ee-core/utils` | extend, is, json, helper, pargv, port |

## Architecture

### Lifecycle

Boot → loadConfig → loadLog → loadDir → loadController → loadSocket → loadElectron

### Controller Loading Pipeline

- **Build time** (ee-bin): `controllerRegistryPlugin` scans `electron/controller/`, generates `ee-core:controller-registry` virtual module with lazy getters.
- **Runtime** (ee-core): If `globalThis.__EE_CONTROLLER_REGISTRY__` exists (bundle mode), loads from registry. Otherwise falls back to filesystem scanning via globby.

### IPC Communication

Main ↔ renderer via `ee-core/cross`, supporting three transport modes:
- Socket.IO (bidirectional real-time)
- HTTP server (RESTful)
- Electron IPC (native)

Channels follow pattern `controller/{name}/{method}`.

### Window Loading Strategy

- Remote mode → Load remote URL
- Dev mode → Wait for frontend dev server, show loading page first
- Prod + cross takeover → Wait for cross-process service
- Prod (default) → Load local HTML file

## Key APIs

### Boot & Application

```ts
import { ElectronEgg } from 'ee-core';

// Start the framework
const app = new ElectronEgg();
```

### Configuration

```ts
import { getConfig, setConfig } from 'ee-core/config';

const config = getConfig();
```

### Controller

```ts
import { getController } from 'ee-core/controller';

const userController = getController('user');
```

### Cross-Process Communication

```ts
import { cross } from 'ee-core/cross';

// Send IPC message to renderer
cross.sendToRenderer('controller/user/info', data);
```

### Jobs

```ts
import { ChildJob, ChildPoolJob, LoadBalancer } from 'ee-core/jobs';

const job = new ChildJob();
const result = await job.exec({ name: 'task', script: './jobs/demo.js' });
```

### Storage

```ts
import { SqliteStorage } from 'ee-core/storage';

const db = new SqliteStorage('./data/app.db');
db.prepare('CREATE TABLE IF NOT EXISTS user (id INTEGER PRIMARY KEY, name TEXT)').run();
```

### Logging

```ts
import { coreLogger, getLoggers } from 'ee-core/log';

coreLogger.info('Application started');
```

### Process/Environment

```ts
import { isDev, getBaseDir, env } from 'ee-core/ps';

if (isDev()) {
  // Development-specific logic
}
```

## Build

```bash
npm run build          # Build CJS + ESM + generate exports map
npm run build:cjs      # Build CommonJS only
npm run build:esm      # Build ESM only
npm run typecheck      # TypeScript type checking
npm run test           # Run vitest
```

The `exports` map in `package.json` is auto-generated by `scripts/gen-exports.js`. Do not edit manually — run `npm run gen-exports` after adding new subpath modules.

## Requirements

- Node.js >= 20.19.0
- Electron >= 28
- Better-sqlite3 requires native rebuild for Electron: use `pnpm re-sqlite` if you see native module errors

## Notes

- ESM imports in ee-core source must include `.js` extensions (required by `module: NodeNext`)
- ee-core is NOT bundled into the Electron main process bundle — it's an esbuild external loaded from `node_modules` at runtime
- The framework uses the `debug` library for verbose logging. Enable with `DEBUG=ee-*` environment variable
